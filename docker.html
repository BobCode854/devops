1. What is Docker?

Understanding the core problem Docker solves and what it actually is.

The "Works on My Machine" Problem

Before Docker: your Java app works locally but crashes on the server because of different JDK versions, OS libraries, or config — the classic "works on my machine" nightmare.
Without Docker
  • App works on dev laptop (JDK 17, Ubuntu)
  • Deployed to server (JDK 11, CentOS)
  • Crashes immediately on startup
  • "But it works on my machine!"
  • 2 days debugging environment mismatches
With Docker
  • App + JDK 17 + all dependencies packaged into one container
  • Runs identically on any machine with Docker installed
  • Same image in dev, staging, and production
  • Zero "works on my machine" problems
  • Ship the environment, not just the code

What Docker Is

Docker is a platform that packages your application + its entire environment into a lightweight, portable container — run it anywhere, get identical behaviour every time.
ConceptAnalogyWhat it means
ContainerShipping containerStandardized box — app + dependencies, runs anywhere
ImageBlueprint / recipeRead-only template used to create containers
DockerfileCooking recipeText file with instructions to build an image layer by layer
Docker HubApp StorePublic registry to share, discover, and download images
Docker EngineThe shipRuntime that runs containers on your machine
Container = Running app Image = Blueprint Dockerfile = Instructions Registry = Storage

2. Docker Architecture

The five core components of the Docker platform and how they interact.

Core Components

Docker Client

The CLI tool you interact with directly. When you type docker run or docker build, the client sends those instructions to the Docker Daemon via a REST API.

Binary: docker
Communicates with: Docker Daemon (dockerd)
Docker Daemon (dockerd)

The background service that does all the actual work — building images, running containers, managing networks and volumes. It listens for Docker API requests.

Process: dockerd
Manages: Images, containers, networks, volumes
Docker Images

Read-only templates made of stacked layers. Each instruction in a Dockerfile adds a new layer. Layers are cached and shared across images for efficiency.

Built from: Dockerfile
Stored in: Registry (Docker Hub, ECR, etc.)
Docker Containers

Running instances of images. Each container is an isolated process with its own filesystem, networking, and process tree — but shares the host OS kernel.

Created from: Image + docker run
Lifecycle: Created → Running → Stopped → Removed
Docker Registry

Storage and distribution system for Docker images. Docker Hub is the default public registry. Organizations run private registries (AWS ECR, GitHub Packages, Harbor).

Public: Docker Hub, GitHub Container Registry
Private: AWS ECR, Azure ACR, self-hosted Harbor

How Docker Works — Step by Step

  1. You write a Dockerfile — a recipe describing how to build your image
  2. docker build reads the Dockerfile and creates an Image (a stack of read-only layers)
  3. docker push uploads the image to a Registry (Docker Hub, ECR, etc.)
  4. On any server: docker pull downloads the image from the registry
  5. docker run starts a Container — a live, writable instance of the image
  6. Container runs your app in complete isolation with its own network and filesystem

Docker vs Virtual Machine

Virtual Machine
  • Full OS per VM (2–4 GB each)
  • Slow boot (1–2 minutes)
  • Heavy — needs a hypervisor (VMware, VirtualBox)
  • Isolated at hardware/OS level
  • Complete OS = large attack surface
Docker Container
  • Shares host OS kernel (10–200 MB)
  • Starts in milliseconds
  • Lightweight — no hypervisor needed
  • Isolated at process level (namespaces/cgroups)
  • Minimal image = small attack surface
AspectVirtual MachineDocker Container
Size2–4 GB10–200 MB
Startup Time1–2 minutesUnder 1 second
OSFull guest OS per VMShares host kernel
IsolationHardware-level (hypervisor)Process-level (namespaces + cgroups)
PerformanceNear-native with overheadNear-native
Best forFull OS isolation, legacy apps, different kernelsMicroservices, CI/CD, cloud-native, dev environments

3. Dockerfile — Building Your First Image

A Dockerfile is the recipe — Docker reads it top to bottom and builds an image one layer at a time.

Dockerfile Basics

A Dockerfile is a plain text file with instructions — Docker reads it top to bottom and builds an image layer by layer. Each instruction creates a new cached layer.
InstructionWhat it doesExample
FROMBase image to start from — every Dockerfile must begin with thisFROM openjdk:17-jre-slim
WORKDIRSet the working directory inside the image for subsequent commandsWORKDIR /app
COPYCopy files/folders from build context into the imageCOPY target/app.jar app.jar
RUNExecute a shell command during the image build stepRUN apt-get install -y curl
EXPOSEDocument which port the container listens on (metadata only)EXPOSE 8080
ENVSet an environment variable available at runtimeENV SPRING_PROFILES_ACTIVE=prod
CMDDefault command to run when container starts (can be overridden)CMD ["java", "-jar", "app.jar"]
ENTRYPOINTFixed command that always runs (CMD provides arguments)ENTRYPOINT ["java", "-jar"]
ARGBuild-time variable — only available during docker buildARG VERSION=1.0
LABELAdd metadata key=value pairs to the imageLABEL maintainer="dev@company.com"
HEALTHCHECKTell Docker how to test if the container is healthyHEALTHCHECK CMD curl -f http://localhost:8080/health
VOLUMECreate a mount point for persistent or shared dataVOLUME /data

Complete Spring Boot Dockerfile (Multi-Stage)

# Stage 1: Build — uses Maven + full JDK to compile FROM maven:3.9.6-eclipse-temurin-17 AS builder WORKDIR /build # Copy pom.xml first so dependency layer is cached separately COPY pom.xml . RUN mvn dependency:go-offline -B COPY src ./src RUN mvn clean package -DskipTests # Stage 2: Run — uses slim JRE only (no Maven, no source code) FROM eclipse-temurin:17-jre-slim WORKDIR /app # Add non-root user for security RUN addgroup --system appgroup && adduser --system --ingroup appgroup appuser # Copy only the built JAR from the builder stage COPY --from=builder /build/target/*.jar app.jar RUN chown appuser:appgroup app.jar USER appuser EXPOSE 8080 HEALTHCHECK --interval=30s --timeout=3s \ CMD curl -f http://localhost:8080/actuator/health || exit 1 ENTRYPOINT ["java", "-Djava.security.egd=file:/dev/./urandom", "-jar", "app.jar"]

Multi-Stage Builds — Why They Matter

Multi-stage builds: use one image to compile, copy only the output to a smaller runtime image — production image shrinks from 800 MB to 150 MB, with zero build tools exposed.
Single Stage (Naive)
  • Final image = 800 MB (Maven + full JDK + source code)
  • Contains build tools unnecessary in production
  • Source code exposed in final image
  • Slow pull and deploy times
  • Larger attack surface
Multi-Stage (Production)
  • Final image = 150 MB (JRE only + JAR)
  • Build tools left behind in the builder stage
  • Source code never reaches production image
  • Faster pull, smaller storage cost
  • Minimal attack surface

4. Essential Docker Commands

The commands you'll use every day — from pulling images to debugging running containers.

Image Commands

# Pull an image from Docker Hub docker pull nginx:latest # List all local images docker images # Build image from Dockerfile in current directory docker build -t myapp:1.0 . # Build with no cache (fresh build — ignores all cached layers) docker build --no-cache -t myapp:1.0 . # Tag an image with a new name/registry path docker tag myapp:1.0 myrepo/myapp:1.0 # Push to Docker Hub (must be logged in) docker push myrepo/myapp:1.0 # Remove a specific image docker rmi myapp:1.0 # Remove all unused images (dangling + unreferenced) docker image prune -a

Container Commands

# Run a container (foreground — blocks your terminal) docker run nginx:latest # Run detached (background) with name and port mapping host:container docker run -d --name my-nginx -p 8080:80 nginx:latest # Run with environment variable docker run -d -e SPRING_PROFILES_ACTIVE=prod myapp:1.0 # Run with volume mount (host path:container path) docker run -d -v /host/data:/container/data myapp:1.0 # List running containers docker ps # List all containers including stopped ones docker ps -a # View logs docker logs my-nginx docker logs -f my-nginx # follow / tail logs in real time # Execute command inside a running container (interactive bash) docker exec -it my-nginx bash # Stop / Start / Restart a named container docker stop my-nginx docker start my-nginx docker restart my-nginx # Remove a container (must be stopped first) docker rm my-nginx docker rm -f my-nginx # force remove a running container # Copy file to/from container docker cp my-nginx:/etc/nginx/nginx.conf ./nginx.conf docker cp ./nginx.conf my-nginx:/etc/nginx/nginx.conf

Inspect & Debug Commands

# Inspect full container config/state as JSON docker inspect my-nginx # Live CPU, memory, network, and disk usage for all containers docker stats # Show port mappings for a container docker port my-nginx # Show running processes inside a container docker top my-nginx # View image layer history (sizes and commands) docker history myapp:1.0 # Scan image for known CVEs (requires Docker Scout) docker scout cves myapp:1.0 # System-wide disk usage summary docker system df # Clean everything unused (images, containers, networks, build cache) docker system prune -a

5. Docker Networking

How containers communicate with each other and the outside world — five network driver types.

Network Types

TypeDescriptionUse Case
bridge Default driver. Creates an isolated virtual network on the host. Containers on the same bridge network can communicate using container names as hostnames. Local dev, single-host multi-container apps
host Container shares the host's network stack directly. No port mapping needed — container ports are the host ports. Performance-critical workloads, Linux only
none Container has no network interfaces. Completely isolated from all networking. Isolated batch jobs, security-sensitive processing
overlay Spans multiple Docker hosts using VXLAN tunneling. Requires Docker Swarm or Kubernetes orchestration. Docker Swarm services, cross-host communication
macvlan Assigns a unique MAC address to each container so it appears as a physical device on the network. Legacy apps that require a physical network presence

Network Commands

# List all networks on the host docker network ls # Create a custom named bridge network docker network create my-network # Run two containers on the same network # They can resolve each other by container name! docker run -d --network my-network --name app myapp:1.0 docker run -d --network my-network --name db postgres:15 # 'app' can now reach 'db' at hostname 'db' — no IPs needed # e.g., SPRING_DATASOURCE_URL=jdbc:postgresql://db:5432/myapp # Inspect a network (see connected containers, subnet, gateway) docker network inspect my-network # Connect an already-running container to a network docker network connect my-network my-container # Disconnect a container from a network docker network disconnect my-network my-container # Remove a network (all containers must be disconnected first) docker network rm my-network
Key Insight — DNS Resolution
Containers on the same user-defined bridge network automatically get DNS resolution by container name. The default bridge network does NOT support this — always create a named network for multi-container apps.

6. Docker Volumes (Persistent Data)

Containers are ephemeral — volumes are how you persist data beyond a container's lifetime.

Why Volumes?

Containers are ephemeral — when they stop or are removed, any data written inside the container filesystem is gone. Volumes persist data outside the container lifecycle.
Without a Volume
  • Write data to container filesystem
  • Container crashes or is removed
  • ALL data is permanently lost
  • Database wiped on every restart
  • Logs, uploads, state — all gone
With a Volume
  • Data stored in a Docker-managed volume
  • Container removed or replaced
  • Volume persists independently
  • New container mounts the same volume
  • Data intact — zero data loss on redeploy

Volume Types

TypeSyntaxUse Case
Named Volume -v my-vol:/data Databases (Postgres, MySQL, Redis). Docker manages the storage location on the host. Best for production data persistence.
Bind Mount -v /host/path:/container/path Development hot-reload (mount source code), config files, sharing files between host and container.
tmpfs Mount --tmpfs /tmp Temporary files, in-memory secrets, sensitive data that must never touch disk.

Volume Commands

# Create a named volume docker volume create my-data # Run postgres with persistent volume (data survives container removal) docker run -d \ --name postgres-db \ -e POSTGRES_PASSWORD=secret \ -v my-data:/var/lib/postgresql/data \ -p 5432:5432 \ postgres:15 # List all volumes docker volume ls # Inspect a volume (shows actual mount path on host) docker volume inspect my-data # Remove a specific volume (container must not be using it) docker volume rm my-data # Remove all unused volumes (WARNING: permanent data loss) docker volume prune # Bind mount example — mount current dir into container (dev hot-reload) docker run -d \ -v $(pwd)/src:/app/src \ -p 3000:3000 \ my-react-app:dev

7. Docker Compose

Define and run multi-container applications with a single YAML file.

What is Docker Compose?

Docker Compose lets you define and run multi-container apps in a single YAML file — one command starts everything in the right order with the right networking.
  • Without Compose: 5 separate docker run commands with long flags, manual network creation, manual volume setup, manual dependency ordering
  • With Compose: one docker-compose.yml file + docker-compose up -d — everything starts in the correct order with proper networking and volumes

Complete docker-compose.yml (Spring Boot + PostgreSQL + Redis + Nginx)

version: '3.8' services: # PostgreSQL Database db: image: postgres:15-alpine container_name: app-db environment: POSTGRES_DB: myapp POSTGRES_USER: appuser POSTGRES_PASSWORD: ${DB_PASSWORD} volumes: - postgres-data:/var/lib/postgresql/data - ./init.sql:/docker-entrypoint-initdb.d/init.sql networks: - backend healthcheck: test: ["CMD-SHELL", "pg_isready -U appuser -d myapp"] interval: 10s timeout: 5s retries: 5 # Redis Cache redis: image: redis:7-alpine container_name: app-redis command: redis-server --requirepass ${REDIS_PASSWORD} networks: - backend # Spring Boot Application app: build: context: . dockerfile: Dockerfile container_name: spring-app environment: SPRING_DATASOURCE_URL: jdbc:postgresql://db:5432/myapp SPRING_DATASOURCE_USERNAME: appuser SPRING_DATASOURCE_PASSWORD: ${DB_PASSWORD} SPRING_REDIS_HOST: redis SPRING_REDIS_PASSWORD: ${REDIS_PASSWORD} ports: - "8080:8080" depends_on: db: condition: service_healthy redis: condition: service_started networks: - backend - frontend restart: unless-stopped # Nginx Reverse Proxy nginx: image: nginx:alpine container_name: app-nginx ports: - "80:80" - "443:443" volumes: - ./nginx.conf:/etc/nginx/nginx.conf:ro - ./ssl:/etc/nginx/ssl:ro depends_on: - app networks: - frontend restart: unless-stopped networks: backend: driver: bridge frontend: driver: bridge volumes: postgres-data: driver: local

Docker Compose Commands

# Start all services in background (build if images don't exist) docker-compose up -d # Start and force rebuild all images docker-compose up -d --build # Stop all services (containers removed, volumes kept) docker-compose down # Stop and remove all volumes — WARNING: permanent data loss docker-compose down -v # View logs from all services (follow/tail) docker-compose logs -f # View logs from a specific service only docker-compose logs -f app # Scale a service to 3 replicas docker-compose up -d --scale app=3 # Execute a command in a running service container docker-compose exec app bash # List all services and their current status docker-compose ps # Restart a single service without touching others docker-compose restart app # Pull latest images for all services docker-compose pull # Validate and view the merged compose config docker-compose config

8. The Full Journey — Code to Docker to Deploy

The complete end-to-end path from a Spring Boot app on your laptop to a running container in production.

Step-by-Step: Spring Boot App to Running Container

  1. Write Spring Boot app — pom.xml, Application.java, controllers, services, repositories
  2. Build the JAR: mvn clean package — produces target/app-1.0.jar
  3. Write Dockerfile — multi-stage: builder stage (Maven + JDK 17) + run stage (JRE slim)
  4. Build the image: docker build -t myapp:1.0 . — Docker reads Dockerfile, creates image layers
  5. Test locally: docker run -p 8080:8080 myapp:1.0 — verify app runs correctly in a container
  6. Tag for registry: docker tag myapp:1.0 myrepo/myapp:1.0 — prepend registry path
  7. Push to registry: docker push myrepo/myapp:1.0 — image now available anywhere
  8. Deploy to server: docker pull myrepo/myapp:1.0 && docker run -d myrepo/myapp:1.0 — runs identically to local

Minimal Spring Boot App — Code Example

// Application.java — main entry point @SpringBootApplication public class Application { public static void main(String[] args) { SpringApplication.run(Application.class, args); } } // HelloController.java — simple REST endpoints @RestController public class HelloController { @GetMapping("/") public String hello() { return "Hello from Docker! Version: " + System.getenv().getOrDefault("APP_VERSION", "1.0"); } @GetMapping("/actuator/health") public Map<String, String> health() { return Map.of("status", "UP"); } }
Pro Tip — Health Checks
Always expose a /actuator/health endpoint (Spring Boot Actuator) and reference it in both your Dockerfile HEALTHCHECK and your Compose healthcheck config. This lets Docker and orchestrators know when your app is truly ready to serve traffic.

9. Docker Best Practices

Security, performance, and maintainability rules that separate production-grade Docker usage from development shortcuts.

Security

Security
  • Never run as root — add a non-root user in Dockerfile: RUN adduser --system appuser && USER appuser
  • Use official/verified base images — always specify exact version tags, never :latest in production
  • Scan images for CVEs — docker scout cves myapp:1.0 or integrate Trivy in your CI pipeline
  • Never hardcode secrets — use environment variables, Docker secrets, or a secrets manager (AWS Secrets Manager, Vault)
  • Minimize image size — smaller image = fewer packages = smaller attack surface
  • Use read-only filesystem where possible: docker run --read-only --tmpfs /tmp myapp:1.0

Performance & Size

Performance
  • Use Alpine-based images — nginx:alpine is 5 MB vs 140 MB for Debian-based; eclipse-temurin:17-jre-alpine is ~80 MB
  • Multi-stage builds — always separate build environment from runtime environment
  • Order instructions by change frequency — least-changing layers first (system packages, then dependencies, then source code)
  • Use .dockerignore — exclude node_modules/, target/, .git/, *.log to keep build context small
  • Combine RUN commands — chain with && to reduce layer count: RUN apt-get update && apt-get install -y curl && rm -rf /var/lib/apt/lists/*

.dockerignore File

Always create a .dockerignore file alongside your Dockerfile — it prevents large or sensitive files from being sent to the build context, speeding up builds significantly.

# Version control .git .gitignore .gitattributes # Build outputs — don't include in build context target/ build/ dist/ node_modules/ # Logs and temp files *.log *.tmp .DS_Store # Documentation *.md docs/ # Secrets — NEVER include these .env .env.local *.pem *.key credentials.json # IDE files .idea/ .vscode/ *.iml

Layer Caching — Order Matters

Cache Busting
Docker rebuilds a layer and all subsequent layers whenever any file in a COPY or RUN instruction changes. Put instructions that change rarely at the top, instructions that change often (like copying source code) at the bottom.
Bad Order (slow rebuilds)
  • COPY . . — copies everything first
  • RUN mvn dependency:go-offline
  • Every code change re-downloads all dependencies
Good Order (fast rebuilds)
  • COPY pom.xml . — pom rarely changes
  • RUN mvn dependency:go-offline — cached
  • COPY src ./src — only re-runs compile

10. Docker in CI/CD

Integrating Docker into a Jenkins pipeline — build, tag, push, and deploy automatically on every commit.

Jenkins + Docker Pipeline

In a typical CI/CD pipeline: Jenkins builds the JAR, builds a Docker image, pushes it to a registry, then deploys by stopping the old container and starting the new one — fully automated.
pipeline { agent any environment { DOCKER_IMAGE = "myrepo/myapp" DOCKER_TAG = "${BUILD_NUMBER}" REGISTRY_CREDS = credentials('docker-hub-creds') } stages { stage('Build JAR') { steps { sh 'mvn clean package -DskipTests' } } stage('Build Docker Image') { steps { sh "docker build -t ${DOCKER_IMAGE}:${DOCKER_TAG} ." sh "docker tag ${DOCKER_IMAGE}:${DOCKER_TAG} ${DOCKER_IMAGE}:latest" } } stage('Scan Image') { steps { // Fail build if critical CVEs found sh "docker scout cves --exit-code --only-severity critical ${DOCKER_IMAGE}:${DOCKER_TAG}" } } stage('Push to Registry') { steps { sh "echo ${REGISTRY_CREDS_PSW} | docker login -u ${REGISTRY_CREDS_USR} --password-stdin" sh "docker push ${DOCKER_IMAGE}:${DOCKER_TAG}" sh "docker push ${DOCKER_IMAGE}:latest" } } stage('Deploy') { steps { // Stop old container if running (|| true prevents failure if not running) sh "docker stop myapp || true" sh "docker rm myapp || true" // Start new container sh """docker run -d \ --name myapp \ -p 8080:8080 \ --restart unless-stopped \ -e SPRING_PROFILES_ACTIVE=prod \ ${DOCKER_IMAGE}:${DOCKER_TAG}""" } } } post { always { // Clean up local image to save disk space on Jenkins agent sh "docker rmi ${DOCKER_IMAGE}:${DOCKER_TAG} || true" sh "docker logout" } failure { mail to: 'team@company.com', subject: "Docker Build Failed: ${JOB_NAME} #${BUILD_NUMBER}", body: "Build ${BUILD_URL} failed. Check the logs." } } }

CI/CD Pipeline Summary

1. Code pushed to Git 2. Jenkins triggered 3. mvn package 4. docker build 5. Security scan 6. docker push 7. docker run (deploy)
Blue-Green & Rolling Deployments
For zero-downtime production deployments, use blue-green (run new version alongside old, switch traffic) or rolling updates (replace containers one at a time). These patterns are natively supported in Kubernetes and Docker Swarm. With plain Docker, use Nginx upstream switching or a load balancer.
