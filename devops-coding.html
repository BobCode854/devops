Task

Write a production-ready multi-stage Dockerfile for a Spring Boot app that produces the smallest possible image.

  • Multi-stage build (builder + runtime stages)
  • JRE-only runtime image (not JDK)
  • Non-root user for security
  • Health check via /actuator/health
  • EXPOSE correct port
Solution
# ── Stage 1: Build ──────────────────────────────────────────
FROM maven:3.9-eclipse-temurin-17 AS builder
WORKDIR /build

# Copy pom.xml first for dependency caching
COPY pom.xml .
RUN mvn dependency:go-offline -B

# Copy source and build JAR
COPY src ./src
RUN mvn clean package -DskipTests -B

# ── Stage 2: Runtime ────────────────────────────────────────
FROM eclipse-temurin:17-jre-slim AS runtime
WORKDIR /app

# Create non-root user
RUN addgroup --system appgroup && \
    adduser --system --ingroup appgroup appuser

# Copy only the built JAR from builder stage
COPY --from=builder /build/target/*.jar app.jar

# Set ownership
RUN chown appuser:appgroup app.jar

# Switch to non-root user
USER appuser

EXPOSE 8080

# Health check
HEALTHCHECK --interval=30s --timeout=10s --start-period=60s --retries=3 \
  CMD curl -f http://localhost:8080/actuator/health || exit 1

ENTRYPOINT ["java", "-jar", "app.jar"]
Key Concepts
  • Multi-stage: Builder stage has Maven + JDK; runtime stage has JRE only — final image is ~200MB instead of ~800MB
  • Dependency cache: Copying pom.xml before source means dependency layer is cached on code-only changes
  • Non-root: Running as appuser limits blast radius if the container is compromised
  • HEALTHCHECK: Docker/orchestrator can detect unhealthy containers and restart them
Pro Tip
Use eclipse-temurin:17-jre-slim rather than openjdk:17-jre — Temurin is the recommended OpenJDK distribution and -slim strips unnecessary OS packages.

  • Exec into the running container

docker stop myapp-container

Find and Fix All Bugs

The following Dockerfile has multiple serious problems. Identify and fix every bug:

# BROKEN Dockerfile — DO NOT USE AS-IS
FROM ubuntu:latest
RUN apt-get install java
COPY . /app
RUN cd /app && mvn package
CMD java -jar app.jar
Bugs Found
  • ubuntu:latest — unpinned tag breaks reproducibility; ubuntu is huge for a JVM app
  • apt-get install java — missing apt-get update first; no -y flag; no cleanup of apt cache
  • COPY . /app — copies everything including .git, IDE files; no WORKDIR set
  • RUN cd /app — cd in RUN does not persist; use WORKDIR
  • CMD java -jar app.jar — shell form; JAR path hardcoded; should use ENTRYPOINT with exec form
Fixed Dockerfile
# ── Stage 1: Build ──────────────────────────────────────────
FROM maven:3.9-eclipse-temurin-17 AS builder
WORKDIR /build
COPY pom.xml .
RUN mvn dependency:go-offline -B
COPY src ./src
RUN mvn clean package -DskipTests -B

# ── Stage 2: Runtime ────────────────────────────────────────
FROM eclipse-temurin:17-jre-slim
WORKDIR /app
RUN addgroup --system appgroup && adduser --system --ingroup appgroup appuser
COPY --from=builder /build/target/*.jar app.jar
RUN chown appuser:appgroup app.jar
USER appuser
EXPOSE 8080
ENTRYPOINT ["java", "-jar", "app.jar"]
ENTRYPOINT vs CMD
Use exec form ["java", "-jar", "app.jar"] — signals are forwarded correctly to the JVM. Shell form CMD java -jar spawns a /bin/sh -c wrapper that blocks signal propagation and causes slow shutdowns.

Task

Write docker-compose.yml for a 3-service stack:

  • Spring Boot app (port 8080)
  • PostgreSQL database (port 5432)
  • Redis cache (port 6379)
  • App must wait until PostgreSQL is healthy before starting
Solution
version: '3.9'

services:
  postgres:
    image: postgres:16-alpine
    container_name: postgres
    environment:
      POSTGRES_DB: mydb
      POSTGRES_USER: myuser
      POSTGRES_PASSWORD: mypassword
    volumes:
      - postgres_data:/var/lib/postgresql/data
    networks:
      - app-net
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U myuser -d mydb"]
      interval: 10s
      timeout: 5s
      retries: 5
      start_period: 30s
    restart: unless-stopped

  redis:
    image: redis:7-alpine
    container_name: redis
    volumes:
      - redis_data:/data
    networks:
      - app-net
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 10s
      timeout: 3s
      retries: 3
    restart: unless-stopped

  app:
    build: .
    container_name: spring-app
    ports:
      - "8080:8080"
    environment:
      SPRING_DATASOURCE_URL: jdbc:postgresql://postgres:5432/mydb
      SPRING_DATASOURCE_USERNAME: myuser
      SPRING_DATASOURCE_PASSWORD: mypassword
      SPRING_REDIS_HOST: redis
      SPRING_REDIS_PORT: 6379
    depends_on:
      postgres:
        condition: service_healthy
      redis:
        condition: service_healthy
    networks:
      - app-net
    restart: unless-stopped

volumes:
  postgres_data:
  redis_data:

networks:
  app-net:
    driver: bridge
Key Concepts
  • depends_on + condition: service_healthy — ensures app waits for actual DB readiness, not just container start
  • Named volumes — data persists across container restarts/upgrades
  • Custom network — service discovery by hostname (e.g., postgres, redis)
  • Environment variables — Spring Boot auto-wires DB/Redis from these env vars

Task

Create an app container and a db container that can communicate internally. The DB must not be reachable from outside the Docker host.

Solution
# 1. Create a custom bridge network
docker network create --driver bridge app-network

# 2. Run PostgreSQL — no -p flag = NOT exposed to host
docker run -d \
  --name postgres \
  --network app-network \
  -e POSTGRES_DB=mydb \
  -e POSTGRES_USER=myuser \
  -e POSTGRES_PASSWORD=secret \
  -v pg_data:/var/lib/postgresql/data \
  postgres:16-alpine

# 3. Run app — exposed on host:8080, also on app-network
docker run -d \
  --name spring-app \
  --network app-network \
  -p 8080:8080 \
  -e SPRING_DATASOURCE_URL=jdbc:postgresql://postgres:5432/mydb \
  -e SPRING_DATASOURCE_USERNAME=myuser \
  -e SPRING_DATASOURCE_PASSWORD=secret \
  myapp:v1

# Verify — app can reach DB by container name
docker exec spring-app ping -c 2 postgres

# Verify — DB is NOT reachable from host
curl localhost:5432  # Should fail — connection refused
Custom Bridge Network
Containers communicate by name. DB not exposed to host. Full isolation.
Default Bridge + -p on DB
DB accessible from internet. No name resolution. Security risk.

Task

Write docker-compose.yml with Nginx as a reverse proxy for 2 backend services. Neither backend should directly expose ports.

  • /api/users → users-service:8081
  • /api/orders → orders-service:8082
  • Only Nginx exposes port 80 to the host
docker-compose.yml
version: '3.9'

services:
  nginx:
    image: nginx:alpine
    container_name: nginx-proxy
    ports:
      - "80:80"
    volumes:
      - ./nginx.conf:/etc/nginx/nginx.conf:ro
    depends_on:
      - users-service
      - orders-service
    networks:
      - backend-net
    restart: unless-stopped

  users-service:
    image: users-service:latest
    container_name: users-service
    # No ports: — not exposed to host
    environment:
      SERVER_PORT: 8081
    networks:
      - backend-net
    restart: unless-stopped

  orders-service:
    image: orders-service:latest
    container_name: orders-service
    # No ports: — not exposed to host
    environment:
      SERVER_PORT: 8082
    networks:
      - backend-net
    restart: unless-stopped

networks:
  backend-net:
    driver: bridge
nginx.conf
events {
  worker_connections 1024;
}

http {
  upstream users_backend {
    server users-service:8081;
  }

  upstream orders_backend {
    server orders-service:8082;
  }

  server {
    listen 80;

    location /api/users {
      proxy_pass http://users_backend;
      proxy_set_header Host $host;
      proxy_set_header X-Real-IP $remote_addr;
      proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }

    location /api/orders {
      proxy_pass http://orders_backend;
      proxy_set_header Host $host;
      proxy_set_header X-Real-IP $remote_addr;
      proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }

    location /health {
      return 200 'OK';
      add_header Content-Type text/plain;
    }
  }
}
Note
Nginx uses Docker's internal DNS to resolve users-service and orders-service by container name within backend-net. The backends are unreachable from outside because no ports: key is set.

Task

Write a bash script that automates the full build-and-push workflow:

  • Tag image with current Git SHA
  • Authenticate with Docker Hub using env vars
  • Push the image
  • Clean up local image after push
build-and-push.sh
#!/bin/bash
set -euo pipefail

# ── Configuration ────────────────────────────────────────────
DOCKER_USERNAME="${DOCKER_USERNAME:?DOCKER_USERNAME env var required}"
DOCKER_PASSWORD="${DOCKER_PASSWORD:?DOCKER_PASSWORD env var required}"
IMAGE_NAME="${DOCKER_USERNAME}/myapp"
GIT_SHA=$(git rev-parse --short HEAD)
IMAGE_TAG="${IMAGE_NAME}:${GIT_SHA}"
LATEST_TAG="${IMAGE_NAME}:latest"

echo "==> Building image: ${IMAGE_TAG}"
docker build -t "${IMAGE_TAG}" -t "${LATEST_TAG}" .

echo "==> Logging into Docker Hub"
echo "${DOCKER_PASSWORD}" | docker login -u "${DOCKER_USERNAME}" --password-stdin

echo "==> Pushing ${IMAGE_TAG}"
docker push "${IMAGE_TAG}"

echo "==> Pushing ${LATEST_TAG}"
docker push "${LATEST_TAG}"

echo "==> Cleaning up local images"
docker rmi "${IMAGE_TAG}" "${LATEST_TAG}" || true

echo "==> Done! Image available at:"
echo "    docker pull ${IMAGE_TAG}"
Security Note
Never hardcode credentials. Use DOCKER_USERNAME and DOCKER_PASSWORD as environment variables, injected from a secrets manager or CI/CD secret store. The --password-stdin flag avoids credentials appearing in the process list.

Task

Rewrite this Dockerfile to maximize Docker layer cache hits. Currently every code change triggers a full Maven dependency download:

# SLOW — rebuilds everything on every change
FROM maven:3.9-eclipse-temurin-17
WORKDIR /app
COPY . .
RUN mvn clean package
CMD ["java", "-jar", "target/app.jar"]
Optimized — Cache-Friendly Dockerfile
# ── Stage 1: Build with maximized layer caching ──────────────
FROM maven:3.9-eclipse-temurin-17 AS builder
WORKDIR /build

# LAYER 1: pom.xml only — invalidated only when dependencies change
COPY pom.xml .

# LAYER 2: Download all dependencies — CACHED until pom.xml changes
RUN mvn dependency:go-offline -B

# LAYER 3: Source code — invalidated on every code change
COPY src ./src

# LAYER 4: Build — only re-runs when src or pom.xml changes
RUN mvn clean package -DskipTests -B

# ── Stage 2: Minimal runtime ─────────────────────────────────
FROM eclipse-temurin:17-jre-slim
WORKDIR /app
COPY --from=builder /build/target/*.jar app.jar
ENTRYPOINT ["java", "-jar", "app.jar"]
Cache Layer Strategy
  • Layer 1 (pom.xml): Only invalidated when you add/change a dependency — maybe once a sprint
  • Layer 2 (go-offline): All Maven deps downloaded once; served from cache on all subsequent builds
  • Layer 3 (src): Invalidated on every code change — but layers above are still cached
  • Result: Code-only changes skip dependency download — build time drops from ~3 min to ~20s
Rule of Thumb
Put instructions that change least often at the top, and instructions that change most often at the bottom. Docker invalidates all layers below any changed layer.

Task

Write a Deployment YAML: 3 replicas, Spring Boot app image myapp:1.0, port 8080, rolling update with maxUnavailable=1.

Solution
apiVersion: apps/v1
kind: Deployment
metadata:
  name: spring-app
  labels:
    app: spring-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: spring-app
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 1
      maxSurge: 1
  template:
    metadata:
      labels:
        app: spring-app
    spec:
      containers:
        - name: spring-app
          image: myapp:1.0
          ports:
            - containerPort: 8080
          resources:
            requests:
              memory: "256Mi"
              cpu: "250m"
            limits:
              memory: "512Mi"
              cpu: "500m"
Rolling Update Explained
  • maxUnavailable: 1 — at most 1 pod can be down during update (so 2 of 3 always serving)
  • maxSurge: 1 — can create 1 extra pod above desired (so up to 4 pods during rollout)
  • Zero-downtime deploy: new pods start, pass readiness, then old pods terminate

            pathType: Prefix
            backend:
              service:
                name: backend-svc
                port:
                  number: 80
          - path: /v2(/|$)(.*)
            pathType: Prefix
            backend:
              service:
                name: backend-v2-svc
                port:
                  number: 80

    - hosts:
        - api.myapp.com
      secretName: api-tls-cert

Task

Add both liveness and readiness probes to a Spring Boot Deployment:

  • Liveness: HTTP /actuator/health, delay 60s, period 20s
  • Readiness: HTTP /actuator/health/readiness, delay 30s, period 10s, failureThreshold 3
Solution
spec:
  containers:
    - name: spring-app
      image: myapp:1.0
      ports:
        - containerPort: 8080
      livenessProbe:
        httpGet:
          path: /actuator/health/liveness
          port: 8080
        initialDelaySeconds: 60   # Wait for JVM startup
        periodSeconds: 20          # Check every 20s
        timeoutSeconds: 5
        failureThreshold: 3        # Restart after 3 failures
        successThreshold: 1
      readinessProbe:
        httpGet:
          path: /actuator/health/readiness
          port: 8080
        initialDelaySeconds: 30   # Start checking earlier
        periodSeconds: 10
        timeoutSeconds: 3
        failureThreshold: 3        # Remove from LB after 3 failures
        successThreshold: 1
Liveness Probe
Is the container alive? Failure → restart the pod. Detects deadlocks, infinite loops.
Readiness Probe
Is the container ready to serve traffic? Failure → remove from Service endpoints. Does NOT restart. Detects DB unavailable, still warming up.
Spring Boot Actuator Setup
Add to application.properties: management.health.livenessstate.enabled=true and management.health.readinessstate.enabled=true. Spring Boot 2.3+ exposes separate liveness and readiness endpoints.

Task

Write a Jenkins pipeline that uses git diff to detect which microservices changed and only builds + deploys those — skipping unchanged services.

Solution
pipeline {
    agent any

    environment {
        SERVICES = 'user-service order-service inventory-service notification-service'
    }

    stages {
        stage('Detect Changes') {
            steps {
                script {
                    // Get list of changed files since last successful build
                    def changedFiles = sh(
                        script: 'git diff --name-only HEAD~1 HEAD',
                        returnStdout: true
                    ).trim().split('\n')

                    echo "Changed files: ${changedFiles}"

                    // Determine which services have changes
                    env.CHANGED_SERVICES = ''
                    for (service in env.SERVICES.split(' ')) {
                        def serviceChanged = changedFiles.any { it.startsWith("${service}/") }
                        if (serviceChanged) {
                            env.CHANGED_SERVICES += "${service} "
                            echo "Service changed: ${service}"
                        }
                    }

                    if (env.CHANGED_SERVICES.trim().isEmpty()) {
                        echo "No service changes detected — skipping build"
                        currentBuild.result = 'NOT_BUILT'
                    } else {
                        echo "Services to build: ${env.CHANGED_SERVICES}"
                    }
                }
            }
        }

        stage('Build Changed Services') {
            when {
                expression { env.CHANGED_SERVICES?.trim() }
            }
            steps {
                script {
                    def buildJobs = [:]

                    for (svc in env.CHANGED_SERVICES.trim().split(' ')) {
                        def service = svc  // Capture for closure
                        buildJobs[service] = {
                            dir(service) {
                                echo "Building ${service}"
                                sh 'mvn clean package -DskipTests -B'
                                sh "docker build -t myorg/${service}:${BUILD_NUMBER} ."
                                sh "docker push myorg/${service}:${BUILD_NUMBER}"
                            }
                        }
                    }

                    // Build all changed services in parallel
                    parallel buildJobs
                }
            }
        }

        stage('Deploy Changed Services') {
            when {
                allOf {
                    branch 'main'
                    expression { env.CHANGED_SERVICES?.trim() }
                }
            }
            steps {
                script {
                    for (svc in env.CHANGED_SERVICES.trim().split(' ')) {
                        sh """
                            kubectl set image deployment/${svc} \
                                ${svc}=myorg/${svc}:${BUILD_NUMBER} \
                                -n production
                            kubectl rollout status deployment/${svc} \
                                -n production --timeout=180s
                        """
                    }
                }
            }
        }
    }
}

This pattern is essential in monorepos with many microservices. Without change detection, every commit rebuilds all services — wasting 10-20 minutes per build. With change detection, only the 1-2 changed services are built, reducing CI time to under 3 minutes.

# ── Step 6: Wait for rollout ─────────────────────────────────
echo "[6/6] Waiting for rollout (timeout: 5 min)..."
kubectl rollout status "deployment/${APP_NAME}" \
    --namespace="${K8S_NAMESPACE}" \
    --timeout=300s

# ── Success ──────────────────────────────────────────────────

echo " DEPLOY SUCCESSFUL"

Error Handling Pattern
  • set -e — exit immediately if any command fails
  • set -u — exit if undefined variable is used
  • set -o pipefail — pipe returns the exit code of the first failed command
  • trap rollback ERR — rollback function runs automatically on any error

Task

Write a monitoring script that:

  • Polls the app health endpoint every 30 seconds
  • Logs result with timestamp to a file
  • Sends an alert via webhook if 3 consecutive failures occur
  • Resets the failure counter on success
health-monitor.sh
#!/bin/bash
set -euo pipefail

# ── Configuration ────────────────────────────────────────────
HEALTH_URL="${HEALTH_URL:?HEALTH_URL env var required}"
WEBHOOK_URL="${WEBHOOK_URL:-}"          # Slack/PagerDuty webhook
APP_NAME="${APP_NAME:-my-app}"
CHECK_INTERVAL="${CHECK_INTERVAL:-30}"  # seconds
FAILURE_THRESHOLD="${FAILURE_THRESHOLD:-3}"
LOG_FILE="${LOG_FILE:-/var/log/${APP_NAME}-health.log}"

# ── State ────────────────────────────────────────────────────
consecutive_failures=0

# ── Functions ────────────────────────────────────────────────
log() {
    local timestamp
    timestamp=$(date '+%Y-%m-%d %H:%M:%S')
    echo "[${timestamp}] $*" | tee -a "${LOG_FILE}"
}

send_alert() {
    local message="$1"
    log "ALERT: ${message}"
    if [[ -n "${WEBHOOK_URL}" ]]; then
        curl -s -X POST "${WEBHOOK_URL}" \
            -H 'Content-Type: application/json' \
            -d "{\"text\": \":red_circle: *HEALTH ALERT* — ${APP_NAME}\n${message}\"}" \
            || log "WARNING: Failed to send webhook alert"
    fi
}

check_health() {
    local http_code
    http_code=$(curl -s -o /dev/null -w "%{http_code}" \
        --max-time 10 \
        "${HEALTH_URL}" 2>/dev/null || echo "000")
    echo "${http_code}"
}

# ── Main monitoring loop ──────────────────────────────────────
log "Starting health monitor for ${APP_NAME}"
log "Health URL: ${HEALTH_URL}"
log "Check interval: ${CHECK_INTERVAL}s | Failure threshold: ${FAILURE_THRESHOLD}"

while true; do
    http_code=$(check_health)

    if [[ "${http_code}" == "200" ]]; then
        if [[ ${consecutive_failures} -gt 0 ]]; then
            log "RECOVERED after ${consecutive_failures} failures (HTTP 200)"
            send_alert "Service RECOVERED after ${consecutive_failures} consecutive failures"
        else
            log "OK (HTTP 200)"
        fi
        consecutive_failures=0
    else
        consecutive_failures=$((consecutive_failures + 1))
        log "UNHEALTHY (HTTP ${http_code}) — failure ${consecutive_failures}/${FAILURE_THRESHOLD}"

        if [[ ${consecutive_failures} -ge ${FAILURE_THRESHOLD} ]]; then
            send_alert "${consecutive_failures} consecutive health check failures!\nURL: ${HEALTH_URL}\nLast HTTP code: ${http_code}"
        fi
    fi

    sleep "${CHECK_INTERVAL}"
done
Running as a Service
Run as a systemd service or Docker container with restart policy always. Pass HEALTH_URL, WEBHOOK_URL, and APP_NAME as environment variables. Log rotation can be handled by logrotate on the host.

Task

Write a backup script that:

  • Dumps PostgreSQL database with pg_dump
  • Compresses the dump with gzip
  • Uploads to S3 with date in the filename
  • Deletes the local dump file to save disk space
  • Deletes S3 backups older than 30 days
  • Prints the final S3 path
backup.sh
#!/bin/bash
set -euo pipefail

# ── Configuration ────────────────────────────────────────────
PGHOST="${PGHOST:?PGHOST required}"
PGPORT="${PGPORT:-5432}"
PGDATABASE="${PGDATABASE:?PGDATABASE required}"
PGUSER="${PGUSER:?PGUSER required}"
PGPASSWORD="${PGPASSWORD:?PGPASSWORD required}"

S3_BUCKET="${S3_BUCKET:?S3_BUCKET required}"
AWS_REGION="${AWS_REGION:-us-west-2}"
RETENTION_DAYS="${RETENTION_DAYS:-30}"

TIMESTAMP=$(date '+%Y%m%d_%H%M%S')
DATE=$(date '+%Y/%m/%d')
DUMP_FILENAME="${PGDATABASE}_${TIMESTAMP}.sql.gz"
TMP_FILE="/tmp/${DUMP_FILENAME}"
S3_KEY="backups/${PGDATABASE}/${DATE}/${DUMP_FILENAME}"

export PGPASSWORD

# ── Trap: always clean up temp file ──────────────────────────
cleanup() {
    rm -f "${TMP_FILE}" || true
}
trap cleanup EXIT

# ── Step 1: Dump and compress ────────────────────────────────
echo "[1/4] Dumping database: ${PGDATABASE}@${PGHOST}:${PGPORT}"
pg_dump \
    --host="${PGHOST}" \
    --port="${PGPORT}" \
    --username="${PGUSER}" \
    --dbname="${PGDATABASE}" \
    --format=plain \
    --no-password \
    --verbose \
    2>/dev/null \
    | gzip > "${TMP_FILE}"

DUMP_SIZE=$(du -sh "${TMP_FILE}" | cut -f1)
echo "      Dump complete: ${TMP_FILE} (${DUMP_SIZE})"

# ── Step 2: Upload to S3 ─────────────────────────────────────
echo "[2/4] Uploading to s3://${S3_BUCKET}/${S3_KEY}"
aws s3 cp "${TMP_FILE}" "s3://${S3_BUCKET}/${S3_KEY}" \
    --region "${AWS_REGION}" \
    --storage-class STANDARD_IA \
    --metadata "database=${PGDATABASE},timestamp=${TIMESTAMP}"

# ── Step 3: Delete local dump (cleanup via trap) ─────────────
echo "[3/4] Removing local dump file"

# ── Step 4: Remove old S3 backups ────────────────────────────
echo "[4/4] Removing S3 backups older than ${RETENTION_DAYS} days"
CUTOFF_DATE=$(date -d "${RETENTION_DAYS} days ago" '+%Y-%m-%d' 2>/dev/null || \
              date -v "-${RETENTION_DAYS}d" '+%Y-%m-%d')

aws s3 ls "s3://${S3_BUCKET}/backups/${PGDATABASE}/" \
    --recursive \
    --region "${AWS_REGION}" \
    | awk '{print $4}' \
    | while read -r key; do
        file_date=$(echo "${key}" | grep -oP '\d{4}/\d{2}/\d{2}' | tr '/' '-' || true)
        if [[ -n "${file_date}" && "${file_date}" < "${CUTOFF_DATE}" ]]; then
            echo "    Deleting old backup: ${key}"
            aws s3 rm "s3://${S3_BUCKET}/${key}" --region "${AWS_REGION}"
        fi
    done

# ── Summary ──────────────────────────────────────────────────
echo ""
echo "=================================================="
echo " BACKUP SUCCESSFUL"
echo " Database:  ${PGDATABASE}"
echo " Size:      ${DUMP_SIZE}"
echo " S3 Path:   s3://${S3_BUCKET}/${S3_KEY}"
echo "=================================================="
Production Setup
  • Schedule via cron: 0 2 * * * /opt/scripts/backup.sh (2 AM daily)
  • Use S3 Lifecycle rules as a safety net on top of the script's retention logic
  • Store PGPASSWORD in AWS Secrets Manager — retrieve at runtime, not in crontab
  • STANDARD_IA storage class for infrequently accessed backups saves ~40% cost vs STANDARD
