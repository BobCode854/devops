1. What is Jenkins?

The world's most popular open-source automation server — the beating heart of modern CI/CD pipelines.

Jenkins is an open-source automation server that automates building, testing, and deploying software — the engine of your CI/CD pipeline.

Before Jenkins
  • Developer commits code
  • Manually runs tests
  • Manually builds JAR
  • Manually copies to server
  • Manually restarts app
  • 45 minutes of human work per deploy
With Jenkins
  • Developer commits code
  • Jenkins detects push via webhook
  • Auto-builds the project
  • Auto-runs all tests
  • Auto-deploys to server
  • Slack notification sent
  • 10 minutes, zero human effort

2. Jenkins Architecture

Understanding the master-agent model is key to scaling Jenkins horizontally for fast, parallel builds.

Core Components

Jenkins Master (Controller)

The brain of Jenkins. Manages build agents, schedules jobs, stores configuration, and serves the web UI. Never run actual builds on the master.

Port: 8080 (UI) & 50000 (agent comms)
Jenkins Agent (Worker)

Executes the actual build steps. Can run on Linux, Windows, or macOS. Connected to master via JNLP or SSH. Multiple agents = parallel builds.

Scales: Add more agents to increase throughput
Executor

A single build slot on an agent. An agent with 2 executors can run 2 builds simultaneously. The number of executors is configurable per agent.

Tip: Set executors = number of CPU cores
Job / Pipeline

The unit of work in Jenkins. A Freestyle Job is click-through config; a Declarative Pipeline is a Jenkinsfile defining all stages as code.

Prefer: Declarative Pipeline over Freestyle
Workspace

The directory on the agent where code is checked out and build commands run. Each job gets its own isolated workspace directory.

Best practice: Always call cleanWs() after builds
Plugin

Extends Jenkins functionality. The Git plugin, Maven plugin, Docker plugin, and Kubernetes plugin are all essential in modern pipelines.

Install via: Manage Jenkins → Plugin Manager

Master-Agent Data Flow

The master orchestrates; agents execute. Separating concerns lets you scale build capacity independently of the Jenkins controller.

  • GitHub sends a webhook push event to the Jenkins master at port 8080
  • Master receives the event, schedules a new build for the matching pipeline job
  • Master picks the first available agent (with a free executor slot)
  • Agent checks out the source code from Git into its workspace
  • Agent runs all Jenkinsfile stages (build, test, Docker, deploy) sequentially
  • Agent streams logs and results back to the master in real time
  • Master records the final build status, archives artifacts, triggers notifications

3. Installation

The fastest path to a working Jenkins instance — Docker-based setup with all dependencies included.

Docker-Based Install (Fastest)

Running Jenkins in Docker is the recommended approach for local dev and CI environments — no Java install required, fully isolated, portable.

# Run Jenkins in Docker — single command docker run -d \ --name jenkins \ -p 8080:8080 \ -p 50000:50000 \ -v jenkins-data:/var/jenkins_home \ -v /var/run/docker.sock:/var/run/docker.sock \ jenkins/jenkins:lts-jdk17 # Get the initial admin password (needed for first-time setup) docker exec jenkins cat /var/jenkins_home/secrets/initialAdminPassword # Access Jenkins at: http://localhost:8080 # Paste the password, install suggested plugins, create admin user
Why Mount docker.sock?

Mounting /var/run/docker.sock into the Jenkins container gives Jenkins access to the host Docker daemon. This lets your Jenkinsfile run docker build and docker push commands from inside the container — the "Docker outside of Docker" (DooD) pattern.

docker-compose.yml for Jenkins

version: '3.8' services: jenkins: image: jenkins/jenkins:lts-jdk17 container_name: jenkins privileged: true user: root ports: - "8080:8080" # Web UI - "50000:50000" # Agent communication volumes: - jenkins-data:/var/jenkins_home # Persist all Jenkins data - /var/run/docker.sock:/var/run/docker.sock # Docker access - /usr/bin/docker:/usr/bin/docker # Docker CLI binary environment: - JAVA_OPTS=-Djenkins.install.runSetupWizard=false restart: unless-stopped volumes: jenkins-data:

First-Time Setup Checklist

  • Unlock Jenkins: Paste initial admin password from /var/jenkins_home/secrets/initialAdminPassword
  • Install suggested plugins: Git, Pipeline, Maven, GitHub Integration, Credentials, Slack
  • Create admin user: Set username + strong password
  • Configure tools: Manage Jenkins → Tools → Add JDK 17, Maven 3.9
  • Add credentials: Manage Jenkins → Credentials → GitHub token, Docker Hub creds
  • Add GitHub webhook: Repo Settings → Webhooks → http://your-jenkins:8080/github-webhook/

4. Jenkinsfile — Pipeline as Code

Store your entire CI/CD pipeline definition in a Jenkinsfile alongside your application code — versioned, reviewed, and reproducible.

Pipeline Types: Freestyle vs Declarative

Freestyle Job — Legacy
  • Click-through GUI configuration
  • Not versioned in Git
  • Hard to code-review or audit
  • Basic use cases only
  • No stages, conditions, or parallel steps
Declarative Pipeline — Industry Standard
  • Jenkinsfile stored in Git repo
  • Code-reviewed, versioned, auditable
  • Powerful stages, conditions, parallel
  • Triggers, approvals, notifications built-in
  • Team-wide standard — clear structure

Complete Declarative Pipeline

A complete, production-ready Jenkinsfile covering every stage from checkout to production deployment with approval gates and notifications.

pipeline { agent any // Run on any available agent tools { maven 'Maven 3.9' // Use Maven configured in Jenkins tools jdk 'JDK 17' } environment { // Environment variables available to all stages APP_NAME = 'my-spring-app' DOCKER_REPO = 'myorg/myapp' DEPLOY_ENV = 'staging' } options { buildDiscarder(logRotator(numToKeepStr: '10')) timeout(time: 30, unit: 'MINUTES') disableConcurrentBuilds() } triggers { githubPush() // Trigger on GitHub push event cron('H 2 * * 1-5') // Nightly build Mon-Fri at 2am } stages { stage('Checkout') { steps { git branch: 'main', url: 'https://github.com/org/repo.git', credentialsId: 'github-token' } } stage('Build') { steps { sh 'mvn clean package -DskipTests -B' archiveArtifacts artifacts: 'target/*.jar', fingerprint: true } } stage('Unit Tests') { steps { sh 'mvn test' } post { always { junit 'target/surefire-reports/*.xml' publishHTML([ reportDir: 'target/site/jacoco', reportFiles: 'index.html', reportName: 'Code Coverage' ]) } } } stage('Code Quality') { steps { withSonarQubeEnv('SonarQube') { sh 'mvn sonar:sonar' } } } stage('Quality Gate') { steps { timeout(time: 5, unit: 'MINUTES') { waitForQualityGate abortPipeline: true } } } stage('Docker Build & Push') { when { branch 'main' // Only run on main branch } steps { script { def image = docker.build("${DOCKER_REPO}:${BUILD_NUMBER}") docker.withRegistry('https://registry.hub.docker.com', 'docker-hub-creds') { image.push("${BUILD_NUMBER}") image.push("latest") } } } } stage('Deploy to Staging') { when { branch 'main' } steps { sshPublisher(publishers: [ sshPublisherDesc( configName: 'staging-server', transfers: [ sshTransfer( execCommand: """ docker pull ${DOCKER_REPO}:${BUILD_NUMBER} docker stop app || true docker rm app || true docker run -d --name app -p 8080:8080 ${DOCKER_REPO}:${BUILD_NUMBER} """ ) ] ) ]) } } stage('Integration Tests') { steps { sh 'mvn verify -P integration-tests -Dapp.url=http://staging:8080' } } stage('Deploy to Production') { when { branch 'main' } input { message "Deploy to production?" ok "Deploy" submitter "admin,lead-dev" // Only these users can approve } steps { sh """ kubectl set image deployment/myapp \ app=${DOCKER_REPO}:${BUILD_NUMBER} \ --record kubectl rollout status deployment/myapp """ } } } post { success { slackSend( channel: '#deployments', color: 'good', message: "Build ${BUILD_NUMBER} deployed successfully! ✅" ) } failure { slackSend( channel: '#deployments', color: 'danger', message: "Build ${BUILD_NUMBER} FAILED! ❌ ${BUILD_URL}" ) emailext( to: 'dev-team@company.com', subject: "Build Failed: ${JOB_NAME} #${BUILD_NUMBER}", body: "Check: ${BUILD_URL}" ) } always { cleanWs() // Always clean workspace to free disk space } } }

Key Jenkinsfile Directives

DirectivePurpose
agent anyRun on any available agent. Use agent { label 'linux' } to target a specific agent type.
toolsPre-install JDK, Maven, Gradle etc. from the tool configuration in Jenkins settings.
environmentDefine environment variables available to all stages. Can reference Jenkins credentials.
optionsBuild-level options: timeout, log rotation, concurrent build prevention, ANSI color.
triggersAuto-trigger: githubPush(), cron(), pollSCM(), upstream().
whenConditional stage execution: branch 'main', expression { ... }, environment name: 'X'.
inputPause pipeline and require human approval before proceeding. Specify allowed submitters.
parallelRun multiple stages concurrently to speed up the pipeline (e.g., unit tests + lint + security scan).
postRuns after stages: always, success, failure, unstable, changed.
BUILD_NUMBERBuilt-in variable: sequential build counter. Others: JOB_NAME, BUILD_URL, WORKSPACE, GIT_COMMIT.

5. The Full Journey — Java Code → JAR → Docker → Deploy

End-to-end walkthrough of exactly what happens from the moment a developer types git push to when the new version is live in production.

Eight concrete steps with real code — this is the complete pipeline story that ties everything together.

Step 1: Developer Writes Code and Pushes to GitHub

# Developer writes a new feature and pushes to the main branch git add . git commit -m "feat: add user registration endpoint" git push origin main # GitHub immediately fires a webhook POST to Jenkins

Step 2: GitHub Sends Webhook to Jenkins

  • GitHub → Repository Settings → Webhooks → Add webhook
  • URL: http://your-jenkins:8080/github-webhook/
  • Content type: application/json
  • Event: "Just the push event" — fires on every git push
  • Jenkins receives the payload and immediately starts the pipeline

Step 3: Jenkins Checks Out the Code

stage('Checkout') { steps { git branch: 'main', url: 'https://github.com/org/repo.git', credentialsId: 'github-token' // Code is now in the agent's workspace directory // Jenkins sets GIT_COMMIT, GIT_BRANCH environment variables } }

Step 4: Maven Builds the JAR

stage('Build JAR') { steps { sh 'mvn clean package -DskipTests -B' // What mvn clean package does: // 1. clean — deletes target/ from previous builds // 2. compile — compiles all .java → .class files // 3. resources — copies src/main/resources into target/ // 4. package — bundles everything into target/myapp-1.0.0.jar // Result: target/myapp-1.0.0.jar is ready } }
Why -DskipTests?

We skip tests during the build stage to fail fast on compilation errors. Tests run as a separate dedicated stage immediately after, so test results are cleanly separated from build results in the Jenkins UI and reports.

Step 5: Tests Run and Results Published

stage('Test') { steps { sh 'mvn test' // JUnit runs all @Test methods in src/test/java/ // Results written to target/surefire-reports/*.xml // JaCoCo coverage report at target/site/jacoco/ } post { always { // Publish test results as a trend graph in Jenkins UI junit 'target/surefire-reports/*.xml' // If a test fails, the build is marked UNSTABLE (yellow) } } }

Step 6: Docker Image Built from the JAR

stage('Docker Build') { steps { sh "docker build -t myapp:${BUILD_NUMBER} ." // Jenkins reads the Dockerfile in the repo root // Dockerfile copies the JAR and wraps it in a JRE container: } } // The Dockerfile that Jenkins uses: FROM eclipse-temurin:17-jre-alpine WORKDIR /app COPY target/myapp-*.jar app.jar EXPOSE 8080 ENTRYPOINT ["java", "-jar", "app.jar"]

Step 7: Image Pushed to Docker Hub / AWS ECR

stage('Push Image') { steps { withCredentials([usernamePassword( credentialsId: 'docker-hub', usernameVariable: 'DOCKER_USER', passwordVariable: 'DOCKER_PASS' )]) { sh """ echo $DOCKER_PASS | docker login -u $DOCKER_USER --password-stdin docker tag myapp:${BUILD_NUMBER} myorg/myapp:${BUILD_NUMBER} docker tag myapp:${BUILD_NUMBER} myorg/myapp:latest docker push myorg/myapp:${BUILD_NUMBER} docker push myorg/myapp:latest """ } // Credentials are injected as env vars — never printed in logs // Jenkins masks the password value in all console output } }

Step 8: Deploy to Server / Kubernetes

stage('Deploy') { steps { sh """ # Update Kubernetes deployment with the new image tag kubectl set image deployment/myapp \ app=myorg/myapp:${BUILD_NUMBER} \ --record # Wait for the rollout to complete (timeout = 5 min) kubectl rollout status deployment/myapp --timeout=300s # Verify the new pods are healthy kubectl get pods -l app=myapp """ } post { failure { // Auto-rollback if rollout fails sh 'kubectl rollout undo deployment/myapp' } } }
The Complete Flow Summary
  • git push → webhook fires → Jenkins starts build
  • mvn clean package → target/myapp.jar produced
  • mvn test → JUnit results published to Jenkins UI
  • docker build → JAR wrapped in JRE container image
  • docker push → image uploaded to registry with build tag
  • kubectl set image → Kubernetes rolls out zero-downtime update
  • Slack notification → team knows immediately whether it passed or failed

6. Jenkins Shared Libraries

Eliminate duplicated pipeline code across dozens of microservices by extracting common logic into a shared, versioned library.

Shared Libraries let you reuse pipeline code across all your Jenkins jobs — define once in a separate repo, import in every Jenkinsfile.

Without Shared Libraries
  • 30 microservices, 30 near-identical Jenkinsfiles
  • Change Docker registry? Edit 30 files
  • Bug in notification logic? Fix 30 places
  • Copy-paste errors accumulate over time
With Shared Libraries
  • Define buildAndDeploy() once in the library
  • All 30 Jenkinsfiles call the same function
  • Update the library → all pipelines update automatically
  • Central quality and security standards enforced

Shared Library Repository Structure

jenkins-shared-library/ # A separate GitHub repo ├── vars/ │ ├── buildJar.groovy # Global function: called as buildJar() │ ├── buildDockerImage.groovy # Called as buildDockerImage() │ ├── deployToK8s.groovy # Called as deployToK8s() │ └── buildAndDeploy.groovy # Full pipeline in one call ├── src/ │ └── com/company/ │ ├── PipelineUtils.groovy # Utility class (must be in src/) │ └── SlackNotifier.groovy └── resources/ └── config.yaml # Shared config files

Shared Library Function Definition

// vars/buildAndDeploy.groovy — complete pipeline as a reusable function def call(Map config) { pipeline { agent any environment { DOCKER_IMAGE = "${config.registry ?: 'docker.io'}/${config.image}" } stages { stage('Build') { steps { sh "mvn clean package -DskipTests -B" } } stage('Test') { steps { sh "mvn test" } post { always { junit 'target/surefire-reports/*.xml' } } } stage('Docker Build & Push') { steps { sh "docker build -t ${DOCKER_IMAGE}:${BUILD_NUMBER} ." sh "docker push ${DOCKER_IMAGE}:${BUILD_NUMBER}" } } stage('Deploy') { when { expression { config.deploy == true } } steps { sh "kubectl set image deployment/${config.app} app=${DOCKER_IMAGE}:${BUILD_NUMBER}" sh "kubectl rollout status deployment/${config.app}" } } } post { failure { slackSend(channel: "${config.slackChannel ?: '#ci-alerts'}", color: 'danger', message: "FAILED: ${JOB_NAME} #${BUILD_NUMBER}") } always { cleanWs() } } } }

Using the Shared Library in a Jenkinsfile

// Jenkinsfile in your microservice repo — just 8 lines! @Library('company-shared-lib@main') _ // Import the library from GitHub buildAndDeploy( // Call the shared function image: 'myorg/user-service', app: 'user-service', slackChannel: '#user-service-deploys', deploy: true )

Register the Library in Jenkins

  • Go to Manage Jenkins → System → Global Pipeline Libraries
  • Name: company-shared-lib (matches the @Library annotation)
  • Source: GitHub repository URL of your shared library
  • Credentials: your GitHub token for private repos
  • Default version: main — or pin to a tag for stability

7. Jenkins Credentials Management

Never hardcode secrets. Jenkins provides an encrypted credential store with fine-grained access control — use it for everything sensitive.

Jenkins encrypts all credentials at rest using the master key. Credentials are injected as environment variables at runtime and automatically masked in all console output.

Credential Types

TypeUse Case
Username / PasswordDocker Hub login, GitHub HTTPS, Nexus, Artifactory, database credentials
Secret TextAPI tokens, Slack webhook URLs, SonarQube token, any single-value secret
SSH Private KeyDeploying to Linux servers via SSH, GitHub SSH access, agent connection
CertificateSSL/TLS client certificates for mutual TLS authentication
AWS CredentialsAWS Access Key ID + Secret Access Key for ECR, EKS, S3, CodeDeploy
Secret Filekubeconfig file, .p12 certificate, any file-based secret

Using Credentials in Pipelines

// Inject multiple credential types into a single block withCredentials([ usernamePassword( credentialsId: 'github-token', usernameVariable: 'GH_USER', passwordVariable: 'GH_TOKEN' ), string( credentialsId: 'sonar-token', variable: 'SONAR_TOKEN' ), sshUserPrivateKey( credentialsId: 'prod-server-key', keyFileVariable: 'SSH_KEY', usernameVariable: 'SSH_USER' ), file( credentialsId: 'kubeconfig', variable: 'KUBE_CONFIG' ) ]) { // All variables available here — values masked in logs sh "git clone https://${GH_USER}:${GH_TOKEN}@github.com/org/repo.git" sh "mvn sonar:sonar -Dsonar.login=${SONAR_TOKEN}" sh "ssh -i ${SSH_KEY} ${SSH_USER}@prod-server 'docker pull myapp:latest'" sh "kubectl --kubeconfig=${KUBE_CONFIG} get pods" } // Variables are automatically cleaned up and unset after the block

Credential Scopes

Global Scope
  • Available to ALL jobs and pipelines on Jenkins
  • Best for: shared Docker Hub creds, org-wide GitHub token
  • Location: Manage Jenkins → Credentials → Global
Folder / Project Scope
  • Available only to jobs within that folder
  • Best for: team-specific or project-specific secrets
  • Least-privilege principle — isolate credentials by team
Security Anti-Patterns to Avoid
  • Never store passwords directly in environment variables in the Jenkinsfile — anyone with read access to the repo can see them
  • Never echo or print credential variables in shell scripts — Jenkins will try to mask them but this is fragile
  • Never commit a .env file or kubeconfig file into your repository

8. Jenkins Best Practices

Patterns that keep your Jenkins setup fast, secure, and maintainable at scale.

DO — Green Light Practices
  • Use Declarative Pipeline over Freestyle jobs
  • Store Jenkinsfile in the application repo next to code
  • Use Shared Libraries for logic reused across jobs
  • Pin tool versions (Maven 3.9, JDK 17) in Jenkins tools config
  • Never hardcode passwords — always use Jenkins Credentials
  • Parallelize independent stages (tests, lint, security scan)
  • Use post { always { cleanWs() } } to clean workspace
  • Set build timeouts to prevent hung builds consuming resources
  • Use log rotation to discard old builds and save disk space
  • Monitor Jenkins with Prometheus + Grafana for capacity planning
DON'T — Red Flag Anti-Patterns
  • Don't use sudo in pipelines — run as non-root user
  • Don't store secrets in Jenkinsfile or plain-text env vars
  • Don't run all stages serially when they can run in parallel
  • Don't skip junit in post — losing test trends kills visibility
  • Don't run 50 builds for the same commit — use disableConcurrentBuilds()
  • Don't let workspaces accumulate — disk fills up silently
  • Don't run builds on the master node — keep master for orchestration only
  • Don't ignore flaky tests — fix them or they erode trust in the pipeline
  • Don't install hundreds of plugins — each is an attack surface and upgrade burden

Parallel Stages — Speed Up Your Pipeline

Independent stages run in parallel to cut total pipeline time. Unit tests, integration tests, and security scans can all run at the same time.

stage('Tests & Quality') { parallel { stage('Unit Tests') { steps { sh 'mvn test -pl module-a -B' } post { always { junit 'module-a/target/surefire-reports/*.xml' } } } stage('Integration Tests') { steps { sh 'mvn verify -P integration -pl module-b -B' } } stage('Security Scan (OWASP)') { steps { sh 'mvn org.owasp:dependency-check-maven:check' } } stage('Code Style') { steps { sh 'mvn checkstyle:check' } } } } // Without parallel: 4 stages × 5 min each = 20 min total // With parallel: all 4 run concurrently = ~5 min total

Interview Gold — Key Concepts to Know Cold
  • What is a Jenkinsfile? — A Pipeline as Code definition stored in the repo, using Groovy DSL, defining all CI/CD stages.
  • Difference: Declarative vs Scripted? — Declarative has fixed structure, easier syntax, better error messages. Scripted is pure Groovy, more flexible but harder to read.
  • What is a Jenkins agent? — A separate machine that runs the actual build steps, managed by the Jenkins master controller.
  • How do secrets work in Jenkins? — Stored encrypted in the credential store, injected at runtime via withCredentials(), masked in console output.
  • What triggers a Jenkins build? — GitHub webhook, cron schedule, SCM polling, manual trigger, or an upstream job completing.
  • What is a Shared Library? — A separate Git repo of reusable Groovy functions that any Jenkinsfile can import and call.
