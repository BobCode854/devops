1. What is AWS and Why Cloud?

Understanding the core problem cloud solves and why AWS is the dominant choice for DevOps teams worldwide.

The Problem With On-Premise Servers

Before cloud: you had to buy physical servers, rack them, power them, cool them, and manage them 24/7. One traffic spike = server down.
On-Premise
  • Buy servers upfront ($50,000+)
  • Fixed capacity — over-provision or crash
  • Weeks to provision a new server
  • You manage hardware, OS, network
  • Datacenter costs: rent, power, cooling
  • Server breaks → your team fixes at 3am
AWS Cloud
  • Pay per hour (EC2 t3.micro = $0.01/hr)
  • Scale up in minutes, scale down to zero
  • New server in 90 seconds
  • AWS manages physical hardware
  • No datacenter costs
  • AWS handles hardware failures automatically

Core AWS Concepts

Before diving into services, understand the foundational concepts every DevOps engineer must know.

Availability Zone (AZ) Isolated datacenter within a region; deploy across 2+ AZs for high availability

IAM Identity & Access Management — who can do what in your AWS account

2. AWS Services Every DevOps Engineer Uses

AWS has 200+ services. These are the ones you'll use every day as a DevOps engineer.

Compute


EC2 — Elastic Compute Cloud

Virtual machines. You pick instance type (CPU/RAM). Run anything. SSH in, deploy manually or with automation tools. Choose from 500+ instance types.

ECS — Elastic Container Service

AWS-managed Docker container orchestration. Simpler than K8s for small teams. Two launch types: EC2 (you manage nodes) or Fargate (serverless).

EKS — Elastic Kubernetes Service

Managed Kubernetes — AWS runs the control plane (API server, etcd). You manage worker nodes. Drop-in compatible with standard kubectl.

Lambda — Serverless Functions

Run code without managing servers. Pay per invocation (first 1M free/month). Great for event-driven tasks, webhooks, scheduled jobs.

Fargate — Serverless Containers

Run ECS or EKS pods without managing EC2 nodes. No patching, no capacity planning. Pay per vCPU and memory used by your containers.

Storage

Service What it Is & When to Use It
S3 Object storage. Store any file. 99.999999999% durability. Used for artifacts, static sites, backups, data lakes. Infinite scale.
EBS Block storage. Attached to EC2 like a hard drive. Fast, persistent. Used for OS volumes, databases. Up to 64 TB per volume.
EFS Network file system. Shared between multiple EC2 instances simultaneously. Like NFS in the cloud. Auto-scales storage.
ECR Elastic Container Registry. Private Docker image registry. Push/pull images for ECS/EKS deployments. Integrated with IAM for access control.

Networking

Service Description
VPC Virtual Private Cloud — your isolated, private network in AWS. Define IP ranges, subnets, route tables.
Subnets Public (internet-accessible via Internet Gateway) vs Private (internal only, no direct internet). Always put databases in private subnets.
Security Groups Stateful firewall rules for EC2/RDS/etc — allow/deny by port + source IP. Acts like a per-resource firewall.
ALB Application Load Balancer — distribute HTTP/HTTPS traffic across EC2 instances or containers. Supports path-based routing, health checks, SSL termination.
Route 53 AWS DNS service — map domain names to AWS resources. Supports weighted routing, latency-based routing, failover.
CloudFront CDN — cache and serve content globally from 450+ edge locations. Dramatically reduces latency for users worldwide.

AWS Native CI/CD

AWS offers a complete native CI/CD toolchain. Many teams use Jenkins + ECR + EKS instead, but knowing these is essential.

Service Description
CodeCommit Managed Git repository (like GitHub, AWS-hosted). Integrates with IAM for access control.
CodeBuild Run build commands — like Jenkins agents, fully managed by AWS. Pay per build minute. Docker-in-Docker supported.
CodeDeploy Deploy to EC2, Lambda, or ECS. Supports blue/green deployments, rolling updates, canary releases.
CodePipeline Orchestrate the full CI/CD pipeline (Source → Build → Test → Deploy). Visual pipeline editor. Triggers on code commits.
ECR Docker image registry — the bridge between your CI system and ECS/EKS container deployments.

Database Services

Service Description & Use Case
RDS Managed relational DB: Postgres, MySQL, MariaDB, Oracle, SQL Server. Automated backups, Multi-AZ for HA, read replicas for scaling. No SSH access needed.
Aurora AWS-optimized MySQL/Postgres-compatible engine. 5x faster than standard MySQL. Auto-scales storage from 10 GB to 128 TB. Serverless option available.
DynamoDB Managed NoSQL (key-value + document). Single-digit millisecond latency at any scale. Auto-scales read/write capacity. Great for session storage, leaderboards.
ElastiCache Managed Redis or Memcached. In-memory caching to reduce database load. Sub-millisecond latency. Use for session caching, rate limiting, leaderboards.

3. AWS CLI — Essential Commands

The AWS CLI is your daily driver for interacting with AWS services from the terminal and inside CI/CD pipelines.

Install: pip install awscli or download from aws.amazon.com/cli. Configure once with aws configure — it stores credentials in ~/.aws/credentials. 
# ===== SETUP =====
aws configure
# Enter: Access Key ID, Secret Access Key, Region (us-east-1), Output (json)

# Verify your identity
aws sts get-caller-identity

# ===== EC2 =====
# List instances with ID, state, and public IP
aws ec2 describe-instances \
  --query 'Reservations[].Instances[].[InstanceId,State.Name,PublicIpAddress]' \
  --output table

# Start / Stop instance
aws ec2 start-instances --instance-ids i-1234567890abcdef0
aws ec2 stop-instances --instance-ids i-1234567890abcdef0

# ===== S3 =====
# List all buckets
aws s3 ls

# Upload a single file
aws s3 cp target/app.jar s3://my-artifacts-bucket/releases/app-1.0.jar

# Sync an entire folder (--delete removes files in S3 not in source)
aws s3 sync ./dist s3://my-static-website/ --delete

# Download from S3
aws s3 cp s3://my-artifacts-bucket/releases/app-1.0.jar ./app.jar

# ===== ECR =====
# Authenticate Docker to ECR (token valid for 12 hours)
aws ecr get-login-password --region us-east-1 | \
  docker login --username AWS --password-stdin \
  123456789.dkr.ecr.us-east-1.amazonaws.com

# Create a new ECR repository
aws ecr create-repository --repository-name my-spring-app

# Tag local image for ECR
docker tag myapp:1.0 123456789.dkr.ecr.us-east-1.amazonaws.com/my-spring-app:1.0

# Push image to ECR
docker push 123456789.dkr.ecr.us-east-1.amazonaws.com/my-spring-app:1.0

# List images in a repository
aws ecr list-images --repository-name my-spring-app

# ===== EKS =====
# Add cluster to local kubeconfig
aws eks update-kubeconfig --region us-east-1 --name my-eks-cluster

# Now standard kubectl commands work against your EKS cluster
kubectl get nodes
kubectl get pods -A
kubectl get services -n production

Common CLI Flags & Tricks

Flag / Pattern What It Does
--output table Pretty-print output as a table — much more readable than raw JSON
--output text Tab-separated plain text — great for scripting and piping to awk
--query '...' JMESPath query to filter/reshape JSON output — learn this, it saves huge time
--profile myprofile Use a named profile from ~/.aws/config — switch between accounts easily
--region us-west-2 Override the default region for this command
--dry-run Check if you have permission to run a command without actually running it (EC2 commands)

4. Deploying to AWS EKS — The Full Picture

The most common production deployment pattern: Jenkins builds your app, pushes to ECR, and deploys to EKS with zero downtime.

Architecture Overview

EKS = managed Kubernetes on AWS. AWS runs the control plane. You manage worker nodes (EC2 or Fargate). ECR stores your Docker images. ALB routes external traffic to your pods.
AWS Manages (Control Plane)
  • Kubernetes API server
  • etcd (cluster state store)
  • Controller manager
  • Scheduler
  • Control plane HA across AZs
  • Control plane upgrades
You Manage (Data Plane)
  • Worker nodes (EC2 instances)
  • Node groups (managed or self-managed)
  • Kubernetes workloads (Deployments, Services)
  • Persistent volumes (EBS/EFS)
  • Ingress controller (ALB Ingress)
  • Node OS patching (or use Fargate)

Step-by-Step: Jenkins → ECR → EKS

The complete automated flow from code push to production deployment:

  1. Developer pushes code to GitHub — triggers a webhook to Jenkins
  2. Jenkins webhook fires and starts a new pipeline build
  3. Maven builds the JAR: mvn clean package
  4. Docker image built: docker build -t myapp:${BUILD_NUMBER} .
  5. Login to ECR: aws ecr get-login-password | docker login --username AWS --password-stdin
  6. Push image to ECR: docker push 123456789.dkr.ecr.us-east-1.amazonaws.com/myapp:${BUILD_NUMBER}
  7. Update Kubernetes deployment: kubectl set image deployment/myapp app=ECR_IMAGE:${BUILD_NUMBER}
  8. Kubernetes performs a rolling update — old pods replaced one by one with the new image
  9. Readiness probes verify each new pod is healthy before moving to the next
  10. ALB routes traffic to new pods — zero downtime throughout the entire process

Complete Jenkinsfile for AWS ECR + EKS

Production-Ready Pipeline
pipeline {
    agent any
    environment {
        AWS_REGION     = 'us-east-1'
        ECR_REGISTRY   = '123456789012.dkr.ecr.us-east-1.amazonaws.com'
        ECR_REPO       = 'my-spring-app'
        K8S_NAMESPACE  = 'production'
        K8S_DEPLOYMENT = 'spring-app'
    }
    stages {
        stage('Build JAR') {
            steps {
                sh 'mvn clean package -DskipTests -B'
            }
        }
        stage('Unit Tests') {
            steps {
                sh 'mvn test'
            }
            post {
                always { junit 'target/surefire-reports/*.xml' }
            }
        }
        stage('Build Docker Image') {
            steps {
                sh "docker build -t ${ECR_REGISTRY}/${ECR_REPO}:${BUILD_NUMBER} ."
                sh "docker tag ${ECR_REGISTRY}/${ECR_REPO}:${BUILD_NUMBER} ${ECR_REGISTRY}/${ECR_REPO}:latest"
            }
        }
        stage('Push to ECR') {
            steps {
                withCredentials([[$class: 'AmazonWebServicesCredentialsBinding',
                                   credentialsId: 'aws-credentials']]) {
                    sh """
                        aws ecr get-login-password --region ${AWS_REGION} | \\
                        docker login --username AWS --password-stdin ${ECR_REGISTRY}
                        docker push ${ECR_REGISTRY}/${ECR_REPO}:${BUILD_NUMBER}
                        docker push ${ECR_REGISTRY}/${ECR_REPO}:latest
                    """
                }
            }
        }
        stage('Deploy to EKS') {
            steps {
                withCredentials([[$class: 'AmazonWebServicesCredentialsBinding',
                                   credentialsId: 'aws-credentials']]) {
                    sh """
                        aws eks update-kubeconfig --region ${AWS_REGION} --name my-eks-cluster
                        kubectl set image deployment/${K8S_DEPLOYMENT} \\
                            app=${ECR_REGISTRY}/${ECR_REPO}:${BUILD_NUMBER} \\
                            -n ${K8S_NAMESPACE}
                        kubectl rollout status deployment/${K8S_DEPLOYMENT} \\
                            -n ${K8S_NAMESPACE} --timeout=5m
                    """
                }
            }
        }
    }
    post {
        success {
            slackSend color: 'good',
                      message: "Deployed ${ECR_REPO}:${BUILD_NUMBER} to EKS"
        }
        failure {
            slackSend color: 'danger',
                      message: "FAILED: ${ECR_REPO}:${BUILD_NUMBER}"
        }
        always {
            sh "docker rmi ${ECR_REGISTRY}/${ECR_REPO}:${BUILD_NUMBER} || true"
            cleanWs()
        }
    }
}

5. IAM — Security in AWS

IAM is the most important service to understand. Every action in AWS goes through IAM. Get it wrong and you either lock yourself out or get hacked.

Why IAM Matters

IAM controls WHO can do WHAT in your AWS account. Never use root credentials. Create users/roles with least privilege — grant only the permissions actually needed.
IAM Concept What It Is DevOps Use Case
User Individual human login with permanent credentials Developer accounts, admin accounts
Group Collection of users sharing the same policies dev-group, ops-group, readonly-group
Role Temporary credentials assumed by services or users EC2 role to access S3; Jenkins role to push ECR; Lambda role to write DynamoDB
Policy JSON document defining allowed/denied actions on resources Allow S3:PutObject on specific bucket; allow ECR:PutImage
MFA Multi-factor authentication — second factor beyond password Require for console login + sensitive API calls like deleting resources

Jenkins IAM Role Policy (Least Privilege)

Principle of Least Privilege

Grant Jenkins only what it needs: push to ECR and describe the EKS cluster. Nothing else.

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "ECRPush",
      "Effect": "Allow",
      "Action": [
        "ecr:GetAuthorizationToken",
        "ecr:BatchCheckLayerAvailability",
        "ecr:PutImage",
        "ecr:InitiateLayerUpload",
        "ecr:UploadLayerPart",
        "ecr:CompleteLayerUpload"
      ],
      "Resource": "arn:aws:ecr:us-east-1:123456789012:repository/my-spring-app"
    },
    {
      "Sid": "EKSAccess",
      "Effect": "Allow",
      "Action": [
        "eks:DescribeCluster",
        "eks:ListClusters"
      ],
      "Resource": "*"
    }
  ]
}
IAM Tips
  • Use IAM roles for services (EC2, Lambda, EKS pods) — never hardcode access keys in application code
  • Use IRSA (IAM Roles for Service Accounts) to give individual Kubernetes pods AWS permissions
  • Attach policies to Groups, not individual users — easier to manage at scale
  • Use IAM Access Analyzer to find overly permissive policies automatically
  • Rotate access keys regularly; prefer short-lived STS tokens via aws sts assume-role

6. S3 for Artifacts and Static Hosting

S3 is AWS's most versatile service. DevOps engineers use it for build artifact storage, static website hosting, log archiving, and CI/CD integration.

Store Build Artifacts from Jenkins

Artifact Storage Pattern

Archive every build's JAR to S3 so you can redeploy any previous version instantly — without re-running the build.

// In Jenkinsfile — archive JAR to S3 after every successful build
stage('Archive to S3') {
    steps {
        withAWS(credentials: 'aws-credentials', region: 'us-east-1') {
            s3Upload(
                bucket: 'my-artifacts',
                path: "releases/${APP_NAME}/${BUILD_NUMBER}/",
                includePathPattern: 'target/*.jar'
            )
        }
    }
}

Host Static Website (React/Angular Build Output)

Static Website Hosting

Deploy a React/Angular/Vue app to S3 + CloudFront — globally distributed, infinitely scalable, and costs pennies per month.

# 1. Build the React app
npm run build

# 2. Sync built files to S3 (--delete removes old files not in the new build)
aws s3 sync ./dist s3://my-react-app --delete

# 3. Invalidate CloudFront cache so users get the new version immediately
#    (without this, CloudFront serves the old cached version for up to 24 hours)
aws cloudfront create-invalidation \
  --distribution-id E1234567890 \
  --paths "/*"
S3 + CloudFront Benefits
  • No server to manage or patch
  • Served from 450+ edge locations globally
  • Scales to any traffic automatically
  • Free SSL certificate via ACM
  • ~$1-5/month for typical apps
S3 Bucket Policy for Website
  • Enable "Static website hosting" in bucket settings
  • Set index document: index.html
  • Error document: index.html (for SPA routing)
  • Block public access OFF for public buckets
  • Attach CloudFront OAC for private buckets

S3 Lifecycle Policies — Manage Costs

Automatically move old artifacts to cheaper storage tiers — save 80-90% on artifact storage costs.

Storage Class Cost (vs Standard) Use Case
S3 Standard $0.023/GB/month Current artifacts, frequently accessed data
S3 Standard-IA $0.0125/GB/month (−46%) Artifacts older than 30 days, infrequent access
S3 Glacier $0.004/GB/month (−83%) Compliance archives, old logs older than 90 days
S3 Glacier Deep Archive $0.00099/GB/month (−96%) Long-term retention, rarely or never accessed

7. AWS DevOps Best Practices

Hard-won lessons from running production workloads on AWS. Follow these to build secure, cost-effective, and highly available systems.

Security

  • Never commit AWS credentials to Git — use IAM roles, environment variables injected at runtime, or AWS Secrets Manager
  • Use IAM roles for EC2/EKS/Lambda instead of access keys — roles provide temporary, auto-rotated credentials
  • Enable CloudTrail for full API audit logs — every API call logged to S3 for compliance and incident investigation
  • Use VPC private subnets for databases and internal services — never expose RDS directly to the internet
  • Enable MFA for all IAM users — especially for console access and sensitive operations
  • Use Secrets Manager or SSM Parameter Store for secrets — not hardcoded env vars, S3 files, or config maps
  • Enable ECR image scanning — automatically scan Docker images for known CVEs on every push

Cost Optimization

  • Use Spot Instances for Jenkins agents — 70-90% cheaper than On-Demand. Jenkins can retry if a spot instance is interrupted.
  • Tag all resources (team, project, environment, cost-center) — required for accurate cost allocation and chargeback
  • Use Auto Scaling Groups to scale down at night and on weekends for non-prod environments
  • Delete unused resources — orphaned EBS volumes ($0.08/GB/month), unattached EIPs ($3.60/month), old AMIs and snapshots
  • Use S3 Lifecycle Policies to automatically move old build artifacts to Glacier — 83% cost reduction
  • Use Reserved Instances or Savings Plans for stable production workloads — 30-60% discount for 1-3 year commitments
  • Set up AWS Budgets alerts — get notified before you hit your budget limit, not after

High Availability

  • Deploy across 2+ Availability Zones — a single AZ failure should never take down your application
  • Use ALB for load balancing — distributes traffic, performs health checks, automatically routes around unhealthy instances
  • RDS Multi-AZ for database failover — standby replica in another AZ; automatic failover in 60-120 seconds
  • EKS worker nodes across 3 AZs — spread pods across zones using topologySpreadConstraints
  • Use health checks at every layer — ALB target health checks, Kubernetes readiness/liveness probes, RDS enhanced monitoring
  • Implement graceful shutdown in your apps — handle SIGTERM and drain connections before exiting to enable zero-downtime deployments
  • Use CloudWatch alarms for critical metrics — CPU, memory, error rates, latency p99 — with SNS notifications

AWS Well-Architected Framework — 6 Pillars

AWS's official framework for building reliable, secure, efficient, and cost-effective systems. Every AWS architect should know these.

Operational Excellence

Run and monitor systems. Automate everything. Refine operations procedures. Learn from failure — run game days.

Security

Protect data and systems. Strong identity, traceability, least privilege, encryption in transit and at rest.

Reliability

Recover from failures. Scale horizontally. Test recovery procedures. Manage change through automation.

Performance Efficiency

Use resources efficiently. Select right instance types. Monitor performance. Make data-driven decisions.

Cost Optimization

Eliminate waste. Right-size resources. Use managed services. Measure, monitor, improve over time.

Sustainability

Minimize environmental impact. Use efficient instance types. Scale to demand. Maximize utilization.
