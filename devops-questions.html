Docker is an open-source platform that packages applications and their dependencies into lightweight, portable containers that run consistently anywhere.
Definition
  • 📦 A container bundles code, runtime, libraries, and config into one isolated unit
  • 🖥️ Containers share the host OS kernel — unlike VMs which need a full OS per instance
  • ⚡ Start in milliseconds; VMs take minutes to boot
  • 🐳 Built on Linux kernel features: namespaces (isolation) + cgroups (resource limits)
Why We Use Docker
  • ✅ "Works on my machine" problem solved — same container runs in dev, staging, and prod
  • 🚀 Fast CI/CD — spin up a clean environment per pipeline run in seconds
  • 📐 Microservices — isolate each service in its own container with independent dependencies
  • 💰 Resource efficiency — run dozens of containers on hardware that supports only a few VMs
  • 🔁 Immutable infrastructure — rebuild, don't patch; rollback is a tag change
Quick Comparison
Virtual Machine
  • ⚖️ Full OS per VM — GBs in size
  • 🐢 Minutes to start
  • 🔒 Strong isolation (hypervisor)
Docker Container
  • 🪶 Shares host OS kernel — MBs in size
  • ⚡ Milliseconds to start
  • 🧱 Process-level isolation (namespaces)

An Image is the read-only blueprint; a Container is the running instance created from that blueprint.
Docker Image
  • 📄 Read-only template of layers
  • 🏗️ Built from a Dockerfile
  • 💾 Stored in a registry (Docker Hub, ECR)
  • ♾️ Can spawn many containers
Docker Container
  • ▶️ Running instance of an image
  • ✏️ Has a thin writable layer on top
  • 🔄 Can be started, stopped, paused, deleted
  • 🗑️ Ephemeral by default — data lost on removal
Analogy
  • 🍰 Image = recipe / mould; Container = the actual cake baked from it
  • 💿 Image = ISO file; Container = the running VM booted from it

# List local images docker images # Pull an image from Docker Hub docker pull nginx:1.25 # Run a container from an image docker run -d --name my-nginx nginx:1.25 # List running containers docker ps # List all containers (including stopped) docker ps -a

ENTRYPOINT defines the fixed executable; CMD provides default arguments that can be overridden at runtime.
ENTRYPOINT
  • 🔒 Fixed — always executed, cannot be replaced by docker run arguments
  • ⚙️ Override requires --entrypoint flag explicitly
  • 💡 Use when container is the executable (e.g., a CLI tool)
CMD
  • 🔓 Overridable — anything after image name in docker run replaces CMD
  • 📎 Acts as default arguments to ENTRYPOINT when both present
  • 💡 Use for default behaviour that operators can easily change
Combined Pattern (Best Practice)
# ENTRYPOINT = executable, CMD = default arguments ENTRYPOINT ["java", "-jar", "app.jar"] CMD ["--server.port=8080"] # docker run myapp → java -jar app.jar --server.port=8080 # docker run myapp --server.port=9090 → java -jar app.jar --server.port=9090

Multi-stage builds use multiple FROM statements in one Dockerfile so you can build in a heavy image and copy only the final artifact into a lean runtime image.
Why Use It
  • 🪶 Drastically smaller images — build tools (Maven, Node, Go compiler) are not in production image
  • 🔒 Better security — smaller attack surface, no source code in final image
  • 🚀 Faster pulls in CI/CD and Kubernetes since images are smaller
  • 📐 Single Dockerfile — no need for separate build scripts
Example — Java Spring Boot
# ---- Stage 1: Build ---- FROM maven:3.9-eclipse-temurin-21 AS builder WORKDIR /build COPY pom.xml . # Download dependencies first (cache layer) RUN mvn dependency:go-offline -q COPY src ./src RUN mvn package -DskipTests -q # ---- Stage 2: Runtime ---- FROM eclipse-temurin:21-jre-alpine WORKDIR /app # Copy ONLY the jar from the builder stage COPY --from=builder /build/target/app.jar app.jar EXPOSE 8080 ENTRYPOINT ["java", "-jar", "app.jar"]
Example — React / Node.js
# Stage 1: Build React app FROM node:20-alpine AS build WORKDIR /app COPY package*.json ./ RUN npm ci COPY . . RUN npm run build # Stage 2: Serve with nginx (tiny image) FROM nginx:1.25-alpine COPY --from=build /app/dist /usr/share/nginx/html EXPOSE 80 CMD ["nginx", "-g", "daemon off;"]
  • 📏 Build image (node:20) ≈ 400 MB; Final image (nginx:alpine) ≈ 23 MB — a 17× reduction

A Docker Volume is a persistent storage mechanism managed by Docker that survives container restarts and deletions.
The Problem Without Volumes
  • 🗑️ Container filesystem is ephemeral — all data written inside is lost when container is removed
  • 🔄 A new container from the same image starts completely fresh
  • 💾 Databases, uploads, logs would all be lost on restart

COPY is simple and explicit; ADD has hidden superpowers (URL fetch, tar auto-extraction) that make it unpredictable — prefer COPY.
COPY (Recommended)
  • 📋 Copies files/directories from build context only
  • 🔒 Predictable — does exactly what it says
  • ✅ Best practice for most use cases
ADD (Use Sparingly)
  • 🌐 Can fetch from URLs (discouraged — no checksum validation)
  • 📦 Auto-extracts tar archives into destination
  • ⚠️ Unexpected behaviour can inflate image size
When ADD is Legitimately Useful
# GOOD use of ADD: auto-extract a tarball ADD app-release.tar.gz /opt/app/ # This extracts the archive directly, saving an extra RUN tar command # Prefer COPY for everything else: COPY package*.json ./ COPY src/ ./src/
Docker Best Practice Recommendation
  • 🚫 Never use ADD to fetch remote URLs — use curl or wget in a RUN command with checksum verification instead

Docker isolation is built on Linux kernel primitives: namespaces for isolation, cgroups for resource limits, and capabilities + seccomp for privilege restriction.
Linux Namespaces (Isolation)
NamespaceIsolatesEffect
pidProcess IDsContainer sees its own PID tree; can't see host or other containers' processes
netNetwork stackOwn network interfaces, routing tables, iptables rules
mntFilesystem mountsOwn root filesystem (via overlay2); can't see host mounts by default
utsHostname / domain nameContainer has its own hostname independent of host
ipcInter-process communicationShared memory, semaphores isolated per container
userUID/GID mappingsContainer root (UID 0) maps to unprivileged user on host
Control Groups — cgroups (Resource Limits)
  • 🧮 CPU: limit how many CPU cores/shares a container can consume
  • 💾 Memory: set hard memory limits; OOM-kill if exceeded
  • 💿 Block I/O: throttle disk read/write bandwidth
  • 🌐 Network bandwidth: rate limiting via tc
Applying Resource Limits
# Limit container to 512MB RAM and 0.5 CPUs docker run -d \ --memory="512m" \ --memory-swap="512m" \ --cpus="0.5" \ my-app
Additional Security Layers
  • 🔐 Linux Capabilities: Docker drops dangerous capabilities (e.g., CAP_NET_ADMIN, CAP_SYS_ADMIN) from containers by default
  • 🛡️ seccomp: default seccomp profile blocks ~44 risky syscalls (e.g., ptrace, mount)
  • 🏔️ AppArmor / SELinux: MAC policies restrict file access and process behaviour

Docker reads the Dockerfile top-to-bottom, executes each instruction to create a read-only layer, and caches each layer by its content hash — re-using cached layers for unchanged steps.
Build Process Step-by-Step
  • 1️⃣ Docker reads build context (current directory) and sends it to the Docker daemon
  • 2️⃣ Parses Dockerfile instructions in order
  • 3️⃣ For each instruction: checks if a cache entry exists for that instruction + previous layer hash
  • 4️⃣ Cache HIT → reuses existing layer (milliseconds)
  • 5️⃣ Cache MISS → executes the instruction, creates new layer, invalidates all subsequent layers
  • 6️⃣ Final image = stack of all layers + image metadata
Cache Invalidation Rules
  • 📝 RUN: cache invalidated if instruction text changes
  • 📂 COPY/ADD: cache invalidated if any copied file's checksum changes
  • 🔽 Once any layer is invalidated, all layers below it are also invalidated
Optimising Layer Order for Caching
# BAD ORDER — code changes invalidate npm install cache every time COPY . . RUN npm install # GOOD ORDER — package.json rarely changes; npm install stays cached COPY package*.json ./ RUN npm install # ← cached unless package.json changes COPY . . # ← only this layer busts on code changes
Force Rebuild Without Cache
docker build --no-cache -t my-app .

Debugging Docker containers requires a systematic approach: check logs, inspect container state, exec into the container, and analyse exit codes.
Debugging Workflow
  • 1️⃣ Check container status — is it running, exited, or restarting?
  • 2️⃣ Read logs — most problems revealed here immediately
  • 3️⃣ Inspect container — env vars, mounts, network, exit code
  • 4️⃣ Exec into running container — interactive shell investigation
  • 5️⃣ Override entrypoint — start shell instead of app to explore
  • 6️⃣ Check resource usage — OOM killed?
Full Debug Command Toolkit

Inspect full container details (env, mounts, network, exit code) docker inspect my-container docker inspect my-container | grep -A5 '"ExitCode"' # 4. Exec into a RUNNING container docker exec -it my-container /bin/sh docker exec -it my-container /bin/bash # 5. Start a FAILED container with shell override (bypass entrypoint) docker run -it --entrypoint /bin/sh my-broken-image # 6. Check container resource stats docker stats my-container # 7. Container running but app unresponsive means the process is alive but the app crashed internally, is deadlocked, OOM constrained, or the port mapping is wrong.
Systematic Investigation
  • 1️⃣ Verify the process is actually running inside the container
  • 2️⃣ Check application logs for exceptions or startup failures
  • 3️⃣ Verify port binding — is the app listening on the correct interface?
  • 4️⃣ Test connectivity from inside the container
  • 5️⃣ Check resource constraints — is the process throttled?
  • 6️⃣ Inspect healthcheck status
Investigation Commands
# Step 1: Confirm processes running inside container docker exec my-container ps aux # Step 2: Check application logs for errors docker logs my-container --tail=200 # Step 3: Check which ports are listening INSIDE the container docker exec my-container netstat -tlnp # or docker exec my-container ss -tlnp # Step 4: Test from inside the container (avoid host network confusion) docker exec my-container curl -v http://localhost:8080/health # Step 5: Check resource usage — is CPU pegged or memory at limit? docker stats my-container --no-stream # Step 6: Check healthcheck status docker inspect my-container | grep -A20 '"Health"' # Step 7: Verify port mapping is correct docker port my-container
Common Root Causes
  • 🔌 App bound to 127.0.0.1 inside container — must bind to 0.0.0.0 to be reachable from outside
  • 💾 App ran out of memory (OOM) and was killed — check docker inspect OOMKilled field
  • 🐛 App crashed on startup but container restart policy keeps container "running"
  • 🔢 Port mapping wrong — e.g., app on 3000 but you mapped 8080:80

docker stop sends SIGTERM (graceful), waits, then SIGKILL. docker kill sends SIGKILL (immediate) — the process has no chance to clean up.
docker stop (Graceful)
  • 1️⃣ Sends SIGTERM to PID 1 in container
  • 2️⃣ Waits 10 seconds (configurable with -t) for graceful shutdown
  • 3️⃣ If still running, sends SIGKILL
  • ✅ Application can close connections, flush buffers, save state
docker kill (Immediate)
  • ⚡ Sends SIGKILL immediately by default (or any signal with --signal)
  • 🗑️ Process is killed instantly — no cleanup chance
  • ⚠️ Risk of data corruption if app was mid-write
Usage Examples
# Graceful stop (default 10s timeout) docker stop my-container # Graceful stop with custom timeout (30 seconds) docker stop -t 30 my-container # Immediate kill (SIGKILL) docker kill my-container # Send a specific signal (e.g., SIGHUP to reload config) docker kill --signal=SIGHUP my-nginx
Best Practice
  • 💡 Always use docker stop in production — your app should handle SIGTERM to flush queues and close DB connections cleanly
  • 💡 Use docker kill only when a container is hung and not responding to stop

Docker uses overlay2 (a Union Filesystem) to stack read-only image layers and add a thin writable layer on top per container — enabling efficient copy-on-write storage.
How Layers Stack
  • 📦 Each RUN, COPY, ADD instruction creates a new read-only layer
  • 🔒 Image layers are shared across all containers using the same image — no duplication on disk
  • ✏️ Each container gets its own thin writable layer (container layer) on top
  • 💡 When a container modifies a file from a lower layer, it is copied up to the writable layer first (Copy-on-Write)
Visual Layer Stack
Container writable layer ← unique per container (thin, ephemeral) ───────────────────────── COPY . . ← image layer 4 (read-only, shared) RUN npm install ← image layer 3 (read-only, shared) COPY package*.json ./ ← image layer 2 (read-only, shared) FROM node:20-alpine ← base image layers (read-only, shared) ───────────────────────── overlay2 merges all layers → container sees a single unified filesystem
Benefits of overlay2
  • 💾 Disk efficiency: 10 containers from the same image share the same layers — only 10 tiny writable layers extra
  • ⚡ Fast container start: no data copying, just a new writable layer is created
  • 🔄 Layer reuse in builds: cached layers dramatically speed up CI/CD rebuilds
Inspect Layers
# See layer sizes and commands docker history my-app:latest # See storage driver in use docker info | grep "Storage Driver"

A Docker HEALTHCHECK instructs Docker to periodically run a command inside the container to determine if the app is working correctly — beyond just "the process is running".
Why It Matters
  • 🔍 A container can be running but unhealthy (deadlock, OOM, startup race)
  • 🔄 Kubernetes and Docker Compose use health status for readiness gating (depends_on: condition: service_healthy)
  • ⚙️ Docker Swarm replaces unhealthy containers automatically

The build context is the set of files sent to the Docker daemon when you run docker build. It determines what files are available for COPY/ADD — and its size directly impacts build speed.
What Happens During docker build .
  • 1️⃣ The . tells Docker to use the current directory as build context
  • 2️⃣ Docker CLI tarballs the entire directory and sends it to the Docker daemon
  • 3️⃣ Daemon unpacks the context and makes files available for COPY/ADD instructions
  • 4️⃣ Large context = slow build start, wasted bandwidth, potential secrets exposure
Why It Causes Problems
  • 📦 node_modules (500MB+) gets sent even if you never COPY it — wastes time every build
  • 🔑 .env files and credentials are sent to the daemon — security risk
  • 🐌 Large .git directory sent unnecessarily — every commit adds history
Solution: .dockerignore
# .dockerignore — exclude from build context node_modules/ .git/ .gitignore *.md test/ coverage/ .env .env.* *.log target/ # Maven build output build/ # Gradle/React build output .DS_Store .idea/ *.iml
Alternative Context Sources
# Use a specific subdirectory as context docker build -f ./docker/Dockerfile ./app # Use a URL (Git repo) as context — only fetches what's needed docker build https://github.com/myorg/myrepo.git#mybranch # Use stdin (no context) — when Dockerfile doesn't need local files docker build - <<EOF FROM alpine RUN echo "hello" EOF

Replaces old pods with new ones gradually — zero downtime by keeping old pods running until new ones are ready.
  • maxUnavailable — how many pods can be down simultaneously during update (e.g., 1 or 25%)
  • maxSurge — how many extra pods above desired count can exist during update
  • Readiness probe gates traffic — new pod only receives traffic when it passes readiness check
  • If new pods fail readiness, rollout pauses automatically — old pods stay serving
  • K8s creates new ReplicaSet for each update and keeps old one for rollback

etcd is a distributed key-value store that holds the entire cluster state — it is the single source of truth for Kubernetes.
  • Stores ALL K8s objects: pods, services, deployments, configmaps, secrets, RBAC policies
  • Watch mechanism — controllers subscribe to changes in etcd to drive reconciliation loops
  • Leader election among control plane nodes uses etcd for coordination
  • MUST be backed up regularly — losing etcd = losing the entire cluster configuration
  • Uses Raft consensus algorithm — requires majority quorum (3 or 5 nodes for HA)
  • High write latency = delayed cluster state propagation — use fast SSDs for etcd

CrashLoopBackOff = container starts, crashes immediately, and K8s keeps retrying with exponential backoff (10s → 20s → 40s → max 5min).
  1. kubectl describe pod <name> -n <ns> → check the Events section at the bottom for error messages
  2. kubectl logs <pod> -n <ns> → check current application output for startup errors
  3. kubectl logs <pod> -n <ns> --previous → see logs from the last crashed container (critical — current logs may be empty)
  4. Check liveness probe settings — too aggressive probes (initialDelaySeconds too short) restart healthy containers
  5. Check for OOMKilled in describe output — memory limit too low; increase limits or fix memory leak
  6. kubectl exec -it <pod> -n <ns> -- /bin/sh — exec into running container to inspect files, env vars, connectivity

An Operator extends Kubernetes to manage complex stateful applications using Custom Resource Definitions (CRDs) + a custom controller that encodes operational knowledge.
  • CRD defines a custom resource type (e.g., KafkaCluster, PostgreSQLCluster)
  • Operator controller watches these CRDs and acts on changes — like a human operator but automated
  • Encodes operational knowledge: how to perform backups, handle failover, upgrade safely, scale read replicas
  • Real-world examples: Prometheus Operator (manages Prometheus instances), Strimzi (Kafka on K8s), cert-manager (TLS cert lifecycle), CloudNativePG (PostgreSQL)

K8s networking has three layers: pod-to-pod routing (CNI), Service virtual IP routing (kube-proxy), and DNS resolution (CoreDNS).
  • CNI Plugin (Calico/Flannel/Cilium) — assigns unique IPs to pods, programs routes so any pod can reach any other pod across nodes without NAT
  • kube-proxy — runs on each node; programs iptables/IPVS rules to intercept traffic to Service ClusterIPs and load-balance to backend pod IPs
  • CoreDNS — cluster DNS server; resolves service-name.namespace.svc.cluster.local to the Service ClusterIP
  • K8s networking rule: every pod can communicate with every other pod without NAT

New pods failing while old ones work points to a configuration/network issue that only affects newly created pods, not the app code itself.
  • NetworkPolicy — check if a NetworkPolicy allows traffic from old pods but not new ones (e.g., based on labels that aren't on new pods)
  • Secret/ConfigMap not mounted — verify DB credentials are being injected into new pods correctly (check env vars)
  • DB connection pool exhausted — old pods holding all connections; new pods timeout on connect (increase max_connections or reduce pool size per pod)
  • Readiness probe masking the error — pod never becomes ready so load balancer doesn't add it, but the real error is DB connectivity
  • Node-specific issue — new pods scheduled on a new node that lacks a label, taint, or DNS config
  • IP whitelist in DB — DB security group/firewall only allows specific pod CIDR ranges; new node's CIDR not whitelisted

apply is declarative and idempotent — use it in CI/CD. create is imperative and fails if the resource already exists.
kubectl create — Imperative
  • Fails with error if resource already exists
  • Not idempotent — cannot re-run safely
  • No tracking of previous state
  • Good for one-time resource creation
  • Cannot be used safely in CI/CD pipelines
kubectl apply — Declarative
  • Creates if not exists, updates if exists
  • Idempotent — safe to run multiple times
  • Tracks last-applied-configuration in annotations
  • The standard for CI/CD GitOps workflows
  • Supports --dry-run=client for validation

Long pipelines kill developer productivity — parallelize independent stages, cache dependencies aggressively, and skip unnecessary work on feature branches.
  • Parallelize — unit tests + lint + security scan + checkstyle all run at the same time, not sequentially
  • Cache Maven .m2 — mount -v /root/.m2:/root/.m2 in Docker agent to avoid re-downloading dependencies every build
  • Docker layer cache — use --cache-from to reuse previous image layers; copy pom.xml and download deps before copying source
  • Skip integration tests on feature branches — use when { branch 'main' } to gate slow tests to main only
  • Incremental compilation — avoid clean on every build if previous artifacts are valid
  • Fail fast — put quickest checks first; add failFast: true to parallel blocks

A misconfigured Jenkins is a major security risk — it has access to your codebase, credentials, and production infrastructure. Apply defence-in-depth.
  • Matrix-based security — grant least-privilege access; developers get build/read, not admin
  • Disable anonymous access — no one should use Jenkins without authenticating
  • CSRF protection — enabled by default; ensure it's not disabled
  • Script Security plugin — Groovy scripts in pipelines require admin approval; prevents code injection via Jenkinsfile
  • Run agents as non-root — compromised agent shouldn't have root on the host
  • Rotate credentials — use short-lived tokens; integrate with HashiCorp Vault for dynamic secrets
  • Keep Jenkins and plugins updated — security vulnerabilities in plugins are common attack vectors
  • Audit logs — use Audit Trail plugin to log who did what
  • Network isolation — Jenkins master should not be publicly accessible; use VPN or bastion
  • Use Folders to scope credentials per project — a team cannot access another team's credentials

Jenkins accumulates build logs, artifacts, workspace files, and Docker layers — disk exhaustion causes build failures and can crash Jenkins entirely.

Structured incident response: Detect → Acknowledge → Communicate → Investigate → Mitigate → Resolve → Postmortem. Restore service first — investigate why later.
  1. Alert fires (PagerDuty/Opsgenie) → on-call engineer acknowledges within SLA (e.g., 5 min)
  2. Declare incident severity — SEV1 (all users affected) or SEV2 (partial) — this sets the response urgency
  3. Notify stakeholders — update status page (Statuspage.io), post in #incidents Slack channel
  4. Assemble incident channel — create #inc-YYYY-MM-DD-description Slack channel, assign incident commander
  5. Investigate — check monitoring dashboards, logs (Kibana), traces (Jaeger), recent deployments (git log, Jenkins)
  6. Mitigate — don't wait for root cause! Roll back deployment, toggle feature flag off, scale up, re-route traffic
  7. Verify recovery — confirm metrics/error rates return to normal, smoke test critical paths
  8. All-clear communication — update status page, notify stakeholders, close incident channel
  9. Blameless postmortem within 48 hours — timeline, impact, root cause, action items, no blame

Detect failure fast, mitigate immediately to restore service, investigate after service is restored — in that exact order. Never investigate while users are impacted.
  1. Monitoring alerts fire — error rate spike, latency increase, health check failures visible in Grafana
  2. Check deployment rollout status: kubectl rollout status deployment/app -n production
  3. Confirm the deployment caused it: kubectl rollout history deployment/app — does the timing match?
  4. Decision: rollback or fix-forward? Simple config change → fix-forward. Unknown bug → rollback
  5. Rollback: kubectl rollout undo deployment/app -n production → watch pods recover
  6. Verify recovery: check dashboards, run smoke tests against production endpoints
  7. Communicate status — update status page, notify stakeholders, post in #incidents
  8. Root cause analysis — check pod logs, application logs, what changed in the deployment
  9. Fix in development → write regression test → re-deploy with proper validation

An operation is idempotent if running it once or multiple times produces the same result — critical for safe automation, retries, and disaster recovery.
Not Idempotent — Dangerous
  • mkdir /app — fails on second run (directory exists)
  • INSERT INTO users VALUES (...) — duplicates data on retry
  • kubectl create -f deploy.yaml — fails if resource exists
  • HTTP POST — creates a new resource each time
  • Retrying a CI step can cause duplicate deploys
Idempotent — Safe to Retry
  • mkdir -p /app — succeeds on any run
  • INSERT ... ON CONFLICT DO NOTHING — safe to retry
  • kubectl apply -f deploy.yaml — creates or updates
  • HTTP PUT — same result every time for same payload
  • CI pipeline retry is always safe — same outcome
  • Terraform apply is idempotent — running it 10 times produces the same infrastructure
  • Ansible tasks are idempotent — "install nginx" checks if already installed before running
  • kubectl apply is idempotent — safe to run in CI/CD pipelines on every commit
  • HTTP PUT is idempotent; HTTP POST is not — design APIs accordingly
  • Makes retries safe in CI/CD — network failures won't cause duplicate deployments

Zero-downtime means users never experience an outage during deployments — achieved by combining rolling update strategy, readiness probes, PodDisruptionBudgets, and graceful shutdown.
  • Rolling update with maxUnavailable: 0 — never take a pod down without a replacement ready
  • Readiness probe — only route traffic to pods that pass the health check
  • PodDisruptionBudget — ensures minimum available pods during voluntary disruptions (node drains)
  • preStop hook + sleep — gives the load balancer time to deregister the pod before it stops accepting connections
  • terminationGracePeriodSeconds — gives in-flight requests time to complete before the process is killed
  • Database migrations must be backward-compatible (expand-contract pattern) — new schema works with both old and new app
