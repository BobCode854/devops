1. What is Kubernetes? Why Do We Need It?

Understanding the production-scale container orchestration problem that Kubernetes was built to solve.

The Problem With Just Docker

Docker runs ONE container. But in production you have 100+ containers across 10 servers — who restarts a crashed container? Who load balances? Who scales up at 9am traffic spike? That's Kubernetes.
Just Docker in Prod
  • Container crashes → stays dead until someone manually restarts
  • Traffic spike → manually add containers on each server
  • Server dies → manually move containers
  • Deploy update → manually stop + start each container
  • No load balancing across multiple containers
  • You become the babysitter
Kubernetes
  • Container crashes → K8s restarts it automatically (self-healing)
  • Traffic spike → HPA scales pods up automatically
  • Node fails → pods reschedule to healthy nodes
  • Deploy → rolling update, zero downtime
  • Built-in load balancing via Services
  • You just declare desired state

What Kubernetes Does

ProblemK8s Solution
Container crashedAuto-restarts via liveness probes
Too much trafficHorizontalPodAutoscaler adds replicas
Server diedReschedules pods to healthy nodes
New version to deployRolling update with zero downtime
Need to find service IPDNS-based service discovery
Store config/secretsConfigMaps + Secrets
Persist dataPersistentVolumes
Route external trafficIngress controller
Self-Healing Auto-Scaling Desired State Service Discovery

2. Kubernetes Architecture

The Control Plane manages the cluster; Worker Nodes run your workloads. Together they form a Kubernetes cluster.

Control Plane Components

API Server

The front door to Kubernetes. Every kubectl command, every CI/CD pipeline interaction, every internal component communication goes through the API Server. It validates requests and persists state to etcd.

Role: REST API gateway for all cluster operations
Process: kube-apiserver
etcd

A distributed, highly-available key-value store that holds ALL cluster state — every pod, service, configmap, secret, node. It is the brain of Kubernetes. If etcd is lost, the cluster is lost. Must be backed up regularly.

Role: Single source of truth for cluster state
Critical: Back this up! It is the cluster.
Scheduler

Watches for newly created pods with no assigned node. Decides which node a pod should run on based on resource availability, affinity/anti-affinity rules, taints and tolerations, and topology constraints.

Role: Assign pods to nodes
Process: kube-scheduler
Controller Manager

Runs multiple controller loops in one process. Each controller watches the state of the cluster and works to move the current state toward the desired state. Includes the ReplicaSet, Deployment, Node, and Job controllers.

Role: Reconcile desired vs actual state
Process: kube-controller-manager

Worker Node Components

kubelet

An agent running on every worker node. It watches the API server for pods assigned to its node, then pulls the container image and starts the container. It also reports pod status back to the API server.

Role: Ensure containers are running on this node
Talks to: API Server + Container Runtime
kube-proxy

Handles all network routing on the node. Maintains iptables (or IPVS) rules to implement Kubernetes Services — forwarding traffic to the correct pod based on Service selectors.

Role: Network proxy + load balancing
Implements: Service networking rules
Container Runtime

The software that actually runs the containers. Kubernetes uses the Container Runtime Interface (CRI) to talk to the runtime. Docker was replaced by containerd as the default since Kubernetes 1.24.

Modern default: containerd
Alternatives: CRI-O, Docker (via shim)

How kubectl Works — Step by Step

  1. You run kubectl apply -f deployment.yaml on your machine
  2. kubectl sends an authenticated HTTPS request to the API Server
  3. API Server validates the manifest and stores the desired state in etcd
  4. Controller Manager notices pods are missing → creates a ReplicaSet → tells Scheduler
  5. Scheduler selects the best node based on resources and constraints → updates pod spec in etcd
  6. kubelet on that node watches etcd via API Server → pulls the container image → starts container via container runtime
  7. kube-proxy updates iptables rules → pod is reachable via Service

3. Core Kubernetes Objects

The fundamental building blocks you declare to describe your desired application state.

Pod

A Pod is the smallest deployable unit in K8s — it wraps one or more containers that share the same network namespace and storage. Pods are ephemeral — they die and get replaced. Never manually restart pods; let Deployments handle that.
Shared Network (same IP) Shared Volumes Ephemeral — can die Identified by Labels 
apiVersion: v1
kind: Pod
metadata:
  name: my-app
  labels:
    app: my-app
    version: "1.0"
spec:
  containers:
  - name: app
    image: myorg/spring-app:1.0
    ports:
    - containerPort: 8080
    env:
    - name: SPRING_PROFILES_ACTIVE
      value: "prod"
    resources:
      requests:
        memory: "256Mi"
        cpu: "250m"
      limits:
        memory: "512Mi"
        cpu: "500m"
    readinessProbe:
      httpGet:
        path: /actuator/health
        port: 8080
      initialDelaySeconds: 30
      periodSeconds: 10
    livenessProbe:
      httpGet:
        path: /actuator/health
        port: 8080
      initialDelaySeconds: 60
      periodSeconds: 20

Deployment

A Deployment manages a set of identical Pods (replicas). It maintains the desired replica count, handles rolling updates with zero downtime, and enables instant rollback to any previous version.
Rolling Update Strategy
  • maxUnavailable: 1 — at most 1 pod down during update
  • maxSurge: 1 — at most 1 extra pod spun up
  • Old pods terminate only after new ones are Ready
  • Zero downtime guaranteed
Deployment Manages
  • ReplicaSet (which manages individual Pods)
  • Rollout history (kubectl rollout history)
  • Rollback to any previous revision
  • Pause / resume rollouts
apiVersion: apps/v1
kind: Deployment
metadata:
  name: spring-app
  namespace: production
spec:
  replicas: 3
  selector:
    matchLabels:
      app: spring-app
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 1
      maxSurge: 1
  template:
    metadata:
      labels:
        app: spring-app
        version: "2.0"
    spec:
      containers:
      - name: app
        image: myorg/spring-app:2.0
        ports:
        - containerPort: 8080
        envFrom:
        - configMapRef:
            name: app-config
        - secretRef:
            name: app-secrets
        resources:
          requests:
            memory: "256Mi"
            cpu: "250m"
          limits:
            memory: "512Mi"
            cpu: "500m"
        readinessProbe:
          httpGet:
            path: /actuator/health/readiness
            port: 8080
          initialDelaySeconds: 30
          periodSeconds: 10
          failureThreshold: 3
        livenessProbe:
          httpGet:
            path: /actuator/health/liveness
            port: 8080
          initialDelaySeconds: 60
          periodSeconds: 20

Service

A Service gives Pods a stable IP address and DNS name. Pods are ephemeral — they get new IPs when restarted. The Service stays fixed and load-balances traffic across all matching pods via label selectors.
TypeAccessible FromUse Case
ClusterIP (default)Inside cluster onlyInternal microservice-to-microservice communication
NodePortExternal via NodeIP:Port (30000–32767)Dev/testing environments, on-premises clusters
LoadBalancerExternal via cloud Load Balancer IPProduction on AWS/GCP/Azure — provisions cloud LB automatically
ExternalNameMaps to an external DNS nameCall external services by an internal Kubernetes name
 
apiVersion: v1
kind: Service
metadata:
  name: spring-app-svc
spec:
  type: ClusterIP
  selector:
    app: spring-app       # matches Deployment pods by label
  ports:
  - port: 80              # Service port (what callers use)
    targetPort: 8080      # Container port (where app listens)

ConfigMap and Secret

ConfigMap
  • Non-sensitive configuration data
  • Environment variables, command args, config files
  • Stored in plain text in etcd
  • Changes can be applied without rebuilding images
Secret
  • Sensitive data — passwords, tokens, keys
  • Stored base64-encoded (not encrypted by default!)
  • Enable etcd encryption + RBAC in production
  • Better: use external secrets manager (AWS SSM, Vault)
# ConfigMap — non-sensitive config
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  SPRING_PROFILES_ACTIVE: "prod"
  LOG_LEVEL: "INFO"
  DB_HOST: "postgres-svc"
  DB_PORT: "5432"
---
# Secret — sensitive data (base64 encoded)
apiVersion: v1
kind: Secret
metadata:
  name: app-secrets
type: Opaque
data:
  DB_PASSWORD: cGFzc3dvcmQxMjM=   # base64 of "password123"
  JWT_SECRET: bXlzZWNyZXRrZXk=    # base64 of "mysecretkey"
Tip: Create secrets via CLI to avoid committing base64 values to git: kubectl create secret generic app-secrets --from-literal=DB_PASSWORD=password123

Ingress

Ingress is a reverse proxy / API gateway for Kubernetes. It routes external HTTP/HTTPS traffic to the correct internal Service based on hostname and URL path — one entry point for your whole cluster.
Host-based routing Path-based routing TLS termination Needs Ingress Controller (nginx, traefik) 
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: app-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
    cert-manager.io/cluster-issuer: "letsencrypt-prod"
spec:
  tls:
  - hosts:
    - myapp.example.com
    secretName: myapp-tls
  rules:
  - host: myapp.example.com
    http:
      paths:
      - path: /api
        pathType: Prefix
        backend:
          service:
            name: spring-app-svc
            port:
              number: 80
      - path: /
        pathType: Prefix
        backend:
          service:
            name: frontend-svc
            port:
              number: 80

5. Auto-Scaling

Kubernetes can automatically scale your workloads horizontally (more pods) or vertically (more CPU/memory) based on live metrics.

HorizontalPodAutoscaler (HPA)

HPA automatically scales pods up when CPU or memory exceeds your target threshold, and scales back down when load drops. Scale from 2 to 20 pods in seconds without any manual intervention.
Scale Up Trigger
  • Average CPU across pods exceeds 70%
  • Average memory exceeds 400Mi
  • Custom metrics (requests/sec via Prometheus adapter)
  • HPA checks metrics every 15 seconds
Scale Down Behaviour
  • Waits 5 minutes (cooldown) before scaling down
  • Prevents flapping on short traffic spikes
  • Never scales below minReplicas
  • Configurable with stabilizationWindowSeconds
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: spring-app-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: spring-app
  minReplicas: 2
  maxReplicas: 20
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70    # scale up if avg CPU > 70%
  - type: Resource
    resource:
      name: memory
      target:
        type: AverageValue
        averageValue: 400Mi
# Apply and inspect HPA
kubectl apply -f hpa.yaml
kubectl get hpa
kubectl describe hpa spring-app-hpa
Prerequisite: HPA requires metrics-server to be installed in the cluster. On EKS/GKE it is included. For local clusters: kubectl apply -f https://github.com/kubernetes-sigs/metrics-server/releases/latest/download/components.yaml

Scaling Comparison

TypeWhat it doesBest for
HPA (Horizontal Pod Autoscaler)Adds/removes pod replicasStateless apps — most common approach
VPA (Vertical Pod Autoscaler)Adjusts CPU/memory requests per podStateful apps, right-sizing resources
Cluster AutoscalerAdds/removes worker nodesCloud-managed clusters (EKS, GKE) — scales infrastructure
KEDAEvent-driven scaling (queue depth, HTTP RPS, etc.)Scale to zero, queue-based workloads

6. Persistent Storage

Pods are ephemeral — when they die, their data dies too. PersistentVolumes decouple storage from pods so data survives restarts.

PV / PVC / StatefulSet

Pods are ephemeral — data dies with them. PV (PersistentVolume) is the actual storage. PVC (PersistentVolumeClaim) is the pod's request for storage. Kubernetes binds the right PV to the PVC automatically.
ObjectWho Creates ItRole
PersistentVolume (PV)Admin or cloud provisionerThe actual storage resource (disk, NFS, EBS, etc.)
PersistentVolumeClaim (PVC)Developer / app manifestRequest for a specific amount/type of storage
StorageClassAdminTemplate for dynamic PV provisioning (gp2, gp3, standard)
StatefulSetDeveloperLike Deployment but for stateful apps — stable pod names + storage
 
# PersistentVolume (usually created by cloud provider or admin)
apiVersion: v1
kind: PersistentVolume
metadata:
  name: postgres-pv
spec:
  capacity:
    storage: 10Gi
  accessModes:
    - ReadWriteOnce
  persistentVolumeReclaimPolicy: Retain
  storageClassName: standard
  hostPath:
    path: /data/postgres     # for local dev; use cloud storage in prod
---
# PersistentVolumeClaim — what the app requests
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: postgres-pvc
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 5Gi
  storageClassName: standard
---
# StatefulSet using PVC for Postgres
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: postgres
spec:
  serviceName: postgres
  replicas: 1
  selector:
    matchLabels:
      app: postgres
  template:
    metadata:
      labels:
        app: postgres
    spec:
      containers:
      - name: postgres
        image: postgres:15-alpine
        env:
        - name: POSTGRES_DB
          value: myapp
        - name: POSTGRES_PASSWORD
          valueFrom:
            secretKeyRef:
              name: app-secrets
              key: DB_PASSWORD
        volumeMounts:
        - name: postgres-storage
          mountPath: /var/lib/postgresql/data
  volumeClaimTemplates:
  - metadata:
      name: postgres-storage
    spec:
      accessModes: ["ReadWriteOnce"]
      resources:
        requests:
          storage: 10Gi
Access Modes
  • ReadWriteOnce (RWO) — one node can mount read/write (most common)
  • ReadOnlyMany (ROX) — many nodes can mount read-only
  • ReadWriteMany (RWX) — many nodes can mount read/write (NFS, EFS)
Deployment vs StatefulSet
  • Deployment pods get random names (app-xyz123)
  • StatefulSet pods get stable names (postgres-0, postgres-1)
  • StatefulSet maintains ordered startup/shutdown
  • Each pod gets its own persistent volume

8. Debugging Common Issues

The most frequent Kubernetes problems and exactly how to diagnose and fix them.

CrashLoopBackOff

CrashLoopBackOff means the container starts, immediately crashes, and Kubernetes keeps restarting it with exponential backoff. The pod is trying — but the app inside keeps dying.
Causes + Fixes
  • Bad CMD/ENTRYPOINT in Dockerfile — test with docker run locally first before pushing to K8s
  • App crashes on startup — run kubectl logs pod-name --previous to see the last crash's output
  • Missing environment variable — kubectl describe pod shows the env section and what was injected
  • OOMKilled (out of memory) — increase memory limits; check kubectl describe pod for OOMKilled reason
  • Wrong image tag / image doesn't exist — kubectl describe pod shows image pull errors in Events

Pod Stuck in Pending

Common Causes
  • Insufficient CPU or memory on all nodes — kubectl describe pod → Events will say "Insufficient cpu" or "Insufficient memory"
  • Node taint rejecting pod — pod needs matching tolerations; check kubectl describe node for taints
  • PVC not bound — kubectl get pvc will show status Pending; check StorageClass and provisioner logs
  • Image pull failure (no registry credentials) — create an imagePullSecret and reference it in the pod spec
  • NodeSelector or affinity rules not matching any node — check node labels with kubectl get nodes --show-labels

Debug Commands

# Full pod details including Events section (always start here)
kubectl describe pod my-pod -n production

# Previous container logs (after crash — most useful for CrashLoopBackOff)
kubectl logs my-pod --previous

# Run an ephemeral debug container alongside the broken pod
kubectl debug -it my-pod --image=busybox --target=app

# Run a standalone debug pod in the same namespace
kubectl run debug --image=busybox --rm -it --restart=Never -- sh

# Check CPU and memory usage per pod
kubectl top pods -n production

# See all cluster events sorted by timestamp (great for recent failures)
kubectl get events -n production --sort-by=.lastTimestamp

# Check why a node is not ready (hardware / kubelet issues)
kubectl describe node <node-name>

# Port-forward to reach a pod directly (bypass Service)
kubectl port-forward pod/my-pod 8080:8080 -n production
Debugging Flow: Always start with kubectl describe pod — the Events section at the bottom tells you exactly what went wrong. Then check kubectl logs for application-level errors.

9. RBAC — Role-Based Access Control

Control who can do what inside your Kubernetes cluster. RBAC is enabled by default and is the Kubernetes security model.

Core RBAC Concepts

ObjectScopePurpose
RoleNamespaceDefines a set of permissions (verbs on resources) within one namespace
ClusterRoleCluster-wideSame as Role but applies across all namespaces (e.g., node access)
RoleBindingNamespaceGrants a Role to a User, Group, or ServiceAccount in one namespace
ClusterRoleBindingCluster-wideGrants a ClusterRole to a subject across all namespaces
ServiceAccountNamespaceIdentity for pods to call the Kubernetes API (not humans)

Developer Read-Only Role

# Read-only role for developers — can view but not modify
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  namespace: production
  name: developer-readonly
rules:
- apiGroups: [""]
  resources: ["pods", "services", "configmaps"]
  verbs: ["get", "list", "watch"]
- apiGroups: ["apps"]
  resources: ["deployments"]
  verbs: ["get", "list", "watch"]
---
# Bind the role to a specific user
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: developer-binding
  namespace: production
subjects:
- kind: User
  name: john.doe@company.com
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: Role
  name: developer-readonly
  apiGroup: rbac.authorization.k8s.io
Common Verbs
  • get — retrieve a single resource
  • list — list all resources of a type
  • watch — stream changes in real-time
  • create — create a new resource
  • update — overwrite a resource
  • patch — partial update
  • delete — remove a resource
RBAC Best Practices
  • Principle of least privilege — only grant what's needed
  • Prefer namespaced Roles over ClusterRoles
  • Use ServiceAccounts for pod-to-API communication
  • Audit with: kubectl auth can-i list pods --as=user
  • Never bind cluster-admin to application pods

10. Helm — Kubernetes Package Manager

Helm packages your Kubernetes YAML manifests into reusable, versioned, configurable Charts — like npm for Kubernetes.

Core Helm Concepts

Helm is like apt/npm for Kubernetes — package your K8s YAMLs as a reusable Chart with configurable values. Deploy the same chart to dev, staging, and prod with different values.yaml files.
ConceptWhat It Is
ChartA package of Kubernetes YAML templates — like a npm package
ReleaseAn installed, running instance of a chart in the cluster
ValuesConfiguration overrides injected into templates at deploy time
RepositoryA collection of charts hosted at a URL (like npm registry)
TemplateA Kubernetes manifest with {{ .Values.xxx }} placeholders
Chart = Package Release = Running Instance Values = Configuration Repo = Registry

Common Helm Commands

# Install Helm
curl https://raw.githubusercontent.com/helm/helm/main/scripts/get-helm-3 | bash

# Add a chart repository
helm repo add bitnami https://charts.bitnami.com/bitnami
helm repo update

# Search for a chart
helm search repo bitnami/postgresql

# Install PostgreSQL with custom values
helm install my-postgres bitnami/postgresql \
  --set auth.postgresPassword=mysecret \
  --set primary.persistence.size=10Gi

# Install from your own chart directory
helm install my-app ./charts/spring-app \
  --namespace production \
  --values values-prod.yaml

# Preview the rendered templates (dry run)
helm install my-app ./charts/spring-app --dry-run --debug

# Upgrade a release with new values
helm upgrade my-app ./charts/spring-app --values values-prod.yaml

# Rollback to a previous release revision
helm rollback my-app 1

# List all installed releases across namespaces
helm list -A

# Get values used for a release
helm get values my-app

# Uninstall a release
helm uninstall my-app

Example Chart Structure

Chart Directory Structure
  • Chart.yaml — metadata (name, version)
  • values.yaml — default values
  • values-prod.yaml — production overrides
  • templates/ — YAML templates with {{ }}
  • templates/deployment.yaml
  • templates/service.yaml
  • templates/ingress.yaml
  • charts/ — chart dependencies
Why Use Helm?
  • One chart, multiple environments (dev/staging/prod)
  • Version-controlled releases with rollback
  • Community charts for Postgres, Redis, Kafka, etc.
  • Template logic — conditionals, loops, defaults
  • Consistent, repeatable deployments across clusters
  • helm test — run integration tests post-deploy
