1. What Is DevOps?

The culture, practices, and tools that unite Development and Operations to deliver software faster and more reliably.

DevOps = Culture + Practices + Tools that unite Development and Operations to deliver software faster and more reliably.

Before DevOps — The Problem
  • 📝 Developer writes code
  • 🧱 Throws it "over the wall" to Ops
  • 🖥️ Ops runs it → it breaks
  • 😤 Blame game begins
  • 🐌 Releases take months
  • 🔒 Siloed teams, no collaboration
With DevOps
  • 🤝 Dev + Ops collaborate daily
  • ⚙️ Automate everything — CI/CD, IaC
  • 🚀 Continuous delivery to production
  • 🔁 Fast feedback loops at every stage
  • ⚡ Releases in hours or days
  • 📊 Shared ownership and metrics
Core Pillars of DevOps
  • 🤝 Culture — Shared responsibility, no blame, cross-functional teams
  • ⚙️ Automation — CI/CD pipelines, Infrastructure as Code (Terraform, Ansible)
  • 📊 Measurement — Metrics, monitoring, alerting (Prometheus, Grafana, DORA)
  • 🗣️ Sharing — Knowledge, tools, feedback loops, post-mortems

The Wall of Confusion

"Dev says 'works on my machine' — Ops says 'not my problem' — this is the wall DevOps breaks down."

ProblemDevOps Solution
Slow manual deploymentsAutomated CI/CD pipelines — deploy in minutes, not days
"Works on my machine"Docker containers — same environment everywhere, every time
Manual server configurationInfrastructure as Code (Terraform, Ansible) — version-controlled infra
No visibility into productionMonitoring + Alerting (Prometheus, Grafana, ELK Stack)
Long release cycles (months)Continuous Delivery — deploy daily or even hourly
Siloed teams, blame cultureCross-functional teams with shared ownership and accountability
Inconsistent environmentsContainerization + IaC ensures dev = staging = production
Late defect discoveryShift-left testing — automated tests run on every commit

DevOps Lifecycle — 8 Phases

The infinite loop of Plan → Code → Build → Test → Release → Deploy → Operate → Monitor — then back to Plan.

DevOps is an infinite loop — every phase feeds the next, creating a continuous improvement cycle that never stops.

1. Plan

Define requirements, user stories, and sprint goals. Break work into deliverable chunks.

Tools: Jira, Confluence, Trello, Azure Boards
2. Code

Developers write, review, and version-control application source code.

Tools: Git, GitHub, GitLab, Bitbucket, VS Code
3. Build

Compile source code, resolve dependencies, and package the application into an artifact.

Tools: Maven, Gradle, npm, Webpack, Docker
4. Test

Automatically run unit, integration, and E2E tests to catch defects before they reach production.

Tools: JUnit, Selenium, Jest, Postman, SonarQube
5. Release

Package, version, and stage the validated artifact for deployment to any environment.

Tools: Jenkins, GitHub Actions, GitLab CI, CircleCI
6. Deploy

Push the release to production or target environment using automated deployment strategies.

Tools: Docker, Kubernetes, Ansible, Terraform, Helm
7. Operate

Manage running infrastructure, ensure uptime, handle scaling and configuration in production.

Tools: Linux, Nginx, Tomcat, AWS, Kubernetes
8. Monitor

Observe system health, collect logs and metrics, alert on anomalies, close the feedback loop.

Tools: Prometheus, Grafana, ELK Stack, Datadog, CloudWatch

2. The Problem with Traditional SDLC

How Waterfall became Agile, and how DevOps took it even further — from months to minutes.

Waterfall vs Agile vs DevOps

Aspect Waterfall Agile DevOps
Release Cycle Months / Years 2-week sprints Daily / Hourly
Team Structure Siloed — Dev, QA, Ops separate Cross-functional Dev + QA Unified Dev + QA + Ops
Feedback After final release End of each sprint Continuous — real-time
Testing After all development Each sprint (manual & auto) Continuous — shift-left, automated
Deployment Manual, infrequent, risky Manual or semi-automated Fully automated pipelines
Failure Recovery Days to weeks Hours to days Minutes — auto rollback
Infrastructure Manual provisioning Still largely manual Infrastructure as Code
Risk Level Very high — big-bang releases Medium — sprint-by-sprint Low — small, frequent increments

Real World Example — Old Way vs DevOps Way

Imagine deploying a Java Spring Boot app the OLD way vs the DevOps way — same code, completely different experience.

Old Way (Manual)
  • 1️⃣ Developer builds JAR locally on their machine
  • 2️⃣ Emails the JAR file to the Ops team
  • 3️⃣ Ops manually copies JAR to production server
  • 4️⃣ Ops manually restarts Tomcat
  • 5️⃣ Everyone hopes it works — no automated tests
  • 6️⃣ Bug found 2 days later → start again
  • ⏱️ Total time: 2–5 days
DevOps Way (Automated)
  • 1️⃣ Developer pushes code to Git repository
  • 2️⃣ GitHub webhook fires → Jenkins pipeline starts
  • 3️⃣ Maven auto-builds the JAR artifact
  • 4️⃣ JUnit + Selenium tests run automatically
  • 5️⃣ Docker image built and pushed to registry
  • 6️⃣ Kubernetes deploys → auto-rolled back if unhealthy
  • ⚡ Total time: under 10 minutes

3. CI/CD Pipeline — The Heart of DevOps

Continuous Integration, Delivery, and Deployment — automate from code commit all the way to production.

What Is CI/CD?

CI = merge code frequently + auto-test every commit. CD = auto-deploy to any environment on demand.

TermFull FormWhat It Does
CI Continuous Integration Auto-build + auto-test on every code commit. Catches integration issues early.
CD Continuous Delivery Auto-deploy to staging; one-click production trigger. Keeps code always release-ready.
CD Continuous Deployment Fully automatic deployment all the way to production — zero manual steps required.

A Real Pipeline — Jenkinsfile

// Jenkinsfile — Declarative Pipeline: Code → Build → Test → Docker → Deploy pipeline { agent any stages { stage('Checkout') { steps { git 'https://github.com/user/app.git' } } stage('Build') { steps { sh 'mvn clean package -DskipTests' } } stage('Test') { steps { sh 'mvn test' } post { always { junit '**/target/surefire-reports/*.xml' } } } stage('Docker Build') { steps { sh 'docker build -t myapp:latest .' sh 'docker tag myapp:latest myregistry/myapp:${BUILD_NUMBER}' sh 'docker push myregistry/myapp:${BUILD_NUMBER}' } } stage('Deploy') { steps { sh 'kubectl apply -f k8s/deployment.yaml' sh 'kubectl rollout status deployment/myapp' } } } post { success { slackSend message: "✅ Build ${BUILD_NUMBER} deployed!" } failure { slackSend message: "❌ Build ${BUILD_NUMBER} failed!" } } }

CI/CD Pipeline Flow — Step by Step

1
Developer pushes code to a feature branch on GitHub/GitLab
2
GitHub webhook fires — notifies Jenkins that new code is available
3
Jenkins picks up the build — checks out the latest code from the repository
4
Maven builds the JAR — compiles, resolves dependencies, packages the artifact
5
JUnit tests run — unit and integration tests execute automatically; failure stops the pipeline
6
SonarQube quality gate — code coverage, complexity, security vulnerabilities checked
7
Docker image built — application packaged into a container image with all its dependencies
8
Image pushed to Docker Hub / ECR — versioned artifact stored in the container registry
9
Kubernetes deployment updated — rolling update applied; new pods gradually replace old ones
10
Health check performed — liveness/readiness probes confirm the new pods are healthy
11
Slack notification sent — team notified of success or auto-rollback on failure
Key Insight
The entire pipeline — from code push to live production — typically runs in 5–15 minutes for a well-optimized DevOps setup. Elite teams run this cycle multiple times per day.

5. DevOps vs SRE vs Platform Engineering

Three related but distinct disciplines — each with different focus, origins, and ways of measuring success.

DevOps is a culture. SRE is a job function that implements DevOps principles with engineering rigor. Platform Engineering is the next evolution — building internal developer platforms.

DevOps — Cultural movement SRE — Google's engineering approach Platform Eng — Modern evolution
Aspect DevOps SRE (Site Reliability Engineering) Platform Engineering
Primary Focus Delivery speed + reliability balance Reliability, uptime, and SLO management Developer productivity and self-service
Origin Cultural movement (2009, Patrick Debois) Google's engineering discipline (~2003) Modern evolution from DevOps
Key Metric Deployment frequency, lead time Error budget, SLO / SLA adherence DORA metrics, cognitive load reduction
Primary Tool CI/CD pipelines, IaC Incident management, runbooks, SLOs Internal Developer Platform (IDP)
Team Structure Embedded in product squads Dedicated SRE team Dedicated platform team
When Things Break Dev + Ops shared on-call SRE owns incident response + postmortems Platform team ensures resilient tooling
Key Concept You build it, you run it Error budget — spend wisely on risk Paved roads — golden paths for developers
SRE in One Concept
  • 📉 SLO (Service Level Objective) — e.g., 99.9% uptime
  • 💰 Error Budget — 0.1% downtime allowed per month
  • ⚖️ Spend budget on features vs. reliability
  • 🚨 Budget exhausted → feature freeze, fix reliability
Platform Engineering Goal
  • 🛣️ Build "golden paths" — opinionated, easy routes
  • 🧰 Self-service infra provisioning for developers
  • ⚡ Reduce cognitive load — devs focus on features
  • 📏 Tools: Backstage, Crossplane, ArgoCD, Flux

6. Key DevOps Metrics — DORA

The 4 research-backed indicators that separate elite DevOps teams from everyone else.

DORA metrics (by Google's DevOps Research and Assessment team) are the 4 key indicators of DevOps performance — elite teams deploy multiple times daily and recover from incidents in under an hour.

Deployment Frequency
How often your organization successfully releases to production. The heartbeat of your DevOps practice.
Elite: Multiple times/day
Lead Time for Changes
Time from a code commit being made to that code successfully running in production.
Elite: < 1 hour
Change Failure Rate
Percentage of production deployments that cause a service degradation or require a hotfix/rollback.
Elite: 0–15%
Time to Restore Service
How quickly you recover when an incident or outage occurs in production. Speed of recovery matters most.
Elite: < 1 hour
Metric What It Measures Low Performers Elite Benchmark
Deployment Frequency How often code reaches production Once per month or less Multiple times per day
Lead Time for Changes Commit to production duration 1 month to 6 months Less than 1 hour
Change Failure Rate % of deployments causing incidents 46–60% of changes fail 0–15%
Time to Restore Service Recovery from production incident 1 week to 1 month Less than 1 hour
Why DORA Metrics Matter
Elite performers on DORA metrics are 2x more likely to meet commercial goals, have 50% lower change failure rates, and recover from incidents 24x faster than low performers. These metrics directly correlate with organizational performance and profitability.

Where DevOps Practices Impact Each Metric

DORA MetricDevOps Practices That Improve It
Deployment Frequency CI/CD automation, trunk-based development, feature flags, small batch sizes
Lead Time for Changes Fast automated tests, pre-commit hooks, streamlined code review, no manual approvals
Change Failure Rate Automated testing, canary deployments, blue-green deployments, comprehensive monitoring
Time to Restore Service Auto-rollback, runbooks, alerting, chaos engineering, on-call rotation, post-mortems
