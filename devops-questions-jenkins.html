J1
What is Jenkins?
Easy
Jenkins is an open-source, Java-based automation server widely used to implement CI/CD pipelines. It automates building, testing, and deploying software.
  • Open-source, written in Java, runs on any OS
  • Originally forked from Hudson in 2011
  • 1800+ plugins in the ecosystem
  • Can be self-hosted or run in containers
  • Supports Pipeline-as-Code via Jenkinsfile
// Minimal Jenkinsfile
pipeline {
    agent any
    stages {
        stage('Hello') {
            steps {
                echo 'Hello from Jenkins!'
            }
        }
    }
}
J2
What is CI/CD?
Easy
CI/CD stands for Continuous Integration / Continuous Delivery (or Deployment). It is a set of practices that automate software building, testing, and release.
  • CI: Developers merge code frequently; automated builds and tests run on every commit
  • CD (Delivery): Software is always in a deployable state; deployment to production is manual trigger
  • CD (Deployment): Every passing build is automatically deployed to production
  • Reduces integration hell and speeds up release cycles
  • Jenkins is one of the most popular CI/CD tools
J3
What is Continuous Integration?
Easy
Continuous Integration (CI) is the practice of automatically building and testing code every time a team member commits changes to version control.
  • Detects integration bugs early when they are cheaper to fix
  • Developers commit small, frequent changes
  • Automated test suite runs on every push
  • Fast feedback loop — developers know within minutes if their code broke something
  • Requires a shared repository and automated build system

What is a Jenkins Job?
Easy
A Jenkins Job (also called a project) is a runnable task configured in Jenkins. It defines what to build, how to build it, and what to do with the results.
  • Types: Freestyle, Pipeline, Multi-branch Pipeline, Organization Folder
  • Each job has: SCM config, build triggers, build steps, post-build actions
  • Jobs are stored as XML config files in JENKINS_HOME
  • Can be parameterized to accept input at runtime
  • Can be chained using upstream/downstream relationships
J6
What is a Jenkins Pipeline?
Easy
A Jenkins Pipeline is a suite of plugins that supports implementing and integrating continuous delivery pipelines into Jenkins. It enables Pipeline-as-Code via Jenkinsfile.
  • Defined in a Jenkinsfile stored alongside application code in SCM
  • Two syntaxes: Declarative (structured) and Scripted (Groovy DSL)
  • Survives Jenkins restarts (durable)
  • Visual stage view shows each stage's status
  • Supports complex workflows: parallel, conditional, approval gates
pipeline {
    agent any
    stages {
        stage('Build') { steps { sh 'mvn package' } }
        stage('Test')  { steps { sh 'mvn test'    } }
        stage('Deploy'){ steps { sh './deploy.sh' } }
    }
}
J7
What is a Jenkinsfile?
Easy
A Jenkinsfile is a text file that contains the definition of a Jenkins Pipeline and is checked into source control. It brings Pipeline-as-Code to Jenkins.
  • Lives in the root of the repository alongside application code
  • Versioned with the code — pipeline changes go through PR review
  • Supports both Declarative and Scripted syntax
  • Jenkins reads it automatically when using Multibranch Pipeline or SCM-triggered jobs
  • Enables the same pipeline definition to be used across branches
J8
What is a Jenkins Agent?
Easy
A Jenkins Agent (formerly called Slave) is a machine or container that connects to the Jenkins controller and executes build steps as directed.
  • Offloads build execution from the controller
  • Can be permanent (always-on VM) or ephemeral (spun up per build)
  • Connects via SSH, JNLP/WebSocket, or Docker
  • Labeled so pipelines can target specific agents (e.g., agent { label 'linux' })
  • Kubernetes plugin creates pod-based ephemeral agents
pipeline {
    agent { label 'linux-docker' }
    stages {
        stage('Build') {
            steps { sh 'docker build -t myapp .' }
        }
    }
}
J9
What is a Jenkins Master/Controller?
Easy
The Jenkins Master (now called Controller) is the central server that manages configuration, schedules builds, dispatches them to agents, and serves the web UI.
  • Hosts the Jenkins web UI and REST API
  • Stores all job configurations, build history, credentials
  • Does NOT run build steps itself in best practice (to avoid resource contention)
  • Manages agent connections and build queue
  • Single point of failure unless configured for HA

  • UI: Manage Jenkins → Plugins → Available → search → Install
  • CLI: jenkins-cli.jar install-plugin <plugin-name>

J11
What is a Build Trigger?
Easy
A Build Trigger is a condition or event that causes Jenkins to start a new build automatically. Triggers can be time-based, event-based, or manually initiated.
  • SCM Poll: Jenkins periodically checks SCM for changes
  • Webhook: SCM pushes a notification to Jenkins on each commit
  • Cron schedule: Build runs at fixed times (H 2 * * *)
  • Upstream trigger: Another job completing triggers this job
  • Manual: User clicks "Build Now" or uses the API
J12
What is a webhook in Jenkins?
Easy
A webhook is an HTTP callback sent by a source control system (GitHub, GitLab, Bitbucket) to Jenkins whenever an event occurs (push, PR, tag). Jenkins then triggers the appropriate pipeline.
  • More efficient than Poll SCM — no polling overhead
  • Requires Jenkins to be reachable from the SCM server (or use a tunnel/relay)
  • GitHub: Settings → Webhooks → add Jenkins URL + /github-webhook/
  • GitLab: Settings → Integrations → Jenkins CI
  • Payload contains branch, commit SHA, author — usable in pipeline
J13
What is a Jenkins workspace?
Easy
A Jenkins workspace is the directory on the agent node where Jenkins checks out source code and runs build steps for a specific job.
  • Default location: JENKINS_HOME/workspace/<job-name>
  • Each job gets its own workspace directory
  • Multiple executors on the same agent get separate workspace directories (with @N suffix)
  • Should be cleaned periodically to avoid disk exhaustion
  • cleanWs() step or "Delete workspace before build" option clears it
J14
What is a Jenkins Executor?
Easy
An Executor is a slot on a Jenkins node (controller or agent) that can run one build at a time. Setting the number of executors controls how many concurrent builds a node handles.
  • Controller default: 2 executors (recommended: 0 for controller in production)
  • Setting executors to 0 on the controller ensures it only orchestrates, never builds
  • Each executor has its own workspace
  • More executors = more parallel builds on that node
  • Tune based on CPU/memory: typically 1 executor per CPU core for CPU-bound builds

What is the difference between Freestyle and Pipeline jobs?
Easy
Freestyle jobs use a GUI-based configuration while Pipeline jobs define the build process as code (Jenkinsfile) stored in SCM.
Freestyle
  • GUI configured
  • No SCM version control for build definition
  • Limited to sequential steps
  • Harder to maintain at scale
  • Good for simple one-off tasks
Pipeline
  • Code-defined in Jenkinsfile
  • Version-controlled with app code
  • Supports parallel/conditional/loops
  • Scales well across teams
  • Supports complex multi-stage workflows
J16
What is a Declarative Pipeline?
Easy
Declarative Pipeline is the newer, more structured syntax for Jenkins pipelines. It uses a predefined structure with pipeline {} as the root block and enforces best practices.
  • Must start with pipeline { } root block
  • Requires agent, stages, stage, steps structure
  • Built-in post, environment, parameters, options directives
  • Easier to read and write; validated at parse time
  • Supports when for conditional stages
pipeline {
    agent any
    environment { APP_ENV = 'staging' }
    stages {
        stage('Build') {
            steps { sh 'mvn package -DskipTests' }
        }
        stage('Test') {
            steps { sh 'mvn test' }
        }
    }
    post {
        failure { mail to: 'team@example.com', subject: 'Build Failed' }
    }
}
J17
What is a Scripted Pipeline?
Easy
Scripted Pipeline is the original Pipeline-as-Code syntax for Jenkins, based on Groovy DSL with a node {} block at the root. It offers full programmatic power but less structure.
  • Uses node { } as root block
  • Full Groovy programming available (loops, classes, try/catch)
  • More flexible but harder to read for non-developers
  • No built-in declarative constructs like post or when
  • Being superseded by Declarative for most use cases
node('linux') {
    try {
        stage('Checkout') { checkout scm }
        stage('Build')    { sh 'mvn package' }
        stage('Test')     { sh 'mvn test'    }
    } catch(e) {
        currentBuild.result = 'FAILURE'
        throw e
    } finally {
        cleanWs()
    }
}
J18
What is the pipeline block in a Jenkinsfile?
Easy
The pipeline block is the mandatory top-level block in a Declarative Pipeline. Everything in the pipeline definition must be inside this block.
  • Must be the outermost block in a Declarative Jenkinsfile
  • Contains: agent, environment, options, parameters, triggers, stages, post
  • Parsed and validated by Jenkins before execution begins
  • Cannot contain arbitrary Groovy code at the top level (use script {} blocks for that)
  • Only one pipeline block allowed per Jenkinsfile
J19
What is the agent directive?
Easy
The agent directive specifies where the pipeline or a specific stage will execute. It can target the controller, any available agent, a labeled agent, or a Docker container.
  • agent any — run on any available agent
  • agent none — no global agent; each stage must declare its own
  • agent { label 'linux' } — run on agent with label 'linux'
  • agent { docker 'maven:3.9' } — run inside a Docker container
  • agent { kubernetes { yaml '...' } } — run in a Kubernetes pod
pipeline {
    agent { docker { image 'node:20-alpine' } }
    stages {
        stage('Install') { steps { sh 'npm ci' } }
    }
}
J20
What is the stages directive?
Easy
The stages directive is a required section in a Declarative Pipeline that contains one or more stage blocks. It represents the sequence of stages in the pipeline.
  • Must contain at least one stage
  • Stages execute sequentially by default (unless parallel is used)
  • Each stage appears as a column in the Stage View UI
  • Only one stages block allowed at the top level of the pipeline
  • Nested stages are supported for grouping
J21
What is the stage directive?
Easy
A stage directive defines a single phase of the pipeline (e.g., Build, Test, Deploy). It groups related steps and shows as a named block in Jenkins UI.
  • Requires a name string argument: stage('Build') { ... }
  • Contains steps, parallel, stages, agent, when, or matrix
  • Each stage's duration, status, and logs are tracked separately
  • Stages are visible in the Stage View and Blue Ocean UI
  • A stage can have its own agent override
J22
What is the steps directive?
Easy
The steps directive contains the actual build instructions to execute within a stage. It is a required block inside every stage (unless using parallel or matrix).
  • Contains Jenkins DSL steps: sh, echo, git, checkout, archiveArtifacts, etc.
  • Steps execute sequentially within the block
  • A failing step (non-zero exit code) marks the stage as failed
  • script {} block inside steps allows arbitrary Groovy
stage('Build') {
    steps {
        sh 'mvn clean package'
        archiveArtifacts artifacts: 'target/*.jar'
        script {
            def version = sh(returnStdout: true, script: 'mvn help:evaluate -Dexpression=project.version -q -DforceStdout').trim()
            echo "Building version: ${version}"
        }
    }
}
J23
What is the sh step?
Easy
The sh step executes a shell command on a Unix/Linux agent. It is one of the most commonly used steps in Jenkins pipelines.
  • Runs commands via /bin/sh -xe by default
  • -x prints each command before execution (visible in logs)
  • -e exits on first non-zero return code (fail-fast)
  • returnStdout: true captures command output as a string
  • returnStatus: true returns exit code instead of throwing on failure
steps {
    // Basic usage
    sh 'npm run build'

    // Capture output
    def tag = sh(returnStdout: true, script: 'git describe --tags').trim()
    echo "Latest tag: ${tag}"

    // Check exit code without failing
    def status = sh(returnStatus: true, script: 'docker inspect myapp')
    if (status != 0) { echo 'Image does not exist yet' }
}
J24
What is the post block?
Easy
The post block defines actions that run after a pipeline or stage completes, based on the build result. It is used for notifications, cleanup, and artifact publishing.
  • Defined at pipeline level or within individual stages
  • Conditions: always, success, failure, unstable, aborted, changed, fixed, regression
  • Multiple conditions can be specified in one post block
  • Executes even if the pipeline was aborted
  • Common uses: send notifications, archive logs, clean workspace
post {
    always    { cleanWs() }
    success   { slackSend channel: '#deploys', message: "SUCCESS: ${env.JOB_NAME}" }
    failure   { emailext to: 'oncall@example.com', subject: 'FAILED: ${env.JOB_NAME}' }
    unstable  { echo 'Tests had failures — marking as unstable' }
}
J25
What is always, success, failure in the post block?
Easy
These are post-condition keywords that determine when the associated steps execute based on the pipeline build result.
Condition When it runs
always Regardless of outcome — cleanup, logging
success Only when build succeeded (result = SUCCESS)
failure Only when build failed (result = FAILURE)
unstable Build result is UNSTABLE (e.g., test failures)
aborted Build was manually aborted
changed Result changed from previous build (failure→success or vice versa)
fixed Previous build failed, this one succeeded
regression Previous build succeeded, this one failed

What is the environment directive in Jenkinsfile?
Easy

The environment directive defines environment variables available to all steps in the pipeline or a specific stage. Values can be static strings or dynamically sourced from Jenkins credentials.

  • Can be defined at pipeline level (global) or stage level (scoped to that stage only)
  • Accessed in steps as ${env.VAR_NAME} or as $VAR_NAME in shell scripts
  • Overrides OS environment variables of the same name during the build
  • credentials('id') helper automatically masks credential values in logs
  • Username/password credentials split into _USR and _PSW suffixed vars automatically
pipeline {
    environment {
        APP_ENV     = 'production'
        REGISTRY    = 'registry.example.com'
        DOCKER_CRED = credentials('docker-registry-cred')
        // Creates DOCKER_CRED_USR and DOCKER_CRED_PSW automatically
    }
    stages {
        stage('Build') {
            environment { BUILD_FLAGS = '--no-cache' }
            steps {
                sh "docker build ${BUILD_FLAGS} -t ${REGISTRY}/myapp:${BUILD_NUMBER} ."
                sh 'echo "$DOCKER_CRED_PSW" | docker login -u "$DOCKER_CRED_USR" --password-stdin'
            }
        }
    }
}
J27
What is the parameters directive?
Easy

The parameters directive declares user-provided input fields that appear on the "Build with Parameters" page, allowing the same pipeline to behave differently based on runtime input.

  • Types: string, booleanParam, choice, text, password, file
  • Accessible inside the pipeline as params.PARAM_NAME
  • First build after adding parameters may not show them — run once to register them in Jenkins
  • Parameters can be set via UI, REST API, or from an upstream job's build step
  • Can drive conditional logic: when { expression { params.DEPLOY == true } }
pipeline {
    agent any
    parameters {
        string(name: 'VERSION', defaultValue: 'latest', description: 'Docker image version')
        booleanParam(name: 'DEPLOY', defaultValue: false, description: 'Deploy to production?')
        choice(name: 'ENV', choices: ['dev', 'staging', 'prod'], description: 'Target environment')
    }
    stages {
        stage('Deploy') {
            when { expression { params.DEPLOY == true } }
            steps { sh "./deploy.sh ${params.ENV} ${params.VERSION}" }
        }
    }
}

What are build parameters?
Easy

Build parameters are user-defined inputs that customize how a Jenkins build runs. They are passed at build time and allow the same job to serve multiple use cases without code changes.

  • Allow the same pipeline to target different environments, versions, or feature flags per run
  • Set via UI ("Build with Parameters" button), REST API, or upstream job's build step
  • Stored in the params map — accessed as params.MY_PARAM
  • Common uses: environment selection, version pinning, feature toggles, rollback targets
  • Sensitive parameters (password type) are masked in logs and input fields

J30
What is GIT_BRANCH environment variable?
Easy

GIT_BRANCH is a Jenkins environment variable set by the Git plugin containing the name of the branch being built (e.g., origin/main). In Multibranch pipelines, use BRANCH_NAME instead.

  • Populated when the pipeline's SCM is configured as Git
  • In Multibranch pipelines, env.BRANCH_NAME is cleaner — no origin/ prefix
  • Used for conditional deployment: only deploy main branch to production
  • GIT_COMMIT contains the full 40-character commit SHA
  • GIT_AUTHOR_NAME and GIT_COMMITTER_EMAIL also available from the Git plugin
// Declarative — cleanest approach
stage('Deploy') {
    when { branch 'main' }
    steps { sh './deploy-prod.sh' }
}

// Scripted — using env.BRANCH_NAME
if (env.BRANCH_NAME == 'main') {
    sh './deploy-prod.sh'
} else {
    echo "Skipping prod deploy on branch: ${env.BRANCH_NAME}"
}
J31
How do you archive artifacts in Jenkins?
Easy

Artifacts are archived using the archiveArtifacts step, which copies matching files from the build workspace to the Jenkins controller for later download, reference, or sharing with downstream jobs.

  • Files stored in JENKINS_HOME/jobs/<name>/builds/<num>/archive/
  • Glob pattern selects files: target/*.jar, reports/**/*.xml
  • fingerprint: true enables artifact tracking and upstream/downstream linking
  • allowEmptyArchive: true — don't fail if no files match (optional artifacts)
  • Clean up old artifacts automatically using buildDiscarder with artifactNumToKeepStr
stage('Archive') {
    steps {
        sh 'mvn package'
        archiveArtifacts artifacts: 'target/*.jar', fingerprint: true, allowEmptyArchive: false
        archiveArtifacts artifacts: 'reports/**/*.xml', allowEmptyArchive: true
    }
}

What is JUnit test result publishing?
Easy

JUnit test result publishing parses XML test reports and displays pass/fail counts, trend graphs, and failed test details directly in the Jenkins build page, providing visibility into test health over time.

  • Step: junit 'target/surefire-reports/*.xml'
  • Works with any framework outputting JUnit XML: JUnit, TestNG, pytest, Jest, Mocha, NUnit
  • Failed tests change build result to UNSTABLE (not FAILURE) by default
  • Trend graphs across builds show test count, failure count, and skip count
  • Failed tests listed with stack traces and class names for direct debugging
post {
    always {
        // Java/Maven
        junit 'target/surefire-reports/*.xml'
        // JavaScript Jest (with jest-junit reporter)
        junit 'junit-results.xml'
        // Python pytest (with --junit-xml flag)
        junit 'test-results/pytest.xml'
    }
}

What is a Jenkins credential?
Easy

A Jenkins credential is a securely stored secret — password, SSH key, API token, certificate, or file — managed by the Credentials plugin, referenced in pipelines by a unique ID string and never exposed in plain text.

  • Stored encrypted in credentials.xml in JENKINS_HOME using the Jenkins master key
  • Never shown in plain text in the UI (masked on all display screens)
  • Scopes: Global (all jobs), System (Jenkins internals), or Folder-scoped
  • Types: Username+Password, Secret Text, SSH Private Key, Certificate, Secret File
  • Referenced by unique ID: credentials('my-docker-cred') or credentialsId: 'my-docker-cred'
J34
How do you use credentials in a Jenkinsfile?
Easy

Credentials are injected via the credentials() helper in the environment block for pipeline-wide use, or via withCredentials step for scoped injection. Values are always automatically masked in logs.

  • environment { CRED = credentials('my-id') } — injects as masked env var pipeline-wide
  • withCredentials([...]) — explicit step block, most flexible, scoped to that block only
  • SSH keys: sshagent(['my-ssh-key']) { sh 'git push ...' }
  • Secret text: withCredentials([string(credentialsId: 'token', variable: 'TOKEN')])
  • Never use Groovy interpolation "${TOKEN}" — use single-quoted sh strings '$TOKEN'
withCredentials([usernamePassword(
    credentialsId: 'docker-hub-cred',
    usernameVariable: 'DOCKER_USER',
    passwordVariable: 'DOCKER_PASS'
)]) {
    sh '''
        echo "$DOCKER_PASS" | docker login -u "$DOCKER_USER" --password-stdin
        docker push myapp:latest
    '''
}

What is a Jenkins node label?
Easy

A node label is a tag assigned to a Jenkins agent that pipelines reference to route builds to agents with specific capabilities — such as OS type, installed tools, or hardware characteristics.

  • Assigned in: Manage Jenkins → Nodes → Configure Node → Labels field
  • Multiple labels per node, space-separated: linux docker maven large-mem
  • Pipeline targeting: agent { label 'linux && docker' }
  • Boolean expressions supported: label 'linux && !arm64'
  • Enables routing: Windows builds to Windows agents, GPU builds to GPU nodes, etc.
J39
How do you schedule a Jenkins job (cron)?
Easy

Jenkins jobs are scheduled using cron syntax in the triggers { cron(...) } directive or the "Build periodically" UI option. The H symbol spreads load across the time window.

  • Format: MINUTE HOUR DOM MONTH DOW (same as Unix cron)
  • H uses a hash of the job name to distribute builds — prevents all jobs firing at minute 0
  • Aliases: @daily, @weekly, @midnight, @hourly
  • Multiple schedules: separate with newline inside the cron string
  • Different from Poll SCM — cron runs unconditionally, Poll SCM only if there are changes
pipeline {
    triggers {
        cron('H 2 * * *')         // nightly at some point in the 2am hour
        cron('H/15 * * * *')      // every 15 minutes
        cron('H 6 * * 1-5')       // weekdays at ~6am
    }
    stages {
        stage('Nightly Build') { steps { sh 'mvn clean install' } }
    }
}
J40
What is Poll SCM in Jenkins?
Easy

Poll SCM is a build trigger where Jenkins periodically checks the source code repository for new commits and triggers a build only when changes are detected — a polling alternative to webhooks.

  • Less efficient than webhooks — creates polling overhead on both Jenkins and the SCM server
  • Configured with cron-like syntax: H/5 * * * * = poll every 5 minutes
  • Useful when Jenkins cannot receive inbound webhooks (strict firewall environments)
  • Pipeline syntax: triggers { pollSCM('H/5 * * * *') }
  • "Git Polling Log" on the job page shows timestamp and result of each poll
J41
What is workspace cleanup in Jenkins?
Easy

Workspace cleanup deletes the build workspace directory (checked-out source and build artifacts) to ensure clean builds and prevent disk space exhaustion on Jenkins agents over time.

  • cleanWs() step (Workspace Cleanup plugin) — deletes workspace at any pipeline point
  • Best practice: call in post { always { cleanWs() } } to clean after every build
  • "Delete workspace before build starts" checkbox available in Freestyle job config
  • ws('custom-dir') step lets you specify a custom workspace path per stage
  • Ephemeral agents (Docker containers / K8s pods) auto-clean on termination — no explicit cleanup needed
J42
How do you chain Jenkins jobs?
Easy

Jenkins jobs are chained using the build job: 'downstream-job' step, which triggers another pipeline job and optionally waits for its result and passes parameters to it.

  • build job: 'job-name' — triggers and waits by default
  • wait: false — fire-and-forget; returns immediately
  • propagate: false — don't fail this build if downstream fails
  • Pass parameters: parameters: [string(name: 'VERSION', value: env.BUILD_NUMBER)]
  • "Build other projects" in Freestyle post-build actions is the GUI equivalent
stage('Trigger Deploy') {
    steps {
        build job: 'deploy-to-staging',
              parameters: [string(name: 'IMAGE_TAG', value: env.BUILD_NUMBER)],
              wait: true,
              propagate: true  // fail this build if deploy fails
    }
}
J43
What is Jenkins upstream/downstream job?
Easy

Upstream jobs trigger other jobs (downstream jobs) upon completion, creating a dependency chain. Upstream = the triggering job; Downstream = the triggered job.

  • Example chain: Build App (upstream) → Run Integration Tests → Deploy to Prod (downstream)
  • Visible on each job page under "Upstream Projects" and "Downstream Projects" sections
  • fingerprint: true on archived artifacts enables automatic upstream/downstream linking
  • Useful for separating build, test, and deploy into independent, reusable job units
  • Multiple upstream jobs can all trigger the same downstream job

What is Jenkins home directory?
Easy

What is a Jenkins build queue?
Easy

The build queue is a list of triggered builds waiting for an available executor. Builds enter the queue when all matching executors are busy, and are dispatched as soon as a slot opens up.

  • Queue shows: job name, wait time, and reason (e.g., "Waiting for next available executor" or "Offline")

  • Can be manually cleared by clicking the X next to a queued build
  • A growing queue is a key health signal — monitor queue depth as a Jenkins performance metric
  • Common causes: all agents busy, required label has no available agent, disableConcurrentBuilds blocking

What is the timeout step in Jenkins?
Easy

The timeout step wraps a block and aborts it — marking the build ABORTED — if it does not complete within the specified time. It prevents hung builds from blocking agents indefinitely.

  • In options {}: applies to the entire pipeline run
  • As a step: applies only to the wrapped block
  • Time units: SECONDS, MINUTES, HOURS
  • Timeout causes ABORTED build result (not FAILURE)
  • activity: true variant aborts only if there is no console output for the duration
// Pipeline-level timeout in options

// Step-level timeout
timeout(time: 5, unit: 'MINUTES') {
    sh 'mvn integration-test'
}

// Activity-based timeout (abort if silent for 10 minutes)
timeout(time: 10, unit: 'MINUTES', activity: true) {
    sh './run-slow-test.sh'
}

What is the retry step in Jenkins?
Easy

The retry step re-executes a block of steps up to N times if it fails, making pipelines resilient to transient errors like network timeouts, rate limits, or flaky tests.

  • retry(3) { ... } — tries up to 3 times total (1 original + 2 retries)
  • On success, exits immediately without unnecessary additional attempts
  • All retries run in the same stage context — workspace is not reset between retries
  • Combine with timeout to prevent infinite hang during each attempt
  • Does not catch ABORTED — only catches FAILURE
// Basic retry for flaky network calls

}

Explain the Jenkins architecture in detail.
Medium

Jenkins uses a Controller-Agent architecture where the controller manages configuration, scheduling, and UI, while agents execute the actual build work. They communicate via SSH or JNLP.

  • Controller: serves the web UI, REST API, manages build scheduling, stores credentials, handles plugin management — never runs builds directly in production setups
  • Agent (Node): executes pipeline steps and build commands, reports results and logs back to controller in real-time

  • Executor slots: each node has a configurable number of executors; multiple executors allow concurrent builds on the same agent
  • Workspace: each build gets an isolated directory on the agent where source code is checked out and artifacts produced

Explain Jenkins internal architecture — how builds are scheduled and executed.
Hard

Jenkins uses a thread-based execution model. The controller's Trigger thread polls for events, adds to queue, and Executors dispatch to agents.

  • Trigger thread: monitors SCM pollers, cron timers, webhook callbacks — calls job.scheduleBuild2()
  • Build Queue: in-memory priority queue; builds wait here until an executor is available
  • Queue Sorter: default sorts by timestamp; priority sorter plugin adds priority levels
  • Label matching: queue item specifies label expression; dispatcher finds matching idle executor
  • Executor: thread on a node that runs a build; each node has a fixed pool of executors
  • CPS (Continuation Passing Style) VM: pipeline steps are serialized and checkpointed to JENKINS_HOME for durability
  • Flyweight tasks: lightweight tasks (e.g., Pipeline orchestration) run on the controller's flyweight executor
  • Work items on agents run via Remoting library (channel over SSH/JNLP)

    steps {
        script {

    }
}

What are the security risks of Jenkins Script Console?
Hard

The Jenkins Script Console executes arbitrary Groovy code on the Jenkins controller with full JVM access — it is one of the highest-privilege surfaces in a Jenkins installation.

    NEW_VERSION  = "${BUILD_NUMBER}"
    NAMESPACE    = 'production'

    stage('Build & Push') {
      steps {
        sh "docker build -t myrepo/${APP_NAME}:${NEW_VERSION} ."
        sh "docker push myrepo/${APP_NAME}:${NEW_VERSION}"

      steps {

Difference between sh 'echo $BUILD_NUMBER' and sh "echo $BUILD_NUMBER" in Groovy?
Tricky

Single quotes pass the literal string to the shell; double quotes let Groovy interpolate before the shell ever sees it — same output, completely different mechanism, and dangerously different for secrets.

  • Single-quoted Groovy string: sh 'echo $BUILD_NUMBER' — the string echo $BUILD_NUMBER is passed verbatim to /bin/sh; the shell expands $BUILD_NUMBER from its environment variables at runtime
  • Double-quoted Groovy GString: sh "echo $BUILD_NUMBER" — Groovy evaluates $BUILD_NUMBER on the JVM, producing the string echo 42, which is then handed to the shell; the shell never sees a variable reference
  • For Jenkins env vars both work — but they work via completely different paths; the double-quoted version bakes the value in at Groovy evaluation time, not at step execution time
  • Secrets gotcha — NEVER use double quotes: sh "curl -H 'Token: ${SECRET}'" expands SECRET in Groovy before the credentials masking plugin can act on log output — the raw secret can appear in logs
  • Lazy evaluation trap: in a closure or method, "${someVar}" is evaluated when the string is constructed, not when the sh step runs — if someVar changes between definition and execution, the GString captures the old value
  • Practical rule: always use single quotes for shell scripts that reference env vars; use double quotes only when you need Groovy-side computation that has no shell equivalent

  • How masking works: the Credentials Binding plugin registers the raw credential value as a string to replace in log output — it is a text scan, not a runtime interception of secret usage
  • Groovy interpolation leaks the secret before masking: sh "curl -H 'Token: ${TOKEN}'" — Groovy resolves ${TOKEN} into the command string on the JVM; the string curl -H 'Token: mysecret123' is formed BEFORE Jenkins logs the command, and the masker may not recognize the value embedded inside the longer string depending on buffering
  • Base64 encoding bypasses masking: sh 'echo -n "$SECRET" | base64' — the Base64-encoded form (e.g., bXlzZWNyZXQ=) is NOT registered with the masker; only the raw value is masked
  • Shell -x flag echoes commands: Jenkins runs sh with -xe by default — set -x causes the shell to print each command before executing; if the command contains the secret (via interpolation), the secret appears in the echo
  • Fix — always single-quote with withCredentials: sh 'curl -H "Token: $SECRET"' — the shell expands $SECRET at runtime, the masker intercepts the log output and replaces the value
  • Fix — disable command echoing: add set +x at the start of the shell script to prevent the shell from printing commands that contain secret values
withCredentials([string(credentialsId: 'api-token', variable: 'TOKEN')]) {

    // DANGEROUS — Base64-encoded form is NOT masked

    // log shows the base64 value unmasked

    sh 'curl -H "Authorization: Bearer $TOKEN" https://api.example.com'

  • env.FOO = 'newval' DOES work: assigning to the env object inside a script {} block updates the environment variable for the current and all subsequent stages in that pipeline run
  • Persistence: changes to env.VAR persist for the rest of the pipeline run — they do NOT reset between stages
  • Parallel branch isolation: environment variable changes made in one parallel branch are NOT visible in sibling parallel branches — each branch runs in its own thread with its own env copy
  • String only: env variables are always strings — you cannot store a Groovy Map or List in an env var; use @Field in a shared library or a global script variable instead
  • Shell visibility: after env.FOO = 'newval' in a script {} block, a subsequent sh 'echo $FOO' step will print the new value — the change propagates to spawned shell processes

  • Executor starvation risk with agent any: if 10 pipelines each hold an executor while waiting on input approval, and you only have 10 executors, the 11th pipeline can never start — classic deadlock scenario
  • agent none: no global executor is allocated; the pipeline controller thread runs on the Jenkins master (or built-in node); each stage must declare its own agent directive

  • Mixed agent types: agent none is required when different stages need different node labels (e.g., build on linux, test on windows, deploy on deploy-node)
  • Stage with no agent in agent none pipeline: if a stage does not declare its own agent, it fails — every stage must have an agent when the top-level is agent none

  • returnStdout: true also swallows errors: when using returnStdout: true, a non-zero exit code DOES still fail the build (unlike returnStatus: true) — but many developers assume they're equivalent

  • Exception thrown: org.jenkinsci.plugins.workflow.steps.FlowInterruptedException — a checked exception that propagates up through the stage and pipeline
  • Post condition behavior: post { always {} } fires; post { aborted {} } fires; post { failure {} } does NOT fire — your Slack/email failure alert is silently skipped
  • Wrapping with catchError: use catchError(buildResult: 'FAILURE', stageResult: 'FAILURE') { timeout(...) { ... } } to convert the aborted timeout into a failure result that triggers failure notifications

  • Activity timeout: timeout(time: 5, unit: 'MINUTES', activity: true) — resets the timer on each log line; useful for processes that should produce output but shouldn't run forever

        failure  { echo 'FAILURE block' }   // does NOT fire on timeout
        aborted  { echo 'ABORTED block' }   // FIRES on timeout
        always   { echo 'ALWAYS block' }    // FIRES on timeout

What is CPS (Continuation Passing Style) in Jenkins pipelines?
Tricky

CPS is a code transformation applied to pipeline Groovy by the workflow-cps plugin that converts the pipeline into a form that can be paused, serialized to disk, and resumed after a Jenkins restart — giving pipelines durability at the cost of performance and some Groovy restrictions.

  • What CPS does: rewrites Groovy bytecode so that every pipeline step invocation is converted into a serializable continuation — instead of a normal call stack, the pipeline's "where we are" state is a data structure that can be written to disk
  • Durability mechanism: pipeline state is periodically serialized to $JENKINS_HOME/jobs//builds//program.dat; if Jenkins crashes or is restarted, the pipeline resumes from the last checkpoint
  • Performance overhead: CPS transformation adds overhead to every step boundary — extremely tight loops (millions of iterations) are significantly slower in CPS-transformed code than in normal Groovy; move CPU-intensive loops to @NonCPS methods
  • @NonCPS annotation: excludes a method from CPS transformation — it runs as a normal JVM method; it is faster for computation, but CANNOT call any pipeline steps (sh, echo, git, etc.) and must return a serializable value
  • CPS restrictions: some Groovy patterns don't survive CPS transformation — method references to non-CPS methods, certain Groovy metaprogramming features, calling Java library methods that rely on thread-local state (e.g., ThreadLocal)
  • Performance durability tuning: Jenkins 2.x added options { durabilityHint('PERFORMANCE_OPTIMIZED') } — reduces checkpoint frequency, faster pipelines, but less durability if Jenkins crashes mid-step

// (thousands of checkpoints, one per iteration)

        script {

            for (int i = 0; i < 100000; i++) {

            }
            echo "Sum: ${sum}"
        }
    }
}

    }
    return sum      // must return serializable value (int is fine)
    // CANNOT call sh(), echo(), or any pipeline steps here
}

        script {
            def sum = computeSum(100000)   // single CPS step, fast computation
            echo "Sum: ${sum}"
        }
    }
}

Why might currentBuild.result be null at the start?
Tricky

currentBuild.result is null while the build is in progress — Jenkins only sets it when the build completes or a step explicitly sets it. Use currentBuild.currentResult for in-flight checks; it is never null.

  • currentBuild.result is null during execution: Jenkins represents an in-progress build as having no result yet — null means "still running, no outcome determined"
  • Silent comparison bug: if (currentBuild.result == 'FAILURE') always evaluates to false during pipeline execution because null != 'FAILURE' — conditions that should fire on failure are silently skipped
  • currentBuild.currentResult is never null: returns the "best result so far" — starts at 'SUCCESS', transitions to 'UNSTABLE' or 'FAILURE' as steps fail; use this for all in-flight checks
  • Setting currentBuild.result: you CAN assign to it — currentBuild.result = 'UNSTABLE' — the assigned value then becomes non-null and readable; useful for marking a build unstable without throwing an error
  • In post {} blocks it IS set: by the time any post condition runs, Jenkins has finalized the build result — currentBuild.result is safe and non-null inside post { always {} }, post { failure {} }, etc.
  • Difference from currentResult: after currentBuild.result = 'UNSTABLE', both result and currentResult reflect the value; but if result is unset mid-pipeline, result is null while currentResult is 'SUCCESS'
pipeline {
    agent any
    stages {
        stage('Test') {
            steps {
                sh 'mvn test'   // suppose this fails, sets currentResult to FAILURE
            }
        }
        stage('Notify In-Flight') {
            steps {
                script {
                    // WRONG — currentBuild.result is null mid-pipeline
                    // null == 'FAILURE' is false — notification is skipped silently
                    if (currentBuild.result == 'FAILURE') {
                        slackSend message: 'Tests failed!'   // never runs
                    }

                    // CORRECT — currentResult is never null
                    if (currentBuild.currentResult == 'FAILURE') {
                        slackSend message: 'Tests failed!'   // runs correctly
                    }
                }
            }
        }
        stage('Mark Unstable') {
            steps {
                script {
                    def testCoverage = sh(returnStdout: true,
                                         script: 'coverage-check.sh').trim().toInteger()
                    if (testCoverage < 80) {
                        currentBuild.result = 'UNSTABLE'  // sets result, now non-null
                        echo "Coverage ${testCoverage}% is below threshold"
                    }
                }
            }
        }
    }
    post {
        always {
            // In post blocks, currentBuild.result IS set — safe to use
            echo "Final result: ${currentBuild.result}"          // 'SUCCESS', 'FAILURE', etc.
            echo "Current result: ${currentBuild.currentResult}" // same in post context
        }
        failure {
            // currentBuild.result == 'FAILURE' here — reliable
            slackSend message: "Build ${env.BUILD_NUMBER} failed: ${currentBuild.result}"
        }
        unstable {
            slackSend message: "Build ${env.BUILD_NUMBER} is unstable — check coverage"
        }
    }
}

In what order do post conditions execute?
Tricky

Post conditions execute in a fixed internal order defined by Jenkins — declaration order in your Jenkinsfile is irrelevant. The order is: always → changed → fixed → regression → aborted → failure → success → unstable → cleanup.

  • Declaration order does NOT matter: Jenkins defines a fixed execution order internally — writing post { failure {} always {} } does not make failure run before always
  • always runs first: regardless of build result, always fires first — good for initializing notification context or gathering artifacts
  • changed, fixed, regression: compare against the previous build's result; useful for "build went from red to green" notifications; these fire before result-specific conditions
  • cleanup runs last: introduced in Jenkins 2.x; guaranteed to run after all other post conditions even if earlier post conditions fail — ideal for workspace cleanup, temp file deletion
  • Multiple conditions can fire: on a FAILURE build, both always AND failure fire; on a previously-failing build that now succeeds, always + fixed + success all fire
  • Post condition failure: if a post condition block itself throws an error, subsequent post conditions still run (Jenkins catches per-condition failures) — cleanup is always reached regardless
pipeline {
    agent any
    stages {
        stage('Build') {
            steps { sh 'mvn package' }
        }
    }
    post {
        // Execution order is FIXED by Jenkins — declaration order below is irrelevant

        // 1. always — runs first, always
        always {
            echo '1: always fires first'
            archiveArtifacts artifacts: '**/target/*.jar', allowEmptyArchive: true
        }

        // 2. changed — if result differs from previous build
        changed {
            echo '2: result changed from previous build'
            slackSend message: "Build status changed to ${currentBuild.currentResult}"
        }

        // 3. fixed — previous build was failure/unstable, this is success
        fixed {
            echo '3: build is fixed!'
            slackSend color: 'good', message: "Build fixed: ${env.JOB_NAME}"
        }

        // 4. regression — previous build was success, this is failure
        regression {
            echo '4: build regressed (was passing, now failing)'
        }

        // 5. aborted — build was aborted (timeout, manual abort, input timeout)
        aborted {
            echo '5: build was aborted'
        }

        // 6. failure — build failed
        failure {
            echo '6: build failed'
            slackSend color: 'danger',
                       message: "FAILED: ${env.JOB_NAME} #${env.BUILD_NUMBER}"
        }

        // 7. success — build succeeded
        success {
            echo '7: build succeeded'
        }

        // 8. unstable — build marked unstable (test failures, low coverage)
        unstable {
            echo '8: build is unstable'
        }

        // 9. cleanup — runs LAST, even if earlier post blocks fail
        cleanup {
            echo '9: cleanup — always runs last'
            deleteDir()    // clean workspace — runs even if failure{} threw an error
        }
    }
}

// On a FAILURE build, execution order is:
// always(1) → failure(6) → cleanup(9)
// On a FIXED build (was failure, now success):
// always(1) → changed(2) → fixed(3) → success(7) → cleanup(9)

Difference between properties([...]) at top level and options {} block?
Tricky

properties([]) is a Scripted Pipeline step that sets job properties at runtime — changes take effect for the NEXT build, and omitting a property removes it. options {} is a Declarative directive validated at parse time that applies to the current build.

  • properties([...]) — Scripted/runtime step: a Groovy function call that programmatically configures job-level properties; can be called conditionally, from within a stage, or based on dynamic logic; takes effect starting from the next build run
  • "Next build" gotcha: when a Scripted Pipeline runs properties([...]), the properties (parameters, build retention, etc.) defined there are saved to the job config but the CURRENT running build was already started with the old configuration — new parameters only appear on the next trigger
  • properties() overwrites ALL properties: every call to properties() replaces the entire job property configuration — if you define parameters in the Jenkins UI and then call properties() without including them, they are silently deleted
  • options {} — Declarative directive: declared inside the pipeline {} block; validated at parse time before execution begins; applies to the current build; merges with existing job configuration
  • Use options {} in Declarative pipelines: never call properties() inside a script {} block in a Declarative pipeline — it fights with the Declarative configuration and produces unpredictable results
  • Common properties() pattern in Scripted: used at the top of Scripted pipelines to declare parameters so params.X is available — but new parameters only show in the "Build with Parameters" UI after one pipeline run
// SCRIPTED PIPELINE — properties() called as a function
// Properties take effect on the NEXT build, not the current one
node {
    properties([
        buildDiscarder(logRotator(numToKeepStr: '10')),
        disableConcurrentBuilds(),
        parameters([
            string(name: 'DEPLOY_ENV',
                   defaultValue: 'staging',
                   description: 'Target environment'),
            booleanParam(name: 'SKIP_TESTS',
                         defaultValue: false,
                         description: 'Skip test stage')
        ])
    ])
    // WARNING: on first run, params.DEPLOY_ENV may not exist yet
    // (default value used), because properties() just NOW defined it

    stage('Build') {
        sh 'mvn package'
    }
    stage('Deploy') {
        if (!params.SKIP_TESTS) {
            sh 'mvn test'
        }
        sh "deploy.sh ${params.DEPLOY_ENV}"
    }
}

// DECLARATIVE PIPELINE — options{} and parameters{} are directives
// Validated at parse time, apply to the current build, merged (not replaced)
pipeline {
    agent any

    options {
        buildDiscarder(logRotator(numToKeepStr: '10'))
        disableConcurrentBuilds()
        timeout(time: 1, unit: 'HOURS')
        timestamps()
    }

    parameters {
        string(name: 'DEPLOY_ENV',
               defaultValue: 'staging',
               description: 'Target environment')
        booleanParam(name: 'SKIP_TESTS', defaultValue: false)
    }

    stages {
        stage('Build') { steps { sh 'mvn package' } }
        stage('Deploy') {
            steps {
                sh "deploy.sh ${params.DEPLOY_ENV}"
            }
        }
    }
}
