Kubernetes (K8s) is an open-source container orchestration platform that automates deployment, scaling, and management of containerized applications. Created by Google in 2014, open-sourced and donated to CNCF.

  • Origin: Born from Google's internal system "Borg" — managing billions of containers per week
  • Problem solved: Running containers at scale is complex — scheduling, healing, networking, storage
  • Core features: Auto-scaling, self-healing, load balancing, rolling updates, secret management
  • CNCF: Graduated CNCF project — the de-facto standard for container orchestration
  • "K8s": 8 letters between K and s in "Kubernetes"

A Pod is the smallest deployable unit in Kubernetes — a wrapper around one or more containers that share the same network namespace, IP address, and storage volumes.

  • Shared network: All containers in a pod share localhost — they communicate via ports
  • Shared storage: Pods can mount shared volumes accessible to all containers inside
  • Ephemeral: Pods are mortal — when they die, they're gone (controllers recreate them)
  • One IP: Each pod gets a unique cluster IP — not each container
  • Co-location: Containers in a pod always run on the same node
apiVersion: v1
kind: Pod
metadata:
  name: my-pod
spec:
  containers:
  - name: app
    image: nginx:latest
    ports:
    - containerPort: 80

A Node is a physical or virtual machine that runs pods. It's the worker machine in a Kubernetes cluster, managed by the control plane.

  • Components: kubelet (agent), kube-proxy (networking), container runtime (Docker/containerd)
  • Two types: Master node (control plane) and Worker node (runs workloads)
  • Resources: CPU, memory, disk — allocatable to pods
  • Node status: Ready, NotReady, SchedulingDisabled
kubectl get nodes
kubectl describe node <node-name>

A Kubernetes Cluster is a set of nodes (machines) that run containerized applications, managed by a control plane. It's the top-level unit of Kubernetes infrastructure.

  • Control plane: Brain of the cluster — API Server, etcd, Scheduler, Controller Manager
  • Worker nodes: Run actual application pods
  • High availability: Production clusters have multiple control plane nodes
  • Scale: A cluster can have thousands of nodes

A Namespace is a virtual cluster within a Kubernetes cluster — it provides a mechanism for isolating groups of resources within a single cluster.

  • Isolation: Resources in one namespace are hidden from others by default
  • Default namespaces: default, kube-system, kube-public, kube-node-lease
  • Use cases: Separate dev/staging/prod, or different teams on same cluster
  • Resource quotas: Limit CPU/memory per namespace
  • Cluster-scoped resources: Nodes, PVs, ClusterRoles are NOT namespaced
kubectl get namespaces
kubectl create namespace dev
kubectl get pods -n dev

A Deployment is a higher-level object that manages a ReplicaSet to ensure a desired number of pod replicas are running. It enables declarative updates, rolling upgrades, and rollbacks.

  • Desired state: You declare "I want 3 replicas of nginx" — Deployment ensures it
  • Self-healing: If a pod crashes, Deployment recreates it automatically
  • Rolling updates: Update image version with zero downtime
  • Rollback: Instantly revert to previous version if something goes wrong
  • Manages ReplicaSets: Creates/manages ReplicaSets under the hood
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deployment
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.21
        ports:
        - containerPort: 80

A Service is an abstraction that defines a stable network endpoint to access a set of pods. Since pods are ephemeral with changing IPs, Services provide a consistent DNS name and IP.

  • Stable endpoint: Services get a ClusterIP that never changes, even as pods restart
  • Load balancing: Distributes traffic across all healthy pods matching the selector
  • DNS: Accessible via service-name.namespace.svc.cluster.local
  • Types: ClusterIP, NodePort, LoadBalancer, ExternalName

  • CRUD operations: get, create, apply, delete, describe, edit
  • Debugging: logs, exec, port-forward, describe
  • Config: Uses ~/.kube/config to connect to clusters
  • Context: Switch between clusters with kubectl config use-context

# Examples

kubectl apply -f deployment.yaml

A ReplicaSet ensures a specified number of pod replicas are running at any given time. If pods fail or are deleted, it creates new ones to maintain the desired count.

  • Pod selector: Uses label selectors to identify which pods it manages
  • Self-healing: Continuously reconciles actual vs desired replica count
  • Rarely used directly: Use Deployment instead — it manages ReplicaSets for you
  • Replaces ReplicationController: Supports set-based selectors

A ConfigMap stores non-confidential configuration data as key-value pairs. It decouples configuration from container images, making apps portable across environments.

  • Non-sensitive data: App settings, feature flags, config files (not passwords!)
  • Mount as env vars: Inject values as environment variables into pods
  • Mount as files: Mount entire config files into container filesystem
  • Size limit: 1MB per ConfigMap
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  APP_ENV: production
  LOG_LEVEL: info
  config.yaml: |
    server:
      port: 8080

A Secret stores sensitive data (passwords, tokens, keys) in base64-encoded form. It keeps confidential data out of pod specs and application code.

  • Base64 ≠ encryption: Secrets are base64-encoded by default, NOT encrypted — enable EncryptionConfiguration for at-rest encryption
  • Types: Opaque, kubernetes.io/tls, kubernetes.io/dockerconfigjson, etc.
  • Access control: Use RBAC to restrict who can read secrets
  • Use: Mount as env vars or as files in pods
kubectl create secret generic db-secret \
  --from-literal=password=mysecretpassword

# In pod spec:
env:
- name: DB_PASSWORD
  valueFrom:
    secretKeyRef:
      name: db-secret
      key: password

A PersistentVolume (PV) is a piece of storage in the cluster provisioned by an admin or dynamically. It has a lifecycle independent of any pod that uses it.

  • Cluster resource: Like nodes, PVs are cluster-level resources (not namespaced)
  • Access modes: ReadWriteOnce, ReadOnlyMany, ReadWriteMany, ReadWriteOncePod
  • Reclaim policy: Retain, Recycle, Delete — what happens when PVC is deleted
  • Backends: NFS, iSCSI, AWS EBS, GCE PD, Azure Disk, etc.

A PersistentVolumeClaim (PVC) is a request for storage by a user. It binds to a matching PV and allows pods to use persistent storage without knowing the underlying details.

  • Abstraction: Pods reference PVCs, not PVs directly — portability across environments
  • Binding: K8s matches PVC to a suitable PV based on size, access mode, StorageClass
  • Namespaced: PVCs are namespace-scoped; PVs are cluster-scoped
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: my-pvc
spec:
  accessModes: [ReadWriteOnce]
  resources:
    requests:
      storage: 5Gi
  storageClassName: standard

A StatefulSet manages stateful applications — it provides stable network identities, stable persistent storage, and ordered deployment/scaling for pods.

  • Stable identities: Pods get predictable names: pod-0, pod-1, pod-2 (not random hashes)
  • Stable storage: Each pod gets its own PVC that persists across rescheduling
  • Ordered ops: Pods start in order (0→1→2) and delete in reverse (2→1→0)
  • Use cases: Databases (MySQL, PostgreSQL, Cassandra), Kafka, ZooKeeper, Elasticsearch

A DaemonSet ensures that a copy of a pod runs on every (or selected) node in the cluster. When nodes are added, pods are automatically added to them.

  • One pod per node: Exactly one pod per matching node — automatically
  • Use cases: Log collectors (Fluentd), monitoring agents (Datadog), network plugins (Calico), storage daemons
  • Node selector: Can target specific nodes using nodeSelector or affinity
  • Auto-cleanup: When node is removed, pod is garbage collected

A Job creates one or more pods and ensures they run to successful completion. Unlike Deployments, Jobs are for batch/finite tasks — not long-running services.

  • Completion tracking: Runs until specified completions are successful
  • Parallelism: Run multiple pods in parallel with parallelism field
  • Retry on failure: backoffLimit controls how many times to retry on failure
  • Use cases: DB migrations, data processing, batch jobs, one-time scripts

A CronJob creates Jobs on a repeating schedule defined by a cron expression. It's Kubernetes' equivalent of cron in Linux — for scheduled tasks.

apiVersion: batch/v1
kind: CronJob
metadata:
  name: daily-backup
spec:
  schedule: "0 2 * * *"  # 2am daily
  jobTemplate:
    spec:
      template:
        spec:
          containers:
          - name: backup
            image: my-backup:latest
          restartPolicy: OnFailure
  • Timezone: Default UTC — use timeZone field (K8s 1.27+)
  • Missed schedules: If >100 schedules missed, controller stops creating jobs
  • concurrencyPolicy: Allow, Forbid, Replace — controls overlapping runs

Labels are key-value pairs attached to Kubernetes objects. Selectors filter objects by their labels — enabling Services, Deployments, and other controllers to target specific pods.

  • Labels: Arbitrary metadata — app: nginx, env: prod, version: 1.2
  • Equality-based: app = nginx, env != dev
  • Set-based: env in (prod, staging), tier notin (frontend)
  • How Services use them: selector: {app: nginx} routes traffic to pods with that label
# Filter pods by label
kubectl get pods -l app=nginx
kubectl get pods -l 'env in (prod,staging)'

Annotations are key-value pairs used to attach arbitrary non-identifying metadata to objects. Unlike labels, annotations are not used for selection — they're for tools, libraries, and humans.

  • Not selectable: You can't use annotations in selectors or queries
  • Large values: Can hold larger, richer data than labels
  • Common uses: Build info, Git SHA, deployment tool configs, Prometheus scrape configs
  • Example: kubectl.kubernetes.io/last-applied-configuration stored by kubectl apply

ClusterIP is the default Service type — it creates an internal IP address accessible only within the cluster. No external traffic reaches it directly.

  • Internal only: Only reachable from within the cluster
  • Virtual IP: kube-proxy implements it via iptables/IPVS rules
  • DNS: my-service.my-namespace.svc.cluster.local
  • Use case: Microservice-to-microservice communication

NodePort exposes a service on a static port on each node's IP. External traffic can reach the service via <NodeIP>:<NodePort>.

  • Port range: 30000–32767 by default
  • Accessible: From outside the cluster using any node's IP + port
  • Superset: Also creates a ClusterIP — internal access still works
  • Not ideal for production: Exposes ports on all nodes; use LoadBalancer or Ingress instead

LoadBalancer Service provisions an external load balancer (from the cloud provider) that routes traffic to the service. It's the standard way to expose services to the internet in cloud environments.

  • Cloud integration: Works on AWS (ELB/ALB/NLB), GCP, Azure automatically
  • External IP: Gets a real external IP/hostname from the cloud provider
  • Superset: Also creates NodePort and ClusterIP
  • Cost: Each LoadBalancer Service creates a separate cloud LB — can be expensive at scale (use Ingress instead)

An Ingress is an API object that manages external HTTP/HTTPS access to services in the cluster. It provides load balancing, SSL termination, and name-based virtual hosting.

  • Layer 7: HTTP-aware routing — route by host name and URL path
  • SSL/TLS: Terminate HTTPS at the Ingress level
  • Cost-efficient: One LoadBalancer for many services vs one LB per service
  • Requires: An Ingress Controller to actually implement the rules
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: my-ingress
spec:
  rules:
  - host: myapp.example.com
    http:
      paths:
      - path: /api
        pathType: Prefix
        backend:
          service:
            name: api-service
            port:
              number: 80

An Ingress Controller is a pod that watches Ingress resources and implements the routing rules — typically using nginx, HAProxy, Traefik, or a cloud-native LB.

  • Popular options: nginx-ingress, AWS ALB Ingress Controller, Traefik, Kong, HAProxy
  • Not built-in: Must be installed separately — Ingress objects do nothing without a controller
  • Multiple: A cluster can have multiple ingress controllers (ingressClassName differentiates them)
  • How it works: Watches Ingress objects via K8s API → configures proxy accordingly

The Control Plane is the brain of a Kubernetes cluster — it makes global decisions about the cluster (scheduling, detecting/responding to events) and contains the API Server, etcd, Scheduler, and Controller Manager.

  • kube-apiserver: Front door to the cluster — all communication goes through it
  • etcd: Distributed key-value store — source of truth for all cluster state
  • kube-scheduler: Assigns pods to nodes based on resource availability and constraints
  • kube-controller-manager: Runs controllers that reconcile actual state to desired state
  • cloud-controller-manager: Integrates with cloud provider APIs (optional)

A Worker Node is a machine that runs application pods. It hosts the kubelet, kube-proxy, and container runtime, and executes workloads assigned by the control plane.

  • kubelet: Agent that ensures containers described in PodSpecs are running
  • kube-proxy: Maintains network rules for pod communication and service routing
  • Container runtime: containerd, CRI-O, or Docker (deprecated) — actually runs containers
  • Scales independently: Add more worker nodes to increase cluster capacity

The kube-apiserver is the central management component — it exposes the Kubernetes REST API, validates requests, and is the only component that communicates directly with etcd.

  • All traffic routes through it: kubectl, controllers, kubelet all talk to the API server
  • Authentication & Authorization: Validates credentials and checks RBAC permissions
  • Admission control: Runs admission webhooks before persisting to etcd
  • Watch mechanism: Clients can watch for changes — controllers use this to react
  • Horizontal scaling: Multiple API server instances can run for HA

etcd is a distributed, consistent key-value store used as Kubernetes' backing store for all cluster data — configuration, state, and metadata of every object.

  • Source of truth: Everything in K8s is stored in etcd — nodes, pods, configs, secrets
  • Raft consensus: Uses Raft algorithm to maintain consistency across multiple etcd nodes
  • Quorum: Needs majority of nodes alive (3-node cluster needs 2, 5-node needs 3)
  • Backup critical: Regular etcd backups are essential for disaster recovery

The kube-scheduler watches for new pods with no assigned node and selects the best node for them based on resource requirements, constraints, affinity, and other policies.

  • Two phases: Filtering (eligible nodes) → Scoring (best node)
  • Factors: CPU/memory availability, node affinity, taints/tolerations, topology spread
  • Pluggable: Custom schedulers can be created for specialized workloads
  • Watch loop: Continuously watches for unscheduled pods via the API server watch mechanism

The kube-controller-manager runs multiple controller loops in a single process — each controller watches the state of the cluster and works to bring current state to desired state.

  • Node controller: Notices and responds when nodes go down
  • ReplicaSet controller: Maintains correct pod count for ReplicaSets
  • Endpoints controller: Populates endpoint objects (joins Services and Pods)
  • Service Account controller: Creates default service accounts in new namespaces
  • Job controller: Tracks and manages Job completion

kubelet is the primary node agent that runs on every worker node. It ensures containers described in PodSpecs are running and healthy, and reports node/pod status to the API server.

  • Pod lifecycle: Starts, stops, and restarts containers per the pod spec
  • Health checks: Runs liveness and readiness probes
  • Resource reporting: Reports node capacity and pod resource usage to API server
  • Not a container: kubelet runs directly on the node OS — not in a container
  • CRI: Talks to container runtime via Container Runtime Interface

kube-proxy is a network proxy that runs on each node. It maintains network rules (iptables or IPVS) to implement Service abstractions — routing traffic to the correct pods.

  • iptables mode: Default — creates iptables rules to redirect ClusterIP traffic
  • IPVS mode: Better performance at scale — uses kernel-level load balancing
  • Not a proxy: Despite the name, it doesn't actually proxy traffic — just sets rules
  • Watches Services/Endpoints: Updates rules whenever service membership changes

A Kubernetes manifest file is a YAML (or JSON) file that declaratively describes the desired state of a Kubernetes resource. It's the primary way to define and manage K8s objects.

  • Declarative: You describe WHAT you want, not HOW to achieve it
  • Version controlled: Store in Git for GitOps and auditability
  • Four key fields: apiVersion, kind, metadata, spec
  • Multi-resource: Multiple resources can be in one file separated by ---

kind specifies the type of Kubernetes resource being defined — Pod, Deployment, Service, ConfigMap, etc. It tells the API server what object type to create/manage.

  • Used together with apiVersion to uniquely identify the resource type
  • Case-sensitive and PascalCase: Pod, ReplicaSet, ConfigMap
  • CRDs (Custom Resource Definitions) add new kinds to the cluster

The metadata section contains identifying information about a resource: its name, namespace, labels, annotations, and system-generated fields like UID and resourceVersion.

metadata:
  name: my-deployment          # required, unique in namespace
  namespace: production        # optional, defaults to "default"
  labels:
    app: my-app
    env: prod
    version: "2.1"
  annotations:
    description: "Main web service"
    git-sha: "abc123"

The spec (specification) section defines the desired state of the resource — what you want Kubernetes to create and maintain. Its structure varies by resource kind.

  • Deployment spec: replicas, selector, template (pod template with containers)
  • Pod spec: containers (image, ports, env, resources, volumes)
  • Service spec: type, selector, ports
  • status (read-only): Kubernetes writes actual current state here — you don't define it

Use kubectl scale to imperatively change replica count, or edit the YAML and apply it. For automatic scaling, use HPA.

# Scale imperatively
kubectl scale deployment my-app --replicas=5

# Scale by editing YAML
kubectl edit deployment my-app
# then change spec.replicas

# Scale to zero (stop all pods)
kubectl scale deployment my-app --replicas=0

# Auto-scale with HPA
kubectl autoscale deployment my-app \
  --min=2 --max=10 --cpu-percent=70

A rolling update gradually replaces old pod instances with new ones — ensuring some pods are always running during the update, achieving zero-downtime deployments.

  • maxUnavailable: Max pods that can be unavailable during update (default 25%)
  • maxSurge: Max extra pods created above desired (default 25%)
  • Triggered by: Any change to pod template (image, env, resources)
  • Monitoring: kubectl rollout status deployment/my-app

Use kubectl rollout undo to revert a deployment to a previous revision. Kubernetes keeps a rollout history that you can use to roll back to any version.

# Rollback to previous version
kubectl rollout undo deployment/my-app

# View rollout history
kubectl rollout history deployment/my-app

# Rollback to specific revision
kubectl rollout undo deployment/my-app --to-revision=2

# Check rollout status
kubectl rollout status deployment/my-app

A readiness probe tells Kubernetes whether a container is ready to receive traffic. If the probe fails, the pod is removed from the Service endpoints — no traffic is sent to it.

  • Traffic gate: Pod must pass readiness probe before Service routes traffic to it
  • Probe types: HTTP GET, TCP socket, exec command, gRPC
  • Continuous: Checked periodically throughout pod lifetime (not just at startup)
  • Pod stays up: Failed readiness doesn't restart the container — just removes from endpoints
readinessProbe:
  httpGet:
    path: /ready
    port: 8080
  initialDelaySeconds: 5
  periodSeconds: 10

A liveness probe tells Kubernetes whether a container is alive and functioning. If the probe fails, Kubernetes kills and restarts the container — recovering from deadlocks or crashed states.

  • Restart trigger: Failed liveness probe = container is restarted (per restartPolicy)
  • Use with caution: Overly aggressive probes can cause restart loops
  • failureThreshold: How many consecutive failures before restarting (default 3)
  • vs readiness: Liveness = "is it alive?"; Readiness = "is it ready for traffic?"
livenessProbe:
  httpGet:
    path: /healthz
    port: 8080
  initialDelaySeconds: 30
  periodSeconds: 15
  failureThreshold: 3

Kubernetes has a master-worker architecture. The Control Plane manages cluster state; Worker Nodes run application workloads. All communication flows through the API Server.

Control Plane Components
  • kube-apiserver — REST API gateway
  • etcd — cluster state store
  • kube-scheduler — pod placement
  • kube-controller-manager — reconciliation loops
  • cloud-controller-manager — cloud integration
Worker Node Components
  • kubelet — pod lifecycle agent
  • kube-proxy — network rules
  • Container runtime (containerd/CRI-O)
  • CNI plugin — pod networking
  • Flow: User → kubectl → API Server → etcd (store desired state) → Controller detects delta → Scheduler assigns pod → kubelet runs container
  • HA: Multiple control plane nodes with load-balanced API servers, etcd cluster with odd number of nodes

The scheduler selects a node through a two-phase process: Filtering (eliminates ineligible nodes) then Scoring (ranks remaining nodes) to find the optimal placement.

  • Step 1 — Pod created: Pod object stored in etcd with nodeName empty
  • Step 2 — Scheduler watches: Detects unscheduled pod via watch mechanism
  • Step 3 — Filter: Removes nodes that can't run the pod (insufficient CPU/RAM, taint not tolerated, node selector mismatch, affinity rules)
  • Step 4 — Score: Remaining nodes scored by: least requested resources, image locality, pod/node affinity, spread
  • Step 5 — Bind: Scheduler writes nodeName to pod spec → kubelet on that node picks it up and starts container

Use envFrom to inject all ConfigMap keys as env vars, or env.valueFrom.configMapKeyRef to inject specific keys.

# Inject ALL keys from ConfigMap as env vars
spec:
  containers:
  - name: app
    envFrom:
    - configMapRef:
        name: app-config

# Inject specific key
    env:
    - name: LOG_LEVEL
      valueFrom:
        configMapKeyRef:
          name: app-config
          key: LOG_LEVEL
  • Not auto-updated: Environment variables are set at container start — ConfigMap changes don't automatically update running pods (volume mounts do update)

Mount a Secret as a volume — each key in the Secret becomes a file in the container. The values are base64-decoded automatically.

spec:
  volumes:
  - name: secret-vol
    secret:
      secretName: my-tls-secret
  containers:
  - name: app
    volumeMounts:
    - name: secret-vol

Kubernetes runs CoreDNS as the cluster DNS server. Every pod gets a DNS resolver pointing to CoreDNS, which resolves service names to ClusterIPs using the pattern service.namespace.svc.cluster.local.

  • Full FQDN: my-svc.my-ns.svc.cluster.local
  • Short name: Within same namespace, just my-svc works
  • Cross-namespace: my-svc.other-ns or full FQDN
  • Pod DNS: Pods get DNS: pod-ip.namespace.pod.cluster.local (dashes for IP octets)
  • ndots:5: K8s sets ndots to 5 — short names searched with suffixes before going external

A Headless Service has clusterIP: None — it doesn't provide a virtual IP or load balancing. Instead, DNS returns the actual pod IPs, allowing direct pod-to-pod communication.

  • DNS returns pod IPs: nslookup my-headless-svc returns all pod IPs directly
  • StatefulSets use this: Each pod gets a stable DNS: pod-0.my-svc.ns.svc.cluster.local
  • Client-side LB: The client chooses which pod to connect to
  • Use case: Databases, peer discovery in clustered apps (Cassandra, etcd, Kafka)

  • Migration helper: Move external service to cluster later without changing app code

An Ingress Controller (e.g., nginx) watches Ingress objects via the K8s API. When rules change, it reconfigures its proxy. External traffic hits the Ingress Controller's LoadBalancer IP, which routes by host/path to backend Services.

  • Traffic flow: Client → DNS → LoadBalancer IP → Ingress Controller pod → Service → Pod
  • TLS termination: Ingress handles HTTPS; backend service can be HTTP
  • Annotations: Controller-specific behavior via annotations (rewrite rules, rate limits, auth)
  • IngressClass: Multiple controllers can coexist; IngressClass specifies which handles a given Ingress

A pod goes through distinct lifecycle phases from creation to termination, with conditions and container states tracked throughout.

  • Pending: Pod accepted by cluster; waiting for scheduling and image pull
  • Running: Pod bound to a node; at least one container running/starting/restarting
  • Succeeded: All containers terminated successfully (exit 0)
  • Failed: All containers terminated; at least one failed (non-zero exit or killed)
  • Unknown: Pod state cannot be determined (usually node communication issue)
  • Init containers run and complete first before app containers start
  • postStart hook: Runs immediately after container starts
  • preStop hook: Runs before container is terminated — graceful shutdown
  • terminationGracePeriodSeconds: Default 30s — time for graceful shutdown before SIGKILL

Init containers run before app containers in a pod. They run sequentially to completion — if any init container fails, the pod restarts until it succeeds (or hits backoffLimit).

  • Sequential: Each init container must complete before the next starts
  • Use cases: Wait for a service to be ready, run migrations, download configs, set permissions, clone Git repos
  • Separate image: Can use a different (minimal) image than the main container
spec:
  initContainers:
  - name: wait-for-db
    image: busybox
    command: ['sh', '-c', 'until nc -z db-service 5432; do sleep 2; done']
  containers:
  - name: app
    image: my-app:latest

A sidecar is a secondary container that runs alongside the main app container in the same pod, sharing its network and storage. It handles cross-cutting concerns without modifying the app.

  • Logging: Fluentd/Logstash sidecar ships logs to Elasticsearch

  • Secret sync: Vault Agent sidecar injects secrets into shared volume
  • Monitoring: Prometheus exporter sidecar exposes metrics
  • K8s 1.29+: Native sidecar containers (init containers with restartPolicy: Always) run alongside app containers

HPA automatically scales the number of pod replicas based on observed CPU/memory utilization or custom metrics — scaling out under load and in during quiet periods.

  • Requires metrics-server: HPA reads CPU/memory from the metrics-server
  • Scale-up: Fast (every 15s by default)
  • Scale-down: Conservative (stabilization window: 5 min default) to prevent thrashing
  • Custom metrics: Scale on Prometheus metrics, SQS queue depth, RPS via Prometheus Adapter or KEDA
kubectl autoscale deployment my-app \
  --min=2 --max=20 --cpu-percent=60

VPA automatically adjusts CPU and memory requests/limits for containers based on historical usage — right-sizing pods to improve resource efficiency without over-provisioning.

Cluster Autoscaler automatically adjusts the number of nodes in a cluster — adding nodes when pods can't be scheduled due to insufficient resources, removing underutilized nodes.

  • Scale up: Triggered when pods are in Pending state due to insufficient resources
  • Scale down: Removes nodes where all pods can be rescheduled elsewhere and utilization is low (<50%)
  • Cloud-aware: Integrates with AWS Auto Scaling Groups, GCP MIGs, Azure VMSS
  • HPA + CA: Together handle both pod-level and node-level scaling — the full autoscaling stack

Rolling update is the default Deployment strategy — it incrementally replaces old pods with new ones, maintaining availability throughout. Controlled by maxUnavailable and maxSurge.

spec:
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 1    # or 25% - pods that can go down
      maxSurge: 1          # or 25% - extra pods above desired
  • Zero downtime: maxUnavailable=0, maxSurge=1 → always full capacity during update
  • Readiness gates: New pod must pass readiness probe before old pod is removed
  • minReadySeconds: Wait this many seconds after pod is ready before proceeding

The Recreate strategy kills all existing pods before creating new ones. It causes downtime but guarantees that only one version is running at a time.

  • Downtime: Gap between all-old-pods-gone and new-pods-ready
  • Use when: App can't run two versions simultaneously (shared DB schema, singleton services)
  • Faster: No surge overhead — simpler update process
spec:
  strategy:
    type: Recreate  # Kill all old, then start all new

A canary deployment sends a small percentage of traffic to a new version while the majority stays on the stable version. It validates the new version in production with limited blast radius.

  • Example: Deploy new version as separate Deployment with 1 replica alongside 9 replicas of old version = 10% canary traffic
  • Ingress canary: nginx-ingress canary annotations split traffic by weight or headers
  • Argo Rollouts: Dedicated controller for progressive delivery with canary support
  • Metric monitoring: Watch error rates on canary before promoting to 100%

Blue-green deployment runs two identical environments (blue=current, green=new). Traffic is switched atomically from blue to green — instant cutover with zero downtime and easy rollback.

RBAC (Role-Based Access Control) controls who can perform which actions on which Kubernetes resources. It uses Roles, ClusterRoles, RoleBindings, and ClusterRoleBindings to grant permissions.

  • Subjects: Users, Groups, ServiceAccounts

  • Resources: pods, deployments, secrets, configmaps, etc.
  • Least privilege: Grant only what's needed — default deny model

Taints repel pods from nodes unless the pod has a matching toleration. They're used to reserve nodes for specific workloads or mark nodes as unschedulable for regular pods.

Node affinity allows you to attract pods to specific nodes based on node labels — like nodeSelector but with more expressive rules including soft preferences.

          values: [us-east-1a, us-east-1b]

When a node fails, the Node Controller marks it NotReady. After a grace period (default 5 minutes), pods on that node are evicted and rescheduled on healthy nodes by their controllers.

  • Node Controller: Monitors node heartbeats via kubelet; marks NotReady after 40s no response
  • Pod eviction: After node-monitor-grace-period (default 40s) → pods get Unknown status
  • Rescheduling: After pod-eviction-timeout (default 5min) → pods evicted and recreated by Deployment/RS
  • StatefulSets: Don't auto-reschedule without explicit deletion — to prevent split-brain

Pod preemption allows the scheduler to evict lower-priority pods to make room for higher-priority pods that can't be scheduled due to resource constraints.

  • PriorityClass: Assign a numeric priority to pods — higher = more important
  • Flow: High-priority pod pending → scheduler finds node → evicts lower-priority pods → high-priority pod scheduled
  • Graceful eviction: Evicted pods get graceful termination period
  • System classes: system-cluster-critical, system-node-critical (built-in)

Finalizers are keys in a resource's metadata that must be cleared before the resource can be fully deleted. They allow controllers to perform cleanup before the object is removed from etcd.

  • Mechanism: When you delete an object with finalizers, K8s sets DeletionTimestamp but doesn't remove it

Kubernetes networking is built on three layers: the CNI plugin provides pod-to-pod connectivity across nodes, kube-proxy implements Service routing via iptables or IPVS, and CoreDNS provides service discovery.

  • CNI (Container Network Interface): Plugin spec that grants each pod a unique IP. When a pod starts, kubelet calls the CNI plugin (Calico/Cilium/Flannel) which allocates IP and configures routes. Pod IPs are routable across all nodes.
  • iptables mode: kube-proxy watches Service/Endpoint changes and creates DNAT iptables rules. Traffic to ClusterIP is randomly forwarded to one pod IP. At scale (10k+ services), iptables has O(n) lookup overhead.
  • IPVS mode: Uses kernel-level load balancing (hash tables = O(1) lookup). Supports multiple algorithms: rr, lc, dh, sh, sed, nq. Better performance at scale.
  • eBPF (Cilium): Bypasses iptables entirely — programs kernel directly for even better performance, with observability and security built in.
  • Pod-to-pod flow: Pod A → veth pair → bridge (cbr0/cni0) → routing table → node B's interface → veth → Pod B

The scheduler uses a plugin-based framework with two phases: Filter (find eligible nodes) and Score (rank them). It's extensible via scheduler plugins and the scheduling framework API.

  • Filter plugins: NodeResourcesFit, NodeAffinity, TaintToleration, VolumeBinding, NodePorts, PodTopologySpread, InterPodAffinity
  • Score plugins: LeastAllocated, BalancedAllocation, NodeAffinity, ImageLocality, InterPodAffinity, NodeResourcesFit
  • Scheduling queue: activeQ (ready to schedule), backoffQ (retrying), unschedulableQ (no eligible node)
  • Binding: After scoring, scheduler calls API server to set pod.spec.nodeName — kubelet picks it up
  • Scheduling profiles: Multiple profiles with different plugins for different workload types

The watch mechanism allows clients to receive real-time streaming updates for resource changes (ADDED, MODIFIED, DELETED events) via long-lived HTTP chunked connections to the API server.

  • List-Watch pattern: Controller first Lists all resources (gets current state + resourceVersion), then Watches from that resourceVersion for incremental changes
  • Informers: Client-go library abstracts List-Watch with local caching (in-memory store) — controllers read from local cache, not directly from API
  • Work queue: Watch events enqueue keys into a rate-limited work queue → controller dequeues and reconciles
  • Resync: Periodic re-sync to handle missed events and drift correction
  • ResourceVersion: etcd change counter used to resume a watch after disconnection without re-listing

etcd uses the Raft consensus algorithm to provide strong consistency (linearizability). All writes go through the leader; reads can be served from followers (with possible stale data) or strongly consistent from leader.

  • Raft quorum: Writes need majority acknowledgment (quorum = floor(n/2)+1). 3-node: need 2; 5-node: need 3
  • Leader election: Takes 150-300ms. During this time, API server is unavailable for writes
  • Linearizable reads: K8s uses linearizable reads by default — goes through leader for guaranteed freshness
  • Network partition: Minority partition stops accepting writes (no quorum) — cluster availability vs consistency tradeoff (CP in CAP theorem)
  • etcd size: Keep etcd DB under 8GB; set --quota-backend-bytes appropriately; compact old revisions

Kubernetes garbage collection automatically deletes objects when their owner is deleted. Owner references create a dependency graph; finalizers allow cleanup before deletion.

  • Owner references: Child objects (ReplicaSet pods, Deployment ReplicaSets) have ownerReferences pointing to their controller
  • Foreground deletion: Owner waits until all dependents are deleted before being removed (DeletionTimestamp set first)
  • Background deletion (default): Owner deleted immediately; GC controller removes dependents asynchronously
  • Orphan: Delete owner but keep children (--cascade=orphan)
  • Image GC: kubelet garbage collects unused container images when disk usage exceeds threshold (default 85%)

The kubelet monitors node resources (memory, disk, inodes) against eviction thresholds. When thresholds are crossed, it evicts pods in order of QoS class and priority to reclaim resources.

  • Eviction order: BestEffort (no requests/limits) → Burstable (requests < limits) → Guaranteed (requests = limits)
  • Soft eviction: Threshold exceeded for eviction-soft-grace-period → warning, then evict
  • Hard eviction: Threshold exceeded → immediate eviction (e.g., memory.available < 100Mi)
  • Within same QoS: Pods exceeding requests evicted first; among equals, highest consumer evicted first
  • Node pressure taints: kubelet adds node.kubernetes.io/memory-pressure taint to prevent new scheduling

Zero-downtime rolling updates depend on: readiness probes ensuring new pods are ready before old ones are removed, preStop hooks for graceful shutdown, and proper terminationGracePeriodSeconds.

  • Step 1: New ReplicaSet created with new pod template; old RS still running
  • Step 2: New pod created, passes readiness probe → added to Service endpoints
  • Step 3: Old pod removed from Service endpoints (readiness probe removed from LB)
  • Step 4: preStop hook runs on old pod (sleep/drain), then SIGTERM sent, then SIGKILL after grace period
  • Critical: maxUnavailable=0 ensures no capacity loss; readiness probe must only pass when app can serve traffic
  • minReadySeconds: Wait after pod is ready before marking it "available" — allows observing stability

When a liveness probe fails failureThreshold times consecutively, kubelet kills the container and restarts it according to the pod's restartPolicy.

  • Container is restarted: Not the whole pod — just the failing container

Existing pods KEEP RUNNING — kubelet runs them independently. But no new pods can be scheduled, no changes can be made, and no self-healing occurs until etcd recovers.

  • Running pods unaffected: kubelet doesn't need etcd to keep containers running — it just executes the local pod spec
  • API server fails: kube-apiserver can't read/write state → kubectl commands fail

Empty logs for a running pod usually means the app is writing to a file instead of stdout/stderr, the app hasn't started yet, or log rotation has truncated the log file.
