Docker is an open-source platform that automates deployment of applications inside lightweight, portable containers.

  • Consistency: Run the same container on dev, staging, and production without "works on my machine" issues.
  • Isolation: Each container runs in its own isolated environment with its own filesystem, network, and processes.
  • Portability: Package an app and all its dependencies into a single image that runs anywhere Docker is installed.
  • Efficiency: Containers share the host OS kernel, making them much lighter than virtual machines.
  • Speed: Containers start in milliseconds compared to minutes for VMs, enabling rapid CI/CD pipelines.

A container is a runnable instance of a Docker image — an isolated process with its own filesystem, network, and process space.

  • Process isolation: Uses Linux namespaces to isolate PID, network, mount, UTS, IPC, and user spaces.
  • Layered FS: Has a thin writable layer on top of read-only image layers (copy-on-write).
  • Ephemeral by default: Data written inside a container is lost when the container is removed unless volumes are used.
  • Lightweight: Shares the host kernel; no hypervisor or full OS needed per container.
  • Stateful or stateless: Can be either — stateless containers are easiest to scale horizontally.

A Docker image is a read-only, layered blueprint used to create containers. It packages the application code, runtime, libraries, and config.

  • Immutable: Images cannot be modified once built — you create a new image to make changes.
  • Layered: Built from stacked layers — each Dockerfile instruction creates a new layer.
  • Sharable: Pushed to and pulled from registries like Docker Hub, ECR, or GCR.
  • Tagged: Identified by name:tag e.g. nginx:1.25 or myapp:latest.
  • Content-addressed: Each image and layer has a SHA256 digest ensuring integrity.

A Dockerfile is a text file containing a series of instructions that Docker reads to automatically build an image.

  • Sequential: Instructions are executed top-to-bottom, each creating a new image layer.
  • Declarative: Describes the desired state of the image — base OS, dependencies, files, entrypoint.
  • Reproducible: Same Dockerfile always produces the same image (assuming deterministic dependencies).
  • Version controlled: Stored in source control alongside application code — infrastructure as code.
FROM node:20-alpine
WORKDIR /app
COPY package*.json ./
RUN npm install
COPY . .
EXPOSE 3000
CMD ["node", "server.js"]

Docker Hub is the default public registry for Docker images — a cloud-based service for storing, sharing, and distributing container images.

  • Official images: Curated images for popular software (nginx, postgres, node, redis) maintained by Docker and vendors.
  • Public & private repos: Free public repos; private repos available on paid plans.
  • Automated builds: Can trigger image builds from GitHub/Bitbucket commits.
  • Rate limiting: Free tier has pull rate limits (100 pulls/6h anonymous, 200/6h authenticated).
  • Alternatives: AWS ECR, GCR, GHCR, GitLab Registry, Harbor for private or self-hosted needs.

docker run creates and starts a new container from an image in a single command.

  • Pulls if needed: Automatically pulls the image from the registry if not found locally.
  • Creates writable layer: Adds a thin writable layer on top of the image layers.
  • Common flags: -d detach, -p port map, -v volume, -e env var, --name name.
  • Combines: Equivalent to docker create + docker start.
docker run -d -p 8080:80 --name myapp nginx:latest
docker run -it ubuntu:22.04 bash          # interactive
docker run --rm alpine echo "hello"       # auto-remove after exit

Use docker ps to list running containers, docker ps -a to list all including stopped.

  • docker ps — shows CONTAINER ID, IMAGE, COMMAND, CREATED, STATUS, PORTS, NAMES.
  • docker ps -a — all containers including exited/stopped.
  • docker ps -q — quiet mode, only shows container IDs (useful in scripts).
  • docker ps --filter status=exited — filter by status.
docker ps                          # running containers
docker ps -a                       # all containers
docker ps -a --format "table {{.Names}}\t{{.Status}}"

Use docker stop <container> to gracefully stop a container by sending SIGTERM then SIGKILL after a timeout.

  • Graceful stop: Sends SIGTERM, waits 10s (default), then sends SIGKILL if not stopped.
  • Custom timeout: docker stop -t 30 myapp waits 30 seconds before killing.
  • Multiple: Stop several at once with docker stop c1 c2 c3.
  • Stop all: docker stop $(docker ps -q) stops all running containers.
docker stop myapp
docker stop -t 30 myapp            # 30s grace period
docker stop $(docker ps -q)        # stop all running

Use docker rm <container> to remove a stopped container. Use -f to force-remove a running one.

  • docker rm myapp — removes a stopped container.
  • docker rm -f myapp — force removes even if running (sends SIGKILL).
  • docker rm $(docker ps -aq) — remove all stopped containers.
  • docker run --rm — automatically removes container when it exits.
  • docker container prune — removes all stopped containers at once.
docker rm myapp
docker rm -f myapp
docker container prune             # remove all stopped containers

FROM sets the base image for the build — it must be the first instruction in a Dockerfile.

  • Required: Every Dockerfile must start with FROM (except ARG before FROM).
  • Syntax: FROM image:tag or FROM image:tag AS alias for multi-stage.
  • Scratch: FROM scratch creates an image from nothing — used for minimal images.
  • Multiple FROM: Multi-stage builds use multiple FROM instructions with AS names.
FROM ubuntu:22.04
FROM node:20-alpine AS builder
FROM scratch                       # empty base for minimal images

RUN executes a command during the image build process, creating a new layer with the results.

  • Build-time only: Runs during docker build, not when container starts.
  • Shell form: RUN apt-get update — uses /bin/sh -c by default.
  • Exec form: RUN ["apt-get", "update"] — no shell, direct execution.
  • Combine commands: Chain with && to reduce layers and keep image smaller.
RUN apt-get update && apt-get install -y curl && rm -rf /var/lib/apt/lists/*
RUN ["pip", "install", "-r", "requirements.txt"]

CMD specifies the default command to run when a container starts — it can be overridden at runtime.

  • Runtime default: Provides a default command, not executed during build.
  • Overridable: Easily overridden by appending a command to docker run.
  • Only one CMD: If multiple CMD instructions exist, only the last one takes effect.
  • Exec form preferred: CMD ["node", "app.js"] — avoids shell signal handling issues.
CMD ["nginx", "-g", "daemon off;"]
CMD ["python", "app.py"]
# Overridden at runtime:
docker run myapp python manage.py shell

EXPOSE documents which ports the container listens on — it is informational only and does not actually publish ports.

  • Documentation only: Does not make the port accessible on the host — you still need -p at runtime.
  • Protocol: Can specify TCP or UDP: EXPOSE 53/udp, defaults to TCP.
  • Inter-container: Useful for Docker Compose / Swarm to know which ports to route between services.
  • docker run -P: The -P flag automatically publishes all EXPOSEd ports to random host ports.
EXPOSE 80
EXPOSE 443
EXPOSE 5432/tcp
docker run -P nginx    # auto-publish all exposed ports

COPY copies files or directories from the build context (host) into the container image filesystem.

  • Build context: Source path is relative to the build context directory.
  • Syntax: COPY <src> <dest> — dest is an absolute or relative path inside the image.
  • Wildcards: Supports glob patterns like COPY *.json /app/.
  • --chown: COPY --chown=user:group files/ /app/ sets ownership without extra RUN layer.
  • Multi-stage: COPY --from=builder /app/dist ./dist copies from another stage.
COPY package.json /app/
COPY --chown=node:node . /app
COPY --from=builder /app/dist /usr/share/nginx/html

Use docker build with a path to the build context — Docker reads the Dockerfile and creates the image layer by layer.

  • Basic: docker build -t myapp:1.0 . — builds from current directory.
  • Custom Dockerfile: docker build -f Dockerfile.prod -t myapp:prod .
  • Build args: Pass build-time variables with --build-arg KEY=value.
  • No cache: --no-cache forces rebuild of all layers.
  • Context: Everything in the build context directory is sent to the daemon — use .dockerignore to limit it.
docker build -t myapp:latest .
docker build -t myapp:1.0 -f Dockerfile.prod .
docker build --build-arg NODE_ENV=production -t myapp .

Use docker pull <image:tag> to download an image from a registry to local storage.

  • Default tag: docker pull nginx pulls nginx:latest by default.
  • Specific version: docker pull postgres:16.2 pulls a specific version.
  • Private registry: docker pull myregistry.com/myapp:1.0 pulls from a custom registry.
  • All tags: docker pull -a nginx pulls all available tags.
docker pull nginx
docker pull postgres:16.2
docker pull myregistry.azurecr.io/myapp:v2.0

Tag your image with your Docker Hub username, then use docker push to upload it to the registry.

  • Login first: docker login authenticates with Docker Hub credentials.
  • Tag correctly: Image must be tagged as username/repo:tag to push to Docker Hub.
  • Push: docker push username/myapp:1.0 uploads all new layers.
  • Layers only: Only pushes layers not already in the registry (deduplication).
docker login
docker tag myapp:latest username/myapp:1.0
docker push username/myapp:1.0

docker ps lists containers. By default shows only running containers; -a shows all including stopped.

  • Columns: CONTAINER ID, IMAGE, COMMAND, CREATED, STATUS, PORTS, NAMES.
  • -a / --all — include stopped containers.
  • -q / --quiet — only display container IDs.
  • --format — customize output with Go templates.
  • --filter — filter by name, status, label, ancestor.
docker ps
docker ps -a
docker ps -q
docker ps --format "table {{.Names}}\t{{.Image}}\t{{.Status}}"
docker ps --filter "ancestor=nginx"

docker logs fetches the stdout/stderr output of a container — the primary way to inspect container output.

  • -f / --follow — stream logs in real-time (like tail -f).
  • --tail N — show last N lines only.
  • --since / --until — filter logs by timestamp.
  • -t / --timestamps — prepend timestamps to each log line.
docker logs myapp
docker logs -f myapp               # follow/stream
docker logs --tail 100 myapp       # last 100 lines
docker logs --since 30m myapp      # last 30 minutes

docker exec runs a command inside an already-running container — useful for debugging and inspection.

  • Interactive shell: docker exec -it myapp bash opens a shell inside the container.
  • Non-interactive: Run a one-off command: docker exec myapp cat /etc/hosts.
  • -i — keep STDIN open; -t — allocate a pseudo-TTY.
  • Environment: -e VAR=value sets env vars for the exec'd process only.
docker exec -it myapp bash
docker exec myapp ls /app
docker exec -e DEBUG=true myapp python debug.py

The -d flag runs the container in detached mode (background) — the container runs independently of the terminal.

  • Detached: Container runs in the background; you get the container ID printed and your terminal is free.
  • Long-running services: Use -d for servers, databases, and daemons.
  • View output: Use docker logs -f <id> to see output from a detached container.
  • Opposite: Without -d, the container runs in the foreground and Ctrl+C stops it.
docker run -d nginx                # detached, runs in background
docker run nginx                   # foreground, blocks terminal
docker attach myapp                # re-attach to a detached container

The -p flag publishes a container port to the host — maps hostPort:containerPort.

  • Syntax: -p hostPort:containerPort e.g. -p 8080:80 maps host 8080 to container 80.
  • Specific IP: -p 127.0.0.1:8080:80 binds only to localhost on the host.
  • Multiple ports: Use multiple -p flags: -p 80:80 -p 443:443.
  • Random host port: -p 80 (without host port) assigns a random available host port.
  • -P flag: Publishes ALL EXPOSEd ports to random host ports.
docker run -p 8080:80 nginx        # host:container
docker run -p 127.0.0.1:8080:80 nginx
docker run -p 80:80 -p 443:443 nginx

The -v flag mounts a volume or bind mount — connects host filesystem or a named volume to a container path.

  • Named volume: -v mydata:/app/data — Docker manages storage location.
  • Bind mount: -v /host/path:/container/path — mounts a specific host directory.
  • Read-only: -v mydata:/app/data:ro — mounts as read-only inside container.
  • Modern syntax: --mount flag is more explicit and preferred over -v.
docker run -v myvolume:/app/data nginx
docker run -v $(pwd):/app node:20        # bind mount
docker run -v /etc/config:/config:ro nginx  # read-only

docker inspect returns detailed low-level JSON information about any Docker object — container, image, volume, or network.

  • Full details: Returns JSON with IP address, mounts, env vars, state, networking config.
  • Format flag: --format extracts specific fields using Go templates.
  • Any object: Works on containers, images, networks, volumes, nodes, services.
  • IP address: Common use: extract container IP with --format '{{.NetworkSettings.IPAddress}}'.
docker inspect myapp
docker inspect --format '{{.NetworkSettings.IPAddress}}' myapp
docker inspect --format '{{.State.Status}}' myapp

docker images lists all locally stored images with repository, tag, image ID, creation date, and size.

  • docker images or docker image ls — both work the same.
  • -a — show all images including intermediate layers.
  • -q — quiet mode, show only image IDs.
  • --filter dangling=true — show only untagged/dangling images.
docker images
docker images nginx
docker images -q
docker image ls --format "table {{.Repository}}\t{{.Tag}}\t{{.Size}}"

Use docker rmi <image> to remove a local image. Cannot remove an image with running containers.

  • docker rmi nginx:latest — removes by name:tag.
  • -f — force remove even if tagged or has stopped containers.
  • docker image prune — removes all dangling images.
  • docker image prune -a — removes all unused images.
docker rmi nginx:latest
docker rmi -f $(docker images -q)
docker image prune -a

A Docker registry is a storage and distribution system for Docker images — where images are pushed to and pulled from.

  • Public: Docker Hub (default), GitHub Container Registry (ghcr.io), Quay.io.
  • Cloud: AWS ECR, GCP Artifact Registry, Azure Container Registry.
  • Self-hosted: Docker Registry, Harbor, Nexus, GitLab Registry.
  • Structure: Registry → Repositories → Tags (each tag points to an image manifest).

Docker Compose is a tool for defining and running multi-container applications using a YAML file (docker-compose.yml).

  • Declarative: Define all services, networks, and volumes in one YAML file.
  • Single command: docker compose up starts all defined services.
  • Networking: Services auto-discover each other by service name on a shared network.
  • v2 vs v1: Modern Docker uses docker compose (plugin); older used docker-compose (Python).

COPY simply copies files; ADD auto-extracts archives and supports remote URLs. Prefer COPY for clarity.

COPY (preferred)
  • Copies files/dirs from context
  • Simple, predictable behavior
  • Supports --from for multi-stage
  • Recommended for most cases
ADD (special cases)
  • All COPY features plus:
  • Auto-extracts .tar archives
  • Supports remote URLs
  • Less predictable — avoid unless needed

ENTRYPOINT sets the fixed executable that always runs when a container starts — not overridden by docker run arguments.

  • Always executes: Cannot be bypassed by passing a command at docker run.
  • Exec form preferred: ENTRYPOINT ["executable", "param"] for proper signal handling.
  • CMD as args: CMD provides default arguments to ENTRYPOINT when both are used.
  • Override: Only via docker run --entrypoint /bin/sh myimage.
ENTRYPOINT ["python", "app.py"]
ENTRYPOINT ["nginx"]
CMD ["-g", "daemon off;"]   # CMD args passed to ENTRYPOINT

ENTRYPOINT is the fixed executable; CMD provides overridable default arguments or command.

CMD
  • Default command or arguments
  • Easily overridden at docker run
  • Acts as args if ENTRYPOINT set
  • Use for flexible containers
ENTRYPOINT
  • Fixed executable for the container
  • Not overridden by run-time args
  • Requires --entrypoint to override
  • Use when container has one purpose

A Docker volume is Docker-managed persistent storage that survives container removal and can be shared between containers.

  • Persistent: Data survives container stop, restart, and removal.
  • Docker-managed: Stored at /var/lib/docker/volumes/ on the host.
  • Shareable: Multiple containers can mount the same volume simultaneously.
  • Best for: Database data, app state, shared config between containers.
docker volume create mydata
docker run -v mydata:/app/data myapp
docker volume ls && docker volume inspect mydata

A bind mount maps a specific host directory or file into a container — the host path is directly accessible inside the container.

  • Host-controlled: You specify the exact host path; Docker doesn't manage it.
  • Dev workflow: Mount source code so container sees live changes without rebuilding.
  • Performance: Near-native on Linux; macOS/Windows has overhead.
  • Security: Container can modify host files — mount carefully.
docker run -v $(pwd):/app node:20
docker run --mount type=bind,source=$(pwd),target=/app node:20

Docker networks enable containers to communicate with each other and the outside world, each providing isolation and connectivity.

  • Drivers: bridge, host, none, overlay, macvlan, ipvlan.
  • Create: docker network create mynet creates a user-defined bridge.
  • DNS: Containers on user-defined networks resolve each other by container name.
  • Isolation: Containers on different networks cannot communicate by default.
docker network create mynet
docker run --network mynet --name api myapp
docker network inspect mynet

The default network is the bridge network — all containers connect to it unless specified otherwise, but it lacks automatic DNS by container name.

  • bridge (default): Virtual bridge docker0; containers get IPs in 172.17.0.0/16.
  • No auto-DNS: Default bridge doesn't resolve container names — use user-defined bridges.
  • host: Container shares host network stack directly — no NAT overhead.
  • none: Container has no network access whatsoever.

docker tag creates a new tag reference to an existing image — it does not copy the image, just adds an alias pointing to the same layers.

  • Alias: Source and target tags point to the same underlying image layers.
  • Registry prep: Retag before pushing to a specific registry or account.
  • Versioning: Tag the same image as both v1.2.3 and latest.
  • Syntax: docker tag SOURCE[:TAG] TARGET[:TAG].
docker tag myapp:latest myapp:1.0.0
docker tag myapp:latest myregistry.com/myapp:latest

A multi-stage build uses multiple FROM instructions — earlier stages compile/build, and only the needed artifacts are copied to the final lean image.

  • Reduces size: Final image contains only runtime artifacts, not compilers or source.
  • Named stages: FROM golang:1.21 AS builder — reference later with COPY --from.
  • Security: No build secrets, credentials, or source code in production image.
  • COPY --from: COPY --from=builder /app/bin /app/bin
FROM node:20 AS builder
WORKDIR /app
RUN npm ci && npm run build

FROM nginx:alpine
COPY --from=builder /app/dist /usr/share/nginx/html

WORKDIR sets the working directory for all subsequent RUN, CMD, ENTRYPOINT, COPY, and ADD instructions.

  • Auto-creates: Creates the directory if it doesn't exist.
  • Absolute paths: Always use absolute paths for clarity.
  • Runtime: Also sets default directory when exec'ing into container.

ENV sets persistent environment variables available at both build time and runtime in the container.

  • Persistent: Available in all subsequent build steps and in running containers.
  • Override at runtime: docker run -e KEY=value overrides the value.
  • Syntax: ENV KEY=value (modern preferred form).
  • vs ARG: ENV persists to runtime; ARG is build-time only.
ENV NODE_ENV=production
ENV PORT=3000 HOST=0.0.0.0
docker run -e NODE_ENV=development myapp

ARG defines build-time variables passed via --build-arg — they are NOT available in running containers.

  • Build-time only: Available only during docker build, not at runtime.
  • Default values: ARG VERSION=latest provides a fallback.
  • Not for secrets: ARG values appear in image history — never use for passwords.
  • Promote to ENV: ENV VERSION=$VERSION makes it a runtime variable.
ARG NODE_VERSION=20
FROM node:${NODE_VERSION}-alpine
ARG BUILD_DATE
LABEL build-date=$BUILD_DATE

docker system prune removes all stopped containers, unused networks, dangling images, and build cache in one command.

  • Removes: Stopped containers + dangling images + unused networks + build cache.
  • -a — also removes all unused images (not just dangling).
  • --volumes — also prunes unused volumes (data loss risk!).
  • -f — skips confirmation prompt.
docker system prune
docker system prune -a
docker system prune -a -f --volumes

  • Sends SIGKILL if not stopped

Use docker stats for a live view of CPU, memory, network I/O, and block I/O for running containers.

  • docker stats — real-time stats for all running containers.
  • docker stats myapp — stats for a specific container.
  • --no-stream — snapshot mode, print once and exit.

docker commit creates a new image from a container's current state — capturing all filesystem changes made to it.

  • Snapshot: Creates an image from running or stopped container's state.
  • Anti-pattern: Discouraged — prefer Dockerfiles for reproducibility.
  • Debug use: Useful for capturing state after manual exploration.
  • Opaque: Creates images with no build history — hard to maintain.

LABEL adds metadata key-value pairs to an image — useful for versioning, documentation, and automation filtering.

  • Metadata: Store maintainer, version, description, build date.
  • Filter: docker images --filter "label=version=1.0" finds images by label.

LABEL maintainer="team@example.com"

HEALTHCHECK defines a command Docker runs periodically to test container health — results in starting/healthy/unhealthy status.

  • Status: starting → healthy or unhealthy.
  • Options: --interval, --timeout, --retries, --start-period.
  • Orchestration: Swarm/Kubernetes use health to route traffic and restart containers.
  • Exit code: 0 = healthy, 1 = unhealthy.
HEALTHCHECK --interval=30s --timeout=5s --retries=3 \
  CMD curl -f http://localhost/health || exit 1

docker info displays system-wide information about the Docker installation including daemon config, resources, and runtime details.

  • Containers: Count of running, paused, stopped containers.
  • Images: Total number of locally stored images.
  • System: OS, kernel, CPU count, total memory, Docker version.
  • Config: Storage driver, logging driver, cgroup driver, swarm status, plugins.
docker info
docker info --format '{{.MemTotal}}'

Docker uses a Union Filesystem (UnionFS) that stacks read-only image layers with a thin writable container layer on top using copy-on-write semantics.

  • Read-only layers: Each Dockerfile instruction creates an immutable layer stored and reused across images.
  • Writable layer: Container has one thin writable layer — changes are isolated to this layer.
  • Copy-on-write: Reading a file reads from the lower layer; writing copies the file up to the writable layer first.
  • overlay2: Default storage driver on modern Linux — uses kernel overlay filesystem for efficiency.
  • Layer sharing: Multiple containers based on the same image share its read-only layers — saves disk space.

Linux namespaces provide the isolation that makes containers possible — each container gets its own view of system resources.

  • PID namespace: Container has its own process tree; container PID 1 maps to a different host PID.
  • NET namespace: Own network stack — interfaces, IP addresses, routing tables, firewall rules.
  • MNT namespace: Own mount points and filesystem view.
  • UTS namespace: Own hostname and domain name.
  • IPC namespace: Isolated inter-process communication (shared memory, semaphores).

Control groups (cgroups) limit and account for resource usage — Docker uses them to enforce CPU, memory, and I/O limits on containers.

  • Resource limits: Set max CPU, memory, swap, block I/O, network bandwidth per container.
  • Memory limit: docker run --memory=512m caps container RAM.
  • CPU limit: docker run --cpus=1.5 limits to 1.5 CPU cores.
  • OOM killer: When container exceeds memory limit, kernel OOM killer terminates processes.
  • cgroups v2: Modern unified hierarchy — better resource tracking; default in newer Linux distros.

Docker offers several networking modes, each providing different levels of isolation and connectivity between container and host.

A bridge network is a software bridge that allows containers connected to the same bridge to communicate while being isolated from other networks.

  • Default bridge: docker0 — all containers attach to it by default; no DNS by name.
  • User-defined bridge: Created with docker network create — supports automatic DNS by container name.
  • NAT: Outbound traffic is NATted to host IP; inbound requires port publishing.
  • Isolation: Containers on different bridge networks cannot communicate directly.
docker network create --driver bridge mybridge
docker run --network mybridge --name web nginx
docker run --network mybridge --name api myapp  # api can reach 'web'

The host network mode removes network isolation — the container shares the host's network stack directly with no NAT overhead.

  • No port mapping needed: Container ports are directly exposed on the host — -p flag is unnecessary.
  • Performance: Best network performance — zero NAT overhead, useful for high-throughput workloads.
  • Port conflicts: Container and host share port space — port collisions are possible.
  • Linux only: Host networking on Mac/Windows routes through a VM — not truly the host network.
docker run --network host nginx
# nginx now listens on host port 80 directly

An overlay network spans multiple Docker hosts — it enables containers on different hosts to communicate securely as if on the same LAN.

  • Multi-host: Required for Docker Swarm to connect containers across nodes.
  • VXLAN: Uses VXLAN encapsulation to tunnel container traffic over the host network.
  • Encrypted: Optional encryption with --opt encrypted for secure inter-host traffic.
  • Service discovery: Built-in DNS for Swarm services on overlay networks.
docker network create --driver overlay --attachable myoverlay
docker service create --network myoverlay myapp

Containers on the same user-defined network communicate by container name (DNS) or IP; they cannot reach containers on different networks without explicit connection.

Docker caches each build layer — if nothing in a layer changed since the last build, Docker reuses the cached version, skipping re-execution.

  • Layer cache key: Based on instruction text + parent layer hash + file contents (for COPY/ADD).
  • Invalidation cascade: Changing any layer invalidates all subsequent layers — order matters.
  • Best practice: Copy dependency files first, install deps, then copy source code — so dep install is cached.
  • --no-cache: Force rebuild all layers.
# Good cache ordering:
COPY package.json package-lock.json ./
RUN npm install          # cached until package.json changes
COPY . .                 # source changes don't bust dep cache

A dangling image is an untagged image — it appears as <none>:<none> and occurs when a new build replaces an existing tag.

  • Cause: Rebuilding an image with the same tag detaches the old image from that tag, making it dangling.
  • Space waste: Dangling images consume disk space without serving any active container.
  • List: docker images -f dangling=true shows all dangling images.
  • Remove: docker image prune removes all dangling images.
docker images -f dangling=true

docker image prune -a         # remove ALL unused images

  • Throttling: Unlike memory, CPU excess is throttled, not killed.

overlay2 implements Union Filesystem by merging multiple directory layers into a single unified view — using Linux kernel's OverlayFS for efficient copy-on-write.

  • OverlayFS layers: lowerdir (read-only image layers) + upperdir (writable container layer) + workdir merged into merged.
  • CoW on write: When a container modifies a file, it's copied from lowerdir to upperdir first — only the diff is stored.
  • Efficient reads: Reading unmodified files reads directly from lowerdir — no copy needed.
  • Layers on disk: Stored at /var/lib/docker/overlay2/ — each layer is a directory with a diff/ subdirectory.
  • overlay2 vs devicemapper: overlay2 is faster and more memory-efficient; devicemapper (legacy) used block devices.

Linux namespaces partition kernel resources — each container gets its own namespace instances, creating the illusion of an isolated system without a separate kernel.

  • PID namespace: Container process tree starts from PID 1; can't see host processes. Host sees the actual PID.
  • NET namespace: Isolated network stack — own lo, eth0, routing tables, iptables, sockets.
  • MNT namespace: Private mount table — container sees only its own filesystem mounts.
  • USER namespace: Map container UIDs to different host UIDs — rootless containers use this.
  • cgroup namespace: Container sees its own cgroup hierarchy root — hides host cgroup paths.

cgroups v2 introduces a unified hierarchy replacing v1's multiple controllers — providing better resource tracking, PSI pressure, and rootless container support.

cgroups v1
  • Multiple controller hierarchies
  • Separate trees for cpu, memory, etc.
  • No memory+swap atomicity
  • No pressure stall info (PSI)
cgroups v2
  • Single unified hierarchy
  • Better memory accounting
  • PSI pressure stall metrics
  • Enables rootless containers

seccomp (Secure Computing Mode) filters system calls a container can make — Docker applies a default profile that blocks ~44 dangerous syscalls to reduce attack surface.

  • Default profile: Docker's default seccomp profile allows ~300 of ~400 syscalls — blocking dangerous ones like keyctl, ptrace, reboot.
  • Custom profile: JSON file specifying allowed/denied syscalls — pass with --security-opt seccomp=profile.json.
  • Disable: --security-opt seccomp=unconfined removes all restrictions (dangerous).
  • Actions: Per-syscall actions: SCMP_ACT_ALLOW, SCMP_ACT_ERRNO, SCMP_ACT_KILL.
docker run --security-opt seccomp=custom-profile.json myapp
docker run --security-opt seccomp=unconfined myapp  # disable (avoid)

Linux capabilities break root privileges into granular units — Docker drops most capabilities by default and allows fine-grained control over which privileged operations a container can perform.

  • Default dropped: Docker runs with a reduced set — no CAP_SYS_ADMIN, CAP_NET_ADMIN, etc. by default.
  • Add capability: docker run --cap-add NET_ADMIN myapp for containers needing network config.
  • Drop all: --cap-drop ALL --cap-add <needed> — principle of least privilege.
  • Privileged mode: --privileged grants ALL capabilities — extremely dangerous, avoid in production.
docker run --cap-drop ALL --cap-add NET_BIND_SERVICE myapp
docker run --cap-add SYS_PTRACE myapp  # for debuggers

AppArmor (Ubuntu) and SELinux (RHEL/Fedora) are Mandatory Access Control systems — Docker automatically applies profiles to containers as a defense-in-depth security layer.

  • AppArmor: Docker loads docker-default AppArmor profile for all containers on Ubuntu/Debian.
  • SELinux: Docker applies svirt_sandbox_file_t labels on RHEL/Fedora — restricts file system access.
  • Custom AppArmor: docker run --security-opt apparmor=myprofile myapp
  • Disable: --security-opt apparmor=unconfined or --security-opt label=disable for SELinux.

Rootless Docker runs the Docker daemon and containers as a non-root user — using user namespaces to map the container's root to an unprivileged host user.

  • Security benefit: Even if a container escapes, the attacker only gets a non-privileged user on the host.
  • User namespace: Container's root (UID 0) maps to a high-numbered unprivileged host UID.
  • Limitations: Some features unavailable: AppArmor, Checkpoint, overlay on older kernels, some network modes.
  • Setup: dockerd-rootless-setuptool.sh install — available since Docker 19.03.
dockerd-rootless-setuptool.sh install
export DOCKER_HOST=unix://$XDG_RUNTIME_DIR/docker.sock
docker run hello-world  # running as non-root user

BuildKit introduces cache mounts, inline cache exports, and registry cache backends — enabling persistent caches across builds and CI runs.

  • Cache mounts: RUN --mount=type=cache,target=/root/.npm npm install — persists package manager caches between builds.
  • Inline cache: --cache-from=myimage:latest uses a previously pushed image's layers as cache source.
  • Registry cache: --cache-to=type=registry,ref=registry/cache exports cache to a registry.
  • Local cache: --cache-to=type=local,dest=/tmp/buildcache stores cache on local disk.
RUN --mount=type=cache,target=/root/.m2 mvn package
docker buildx build --cache-from=type=registry,ref=myrepo/cache \
  --cache-to=type=registry,ref=myrepo/cache,mode=max .

Cache invalidation occurs when a layer's inputs change — Docker recomputes the cache key and must rebuild that layer and all subsequent ones.

  • Cache key: Hash of instruction text + parent layer digest + file contents (COPY/ADD) + build args.
  • Cascade: One invalidated layer invalidates all following layers.
  • RUN invalidation: RUN uses instruction text only — apt-get update alone won't invalidate unless the text changes.
  • Control strategy: Put stable instructions early (FROM, install deps) and volatile ones late (COPY source).
# Bad: changing source busts the npm install cache
COPY . .
RUN npm install

# Good: stable layers first
COPY package.json package-lock.json ./
RUN npm install
COPY . .

docker buildx uses QEMU emulation or cross-compilation to build images for multiple architectures simultaneously from a single Dockerfile.

  • QEMU emulation: Registers binfmt_misc handlers so foreign-arch binaries can run; slower but simple.
  • Native nodes: Add arm64 build nodes to the builder for native speed without emulation.
  • Manifest list: buildx automatically creates a multi-arch manifest list — docker pull gets the right arch.
  • Build and push: --push is required for multi-platform — local docker doesn't support multi-arch images.
docker buildx create --use --name multibuilder
docker buildx build \
  --platform linux/amd64,linux/arm64,linux/arm/v7 \
  -t myapp:latest --push .

Image Manifest v2 (schema 2) is the JSON format describing an image's layers, config, and media types — supporting fat manifests for multi-arch images.

  • Manifest: References image config blob + ordered list of layer blobs, each with digest and size.
  • Manifest list (OCI index): Contains multiple manifest references, one per platform.
  • Content-addressable: Each layer and manifest identified by SHA256 digest — immutable and verifiable.
  • OCI spec: Open Container Initiative Image Spec is largely compatible with Docker Manifest v2.
docker manifest inspect nginx:latest --verbose
# Shows: schemaVersion, mediaType, config digest, layers[]

Docker stores every image layer and manifest by its SHA256 digest — the content IS the address, enabling deduplication, integrity verification, and immutability.

  • Digest = SHA256 of content: Any change produces a completely different digest.
  • Deduplication: Identical layers shared across images — stored once regardless of how many images reference them.
  • Immutable: Cannot modify a layer without changing its digest — guarantees integrity.
  • Pull by digest: docker pull nginx@sha256:abc123... guarantees the exact image.

Docker manipulates iptables to implement port publishing, container-to-container routing, masquerading (NAT), and network isolation between bridge networks.

  • DOCKER chain: Docker adds rules to the DOCKER chain for port mappings — DNAT rules translate host port to container IP:port.
  • MASQUERADE: POSTROUTING MASQUERADE rule NATts outbound container traffic to host IP.
  • DOCKER-ISOLATION: Rules preventing containers on different networks from communicating.
  • --iptables=false: Disable Docker's iptables management — requires manual firewall setup.
iptables -t nat -L DOCKER     # view Docker NAT rules
iptables -L DOCKER-ISOLATION  # view isolation rules
# Docker adds: -A DOCKER -p tcp --dport 8080 -j DNAT --to-dest 172.17.0.2:80

macvlan assigns each container its own MAC address and IP on the physical network — making containers appear as first-class network citizens to the LAN.

  • Appears on LAN: Containers are directly reachable from the physical network without NAT or port mapping.
  • Use case: Legacy applications that expect a dedicated IP; network monitoring tools; broadcast-sensitive apps.
  • Parent interface: Requires specifying the host physical interface (eth0).
  • Promiscuous mode: NIC must support promiscuous mode; some cloud providers block this.
docker network create -d macvlan \
  --subnet=192.168.1.0/24 --gateway=192.168.1.1 \
  -o parent=eth0 macnet
docker run --network macnet --ip 192.168.1.50 nginx

ipvlan is similar to macvlan but all containers share the parent interface's MAC address — only IP addresses differ, bypassing promiscuous mode requirements.

  • L2 mode: Containers share parent MAC; switch routes by IP at L2 — works where macvlan doesn't.
  • L3 mode: Docker acts as router between container subnets — more scalable, no bridge flooding.
  • vs macvlan: ipvlan shares MAC — works in environments that restrict multiple MACs per port (AWS, Azure VMs).
  • Cloud-friendly: Better suited for cloud VMs where macvlan's promiscuous mode is blocked.

Docker overlay networks use VXLAN (UDP 4789) to encapsulate container Ethernet frames inside UDP packets — tunneling container traffic over the host network between Swarm nodes.

  • VXLAN encapsulation: Container L2 frame wrapped in VXLAN/UDP/IP — traverses any IP network.
  • VTEP: Each host has a VXLAN Tunnel Endpoint (VTEP) that handles encap/decap at the host level.
  • VNI: Each overlay network gets a unique VXLAN Network Identifier (VNI) for isolation.
  • Gossip protocol: Swarm uses gossip to distribute VTEP/container-to-host mapping information.

The ingress network is a special overlay network in Docker Swarm that implements the routing mesh — allowing any node to receive external traffic for a service, even if it's not running a replica.

  • Routing mesh: Any Swarm node accepts connections on a published port and routes to healthy service replicas.
  • VIP load balancing: Each service gets a virtual IP; Swarm's IPVS load balances requests across replicas.
  • Automatic: Created automatically when Swarm is initialized — docker network inspect ingress.
  • IPVS: Uses Linux IPVS (IP Virtual Server) for high-performance L4 load balancing.

Docker Swarm uses the Raft consensus algorithm among manager nodes to maintain a consistent, fault-tolerant view of cluster state — requiring a quorum of managers to make decisions.

  • Leader election: Raft elects one leader manager; all writes go through the leader which replicates to followers.
  • Quorum: Need (N/2)+1 managers available — with 3 managers, tolerate 1 failure; 5 managers tolerate 2.
  • Odd numbers: Always use an odd number of managers (3, 5, 7) to avoid split-brain.
  • State: All service definitions, secrets, configs, network configs stored in Raft log.

Docker Swarm rolling updates replace replicas in batches while maintaining service availability — controlled by update-parallelism, update-delay, and failure action settings.

  • Parallelism: --update-parallelism 2 updates 2 replicas at a time.
  • Delay: --update-delay 10s waits 10 seconds between batches.
  • Failure action: --update-failure-action rollback auto-rolls back on failure.
  • Rollback: docker service rollback myservice reverts to the previous version.
docker service update \
  --image myapp:2.0 \
  --update-parallelism 1 \
  --update-delay 30s \
  --update-failure-action rollback \
  myservice

Docker plugins extend the Docker Engine's capabilities — running as managed processes that implement volume, network, or authorization driver APIs via a Unix socket interface.

  • Volume plugins: Add support for external storage — NFS, AWS EFS, Azure File, GlusterFS.
  • Network plugins: Weave, Calico, Cilium — alternative network implementations.
  • Auth plugins: Intercept Docker API calls for custom authorization policies.
  • Managed lifecycle: docker plugin install, enable, disable, remove manage plugin lifecycle.

The Docker daemon exposes a REST API over a Unix socket (/var/run/docker.sock) or TCP — all Docker CLI commands are API calls to this daemon.

  • Transport: Unix socket (default, secure, local-only) or TCP (remote, requires TLS).
  • REST: HTTP REST API — e.g., POST /containers/create, GET /containers/json.
  • SDK: Official SDKs for Python (docker), Go (docker/client), and community SDKs for other languages.
  • Security risk: Access to docker.sock = root on host — protect it carefully.
curl --unix-socket /var/run/docker.sock http://localhost/containers/json
curl --unix-socket /var/run/docker.sock \

containerd is a CNCF-graduated container runtime that Docker uses internally — it handles image management, container lifecycle, snapshots, and networking at a lower level than the Docker daemon.

  • Docker's runtime: Docker Engine delegates to containerd for container execution and image management.
  • Also used by: Kubernetes uses containerd directly (bypassing Docker since k8s 1.24).
  • API: gRPC API (different from Docker's REST API).
  • Separate daemon: Runs as containerd daemon; Docker daemon is a higher-level layer on top.

runc is the OCI-compliant container runtime that actually creates and runs containers — it's the low-level tool called by containerd to set up namespaces, cgroups, and exec the container process.

  • Low-level: runc is the bottom of the stack — directly calls Linux syscalls to create namespaces and cgroups.
  • OCI spec: Implements the OCI Runtime Specification — accepts an OCI bundle (rootfs + config.json).
  • Alternatives: crun (faster, C), kata-containers (VM-based), gVisor (sandboxed) are OCI-compatible alternatives.
  • Chain: Docker daemon → containerd → containerd-shim → runc → container process.

OCI defines open standards for container images and runtimes — ensuring any OCI-compliant image can run on any OCI-compliant runtime across all container platforms.

  • Image Spec: Defines image manifest, layer format, config schema — Docker images are OCI-compliant.
  • Runtime Spec: Defines the OCI bundle format (rootfs + config.json) and container lifecycle operations.
  • Distribution Spec: Standardizes the HTTP API for pushing and pulling images from registries.
  • Portability: An image built with Docker runs on containerd, Podman, or any OCI runtime.

Podman is a daemonless, rootless OCI container tool — Docker-compatible CLI but without a central daemon, using fork-exec model instead of client-server architecture.

  • Client-server (daemon)

PID 1 in a container is responsible for reaping zombie processes — if your app is PID 1 and doesn't handle this, zombies accumulate causing resource leaks.

  • PID 1 responsibility: In Linux, PID 1 must reap (wait on) orphaned child processes — or they become zombies.
  • App as PID 1: Most apps (node, python, java) don't handle SIGCHLD/zombie reaping properly.
  • tini solution: Docker's --init flag adds tini as PID 1 — a minimal init that reaps zombies and forwards signals.
  • SIGTERM forwarding: tini also properly forwards SIGTERM from Docker stop to the child process.
docker run --init myapp
# OR in Dockerfile:
RUN apk add --no-cache tini
ENTRYPOINT ["/sbin/tini", "--"]
CMD ["node", "app.js"]

tini is a tiny init system designed specifically for containers — it acts as PID 1, reaps zombie processes, and properly forwards signals to child processes.

  • Signal forwarding: Docker's docker stop sends SIGTERM to PID 1; tini forwards it to the app process.
  • Zombie reaping: Calls waitpid() on orphaned processes so they don't accumulate as zombies.
  • Minimal: Only ~20KB binary — adds negligible overhead.
  • --init flag: docker run --init injects tini automatically without Dockerfile changes.

Notary implements The Update Framework (TUF) for Docker image signing — providing cryptographic verification of image authenticity and preventing supply chain attacks.

  • Key roles: Root key (most trusted), targets key (signs images), snapshot key, timestamp key.
  • Delegation: Root key delegates to team-specific targets keys — separate signing by team/environment.
  • DCT integration: DOCKER_CONTENT_TRUST=1 uses Notary transparently for pull verification.
  • Supply chain: Ensures images from CI/CD to production are not tampered with.
export DOCKER_CONTENT_TRUST=1
export DOCKER_CONTENT_TRUST_SERVER=https://notary.example.com
docker push myrepo/myapp:1.0  # signs the image

Docker Scout is Docker's supply chain security tool — it analyzes images for CVEs, generates SBOMs, and provides actionable remediation advice integrated into the build workflow.

  • SBOM generation: Generates Software Bill of Materials listing all packages in an image.
  • CVE scanning: Cross-references SBOM against vulnerability databases (NVD, OSV, etc.).
  • Remediation: Suggests base image upgrades or package updates that fix identified CVEs.

Container-to-container packets flow through Linux bridge (docker0) using kernel bridging — iptables FORWARD rules control which traffic is allowed between bridge networks.

  • Same network: Packets bridge directly via the Linux bridge — kernel handles L2 forwarding without going through the host network stack.
  • Different networks: DOCKER-ISOLATION iptables chains drop FORWARD traffic between bridges.
  • External traffic: POSTROUTING MASQUERADE NATts outbound packets to host IP; PREROUTING DNAT handles inbound port mappings.
  • ip_forward: Kernel must have net.ipv4.ip_forward=1 — Docker enables this automatically.

A service mesh adds L7 networking capabilities (mTLS, traffic policies, observability) via sidecar proxies alongside each container — commonly Envoy, Istio, or Linkerd in container environments.

  • Sidecar pattern: Each container has a proxy sidecar that intercepts all network traffic transparently.
  • mTLS: Mutual TLS between services — all inter-service traffic encrypted and authenticated automatically.
  • Traffic management: Circuit breaking, retries, rate limiting, canary releases at the network layer.
  • Observability: Distributed tracing, metrics, and access logs for all service-to-service communication.

Copy-on-write (CoW) means that reading a file reads from the lower (image) layer, but the first write triggers a copy of the file into the container's upper writable layer before modification.

  • Read performance: Reading shared files has minimal overhead — reads go directly to the image layer.
  • Write performance: First write to a file triggers copy-up — copying the entire file to the container layer. Large files are slow on first write.
  • Storage efficiency: Multiple containers based on the same image share read-only layers — only writable diff stored per container.
  • Volume bypass: Volume mounts bypass CoW entirely — writes go directly to the volume, avoiding the performance hit.

Layer squashing merges multiple image layers into one — reducing image size by eliminating intermediate files that were added then deleted in separate layers.

  • Problem solved: If you add a 100MB file and delete it in a later layer, both layers still exist — squashing merges them and removes the file entirely.
  • --squash flag: docker build --squash — experimental feature that squashes all layers into one.
  • Downside: Squashed images don't benefit from layer caching — full image must be pulled each time.
  • Better approach: Avoid the problem via multi-stage builds or combining RUN commands instead of squashing after the fact.

Use --read-only flag to make the container's root filesystem read-only — the container cannot write anywhere except explicitly mounted writable volumes or tmpfs mounts.

  • Immutable runtime: Prevents attackers from modifying container binaries or planting backdoors.
  • Writable exceptions: Use --tmpfs /tmp for temp writes; volumes for persistent data.
  • App compatibility: Many apps need /tmp or /var — add tmpfs mounts for those paths.
  • Security hardening: Combine with --cap-drop ALL and --no-new-privileges for defense-in-depth.
docker run --read-only \
  --tmpfs /tmp \
  --tmpfs /var/run \
  -v mydata:/app/data \
  myapp

Live restore keeps containers running when the Docker daemon is stopped or upgraded — decoupling container lifecycle from daemon lifecycle for zero-downtime daemon maintenance.

  • Enable: Set "live-restore": true in /etc/docker/daemon.json.
  • Daemon restart: Containers continue running during systemctl restart docker.
  • Reconnect: After daemon restarts, it reconnects to existing container processes.
  • Limitation: Not compatible with Swarm mode — for Swarm, rolling updates are used instead.
# /etc/docker/daemon.json:
{
  "live-restore": true
}
systemctl reload docker  # daemon restarts; containers keep running

Set custom DNS servers per-container with --dns flag, or globally via daemon.json — overrides the default (host DNS or 8.8.8.8 fallback).

  • Per-container: docker run --dns 1.1.1.1 --dns-search example.com myapp
  • Daemon default: Set "dns": ["1.1.1.1"] in /etc/docker/daemon.json for all containers.
  • Docker's embedded DNS: 127.0.0.11 handles Docker-internal names; external queries forwarded to configured DNS.
  • resolv.conf: Docker generates /etc/resolv.conf inside container with configured DNS servers.

Container resource quotas are enforced via cgroups — Docker translates --memory, --cpus, and I/O flags into cgroup controller settings written at container start.

A restart loop means PID 1 keeps exiting — check logs, inspect exit code, review the restart policy, and investigate the application error.

  • Step 1: docker logs myapp --tail 50 — catch the error message before last exit.
  • Step 2: docker inspect myapp --format '{{.State.ExitCode}}' — exit code 137 = OOM kill; 1 = app error; 2 = misuse.
  • Step 3: Override entrypoint to prevent exit: docker run --entrypoint sleep myimage 3600 then exec inside.
  • Step 4: Check resource limits — docker stats for OOM, docker events for OOM events.
  • Step 5: Review restart policy and consider changing to on-failure:3 to stop endless loops.
docker logs myapp --tail 100
docker inspect myapp --format '{{.State}}'
docker events --filter container=myapp

Immediately increase memory limit or add swap, then investigate the leak — profile the app before permanent fix.

  • Immediate: docker update --memory=2g myapp increases limit without restart on Linux.
  • Monitor: docker stats myapp to track live memory usage and trend.
  • Diagnose: Exec into container: docker exec myapp top or cat /proc/meminfo.
  • Profile: Use language-specific memory profilers (JVM heap dumps, Node heapdump, Python memory_profiler).
  • Long term: Fix memory leak, tune JVM heap settings, or add horizontal scaling to distribute load.

Slow builds are usually caused by large build context, cache-busting layer ordering, or re-downloading dependencies. Fix each systematically.

  • Add .dockerignore: Exclude node_modules, .git, dist/ — reduce context size from GBs to MBs.
  • Layer ordering: Copy dependency files first, install, then copy source — deps cached until lockfile changes.
  • BuildKit cache mounts: RUN --mount=type=cache,target=/root/.npm npm install — persist package caches.
  • Registry cache: Use --cache-from with a registry cache in CI pipelines.
  • Parallel stages: Use BuildKit multi-stage with independent stages that build in parallel.

Zero-downtime requires traffic routing and rolling replacement — use a load balancer (Nginx/Traefik) and replace containers one at a time with health checks gating traffic.

  1. Pull new image: docker pull myapp:v2
  2. Start new container on a different port: docker run -d -p 8081:80 --name myapp-v2 myapp:v2
  3. Health check new container: wait for it to pass health check
  4. Update load balancer (Nginx/Traefik) to route traffic to new container
  5. Stop old container: docker stop myapp-v1
  6. Remove old container after verification period
  • Swarm way: docker service update --image myapp:v2 myservice does rolling updates natively.
  • Traefik: Label-based routing auto-updates when new container comes up with same label.

Network connectivity issues between containers — check network membership, DNS resolution, port binding, and firewall rules systematically.

If separated into two RUN commands, the apt-get update layer gets cached permanently — subsequent installs may use a stale package index, causing version mismatches or 404 errors.

  • Cache problem: RUN apt-get update creates a cached layer. Later, when you add a new package to RUN apt-get install, Docker uses the cached (stale) index from weeks ago.
  • Package not found: Package exists in new index but not stale cached one → install fails with 404.
  • Correct pattern: Always combine: RUN apt-get update && apt-get install -y pkg && rm -rf /var/lib/apt/lists/*
  • Why cleanup: Removing /var/lib/apt/lists/* in the same layer prevents the package index from bloating the layer.

docker stop sends SIGTERM to PID 1 — but if your app isn't PID 1 (wrapped in a shell), it may never receive the signal and gets SIGKILL'd after the grace period.

  • Shell form trap: CMD node app.js runs as /bin/sh -c "node app.js" — shell is PID 1 and does NOT forward SIGTERM to node.
  • Exec form correct: CMD ["node", "app.js"] — node IS PID 1 and receives SIGTERM directly.
  • Result: Shell form containers always get SIGKILL after 10s — no graceful shutdown, open connections dropped, requests lost.
  • Verify: Check that your app handles SIGTERM with graceful shutdown logic (drain connections, close DB).

On user-defined bridge networks, all containers are allowed to communicate by default — Docker's FORWARD rules allow traffic between containers on the same bridge, and this is by design (not a misconfiguration).

  • By design: Containers on the same network are assumed to be part of the same application stack and trusted to communicate.
  • iptables FORWARD: Docker adds ACCEPT rules in the FORWARD chain for same-bridge traffic.
  • Isolation via networks: Isolate by putting containers on DIFFERENT networks — cross-network traffic is blocked by DOCKER-ISOLATION rules.

The bind mount hides (shadows) the existing container directory — the container sees only the host directory contents, and the original container files are inaccessible during the mount.

  • Shadowing: Like Linux mount behavior — original directory contents become invisible while the mount is active.
  • Not deleted: Original image files are not deleted — they're just inaccessible. Remove container without the volume mount and they reappear.
  • Common mistake: Mounting an empty host directory over /app that had compiled files — app fails to start because files are hidden.
  • Volume differs: Docker named volumes pre-populate from container directory on first use — bind mounts never do this.

Docker's cache is a chain — each layer's cache key includes its parent layer's digest. Changing an early layer changes its digest, making all child layer digests different, invalidating the entire chain below it.

  • Chain dependency: Layer N's cache key = hash(instruction + parent layer hash). Change parent → all descendants get new hashes.
  • Practical impact: Changing README.md at top of Dockerfile forces re-running every subsequent RUN — even npm install and gcc compilation.
  • Ordering strategy: Place COPY instructions for frequently changing files (source code) as late as possible.
  • RUN exception: RUN instructions invalidate only when the instruction text changes — they don't scan for external file changes.

COPY --chown sets ownership during the copy in a single layer; running chown in a separate RUN creates a new layer containing full copies of all changed files, potentially doubling image size.

  • COPY --chown efficiency: Sets ownership metadata without creating duplicate file content in a new layer.
  • RUN chown problem: RUN chown -R user:group /app — CoW copies every file in /app into the new layer just to change ownership metadata, potentially doubling disk usage.
  • Example: If /app is 500MB, RUN chown creates another 500MB layer just for permission metadata changes.
  • Always prefer: COPY --chown=node:node . /app over COPY . /app followed by RUN chown.

Linux requires PID 1 to reap orphaned child processes by calling waitpid() — if your app is PID 1 and doesn't handle SIGCHLD, orphaned children become permanent zombies consuming PID table entries.

  • Zombie creation: When a child process exits but its parent hasn't called wait(), it becomes a zombie — taking up a PID entry but consuming no other resources.
  • PID 1 responsibility: The kernel orphans all processes to PID 1 when their parent dies — PID 1 must reap them.
  • App as PID 1: Node.js, Python, Java do not reap zombies — they only wait on processes they explicitly spawned.
  • PID exhaustion: Zombie accumulation can exhaust the PID table (default max 32768) — new processes cannot be created.
  • Fix: Use docker run --init to add tini as PID 1, or use ENTRYPOINT ["/sbin/tini", "--"] in Dockerfile.
# Best practice: use tini
FROM node:20-alpine
RUN apk add --no-cache tini
ENTRYPOINT ["/sbin/tini", "--"]
CMD ["node", "app.js"]
# OR simpler:
docker run --init myapp
